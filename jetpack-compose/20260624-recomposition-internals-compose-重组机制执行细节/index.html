深入理解 Compose Recomposition：从初始化到重组的执行细节

一、概述

Section titled “一、概述”

Recomposition 是 Compose 运行时的核心机制：当状态发生变化时，运行时选择性地重新执行受影响的 composable 函数，更新 SlotTable 中的数据，并通过 Applier 将变更应用到 UI 树。

本文通过一个完整的 Demo，详细拆解从首次 composition 到状态变更触发 recomposition 的完整流程，涵盖 Composer 的模式切换、Slot 比对、ChangeList 记录与 apply 等关键环节。

三、Composer 的五种模式

Section titled “三、Composer 的五种模式”

Composer 没有一个统一的 Mode 枚举，而是通过多个独立的布尔标志位组合出不同模式。理解这些模式是理解 recomposition 流程的关键。

3.1 模式一览

Section titled “3.1 模式一览”
模式标志位含义
Insertinginserting = true正在创建新的 group/node，写入 insertTable
Skippingskipping = true可以跳过未变化的 composable
Pendingpending != null检测到结构变化，正在追踪 key 的增删移动
Reusingreusing = true软复用模式，旧数据被视为空（类似 inserting 但保留节点）
ComposingisComposing = true正在执行 composable 函数

3.2 Inserting 模式

Section titled “3.2 Inserting 模式”

含义：Composer 正在创建全新的 group 和 node，数据直接写入 SlotWriter。

触发时机：

  1. 首次 composition：整个过程 inserting = true
  2. Recomposition 中遇到新 key：在 start() 方法中，当 pending 不为 null 且 key 在旧 group 中找不到匹配时
  3. Recomposition 中 key 不匹配：在 startReplaceGroup() 中，当 SlotTable 当前 group 的 key 与期望的 key 不同时

Inserting 模式下的行为差异：

  • nextSlot() 返回 Composer.Empty（无旧数据可读）
  • changed() 始终返回 true（所有值都是”新”的）
  • 数据通过 writer.update(value) 直接写入 SlotTable
  • 退出时通过 recordInsert() 将 insertTable 中的新 group 插入到主 SlotTable

3.3 Skipping 模式

Section titled “3.3 Skipping 模式”

含义：Composer 可以跳过未变化的 composable 函数，不重新执行其函数体。

计算方式：

只有当以下条件全部满足时，skipping 才为 true：

  • 不在 inserting 模式
  • 不在 reusing 模式
  • CompositionLocal provider 未变化
  • 当前 RecomposeScope 未被标记为需要 recompose
  • 未强制 recompose（如 Live Edit）

3.5 Reusing 模式

Section titled “3.5 Reusing 模式”

含义：一种”软插入”模式，旧数据被视为空（nextSlot() 返回 Empty），但节点对象可以被复用而非销毁重建。

触发场景：

  • startReusableGroup() 中当 data key 变化时
  • enableReusing() / startReuseFromRoot() 显式启用
  • 暂停恢复（pausable composition）时从 scope 恢复 reusing 状态

与 Inserting 的区别：

  • Inserting：创建全新节点
  • Reusing：节点可以复用（如 LazyColumn 的 item 回收）

6.1 入口逻辑

Section titled “6.1 入口逻辑”

三分支的设计原因：

  • content != null：首次 composition 或 Recomposer 传入了新的 content，直接执行
  • forciblyRecompose || providersInvalid：content 为 null（重组路径），但 Provider 发生了变化或工具链强制 recompose。此时 savedContent 从 SlotTable 的 nextSlot() 读取旧 content，重新执行以传播 Provider 变化
  • else：真正的重组路径，content 为 null 且无 Provider 变化。此时不需要执行 root group，因为变更来自子 scope 的 invalidation，会在 recomposeToGroupEnd() 中被选择性处理

startGroup(invocationKey, invocation) / endGroup() 包装的作用：invocation group 是 Compose 运行时识别 composable 调用边界的特殊 group，确保 content lambda 的执行被正确包裹在 SlotTable 的 group 结构中。

九、ChangeList 与 applyChanges()

Section titled “九、ChangeList 与 applyChanges()”

9.1 ChangeList 是什么

Section titled “9.1 ChangeList 是什么”

Recomposition 过程中，所有变更（slot 更新、group 插入/删除/移动、node 创建/插入/移除）都被记录到一个 ChangeList 中。ChangeList 是一个操作序列，在 recomposition 完成后统一执行。

9.2 为什么不在 Recomposition 中直接修改 SlotTable？

Section titled “9.2 为什么不在 Recomposition 中直接修改 SlotTable？”

这是一个关键设计决策。在 recomposition 过程中：

  1. SlotReader 持有旧数据的快照：用于与新数据比较
  2. SlotWriter 用于 inserting 模式：写入新 group 到 insertTable
  3. ChangeList 记录所有变更：作为中间层

这种设计的好处：

  • Recomposition 可以安全地读取旧数据进行比较
  • 多个 Composition 的变更可以合并执行
  • 出错时可以安全丢弃 ChangeList
  • 变更执行可以精确控制 Applier 的回调顺序

9.3 applyChanges() 的执行

Section titled “9.3 applyChanges() 的执行”

changes.execute() 会遍历 ChangeList 中的每一条操作：

  • Slot 更新：直接修改 SlotTable 的 slots 数组
  • Node 创建：调用 factory() 创建节点对象，写入 Node slot
  • Node 插入/移动：通过 Applier 回调通知 UI 层（如 Android ViewGroup）

Section titled “十、remember() / cache() 机制”

10.1 remember() 的编译器转换

Section titled “10.1 remember() 的编译器转换”

无参版本直接传 false 而非 changed(0)，因为无 key 意味着永远不需要重新计算——缓存值在 scope 存续期间始终有效，不需要读取旧 slot 进行比较。

十一、skipToGroupEnd() 与 skipCurrentGroup()

Section titled “十一、skipToGroupEnd() 与 skipCurrentGroup()”

11.1 skipCurrentGroup()

Section titled “11.1 skipCurrentGroup()”

skipCurrentGroup() 是编译器调用的 API，用于跳过一个已进入的 group。它与 skipToGroupEnd() 不同：后者用于 composable 函数体内跳过剩余子 group，而 skipCurrentGroup() 用于跳过整个 group（包括进入和退出）。

11.2 skipGroup() 的快速跳过

Section titled “11.2 skipGroup() 的快速跳过”

skipGroup() 直接将 currentGroup 指针向前移动整个 group 的 Size，跳过所有子 group，不读取任何 slot 数据。

11.3 skipToGroupEnd()

Section titled “11.3 skipToGroupEnd()”

skipToGroupEnd() 是编译器在 composable 函数体内调用的 API，用于跳过当前 group 的剩余子 group。它与 skipCurrentGroup() 的关键区别在于：它还会标记当前 scope 为”已跳过”（scopeSkipped()），这对 RecomposeScope 的生命周期管理很重要。
