从一个 Text 组件看透 Jetpack Compose 渲染全流程

引言

Section titled “引言”

Jetpack Compose 的渲染管线常被形容为”黑盒”——写一个 Text("Hello")，文字就出现在屏幕上，中间发生了什么？本文以一个最简场景为线索，逐层拆解从 setContent 调用到像素上屏的完整链路。所有分析基于 AndroidX 源码。

1 场景

Section titled “1 场景”

一个传统 TextView，一个 Compose Text，并排放在同一个 LinearLayout 里。我们的问题是：Text("Hello Compose") 这行代码，究竟经历了什么才把文字画到屏幕上？

2 全局视角：五阶段管线

Section titled “2 全局视角：五阶段管线”

Compose 渲染是一条五阶段管线，每一阶段的输出是下一阶段的输入：

阶段职责核心类
Composition执行 @Composable 函数，生成树变更Composer, SlotTable
Apply将变更应用到 LayoutNode 树UiApplier, LayoutNode
Measure确定每个节点尺寸MeasurePolicy, MeasurePassDelegate
Layout确定每个节点位置MeasureAndLayoutDelegate
Draw将内容绘制到 CanvasNodeCoordinator, LayoutNodeDrawScope

下面逐阶段展开。

3 阶段一：Composition——执行 Composable，记录变更

Section titled “3 阶段一：Composition——执行 Composable，记录变更”

3.1 setContent 触发组合

Section titled “3.1 setContent 触发组合”

ComposeView.setContent() 保存 content lambda 并调用 createComposition()：

createComposition() 做两件事：

  1. 创建 AndroidComposeView——一个真正的 ViewGroup，作为 Compose 与 Android View 系统的桥梁。
  2. 创建 Composition 对象，调用 composition.setContent(content)，启动组合。

3.2 Composer 执行 Composable 函数

Section titled “3.2 Composer 执行 Composable 函数”

Composition.setContent() 将 content lambda 交给 Composer 执行。Composer 是一台状态机，逐行执行 Composable 函数体，同时做三件事：

  • 写入 SlotTable：将 group 信息、remember 值、节点引用等持久化到 SlotTable。SlotTable 使用 Gap Buffer 实现——一种在编辑器中广泛使用的线性数据结构，核心特点是：在当前位置（即 gap 所在位置）的插入/删除操作接近 O(1)。
  • 追踪状态读取：当 Composable 中读取 State.value 时，Composer 记录”当前 scope 依赖了这个 State”，为后续重组建立订阅关系。
  • 生成变更操作：对比 SlotTable 新旧状态，生成 insert / remove / move / update 等操作，存入 ChangeList。

3.4 SlotTable：Composition 阶段的数据结构

Section titled “3.4 SlotTable：Composition 阶段的数据结构”

SlotTable 使用 Gap Buffer 实现——一个在连续位置插入/删除极快的线性数据结构：

Gap Buffer 的核心优势：组合过程中大量操作集中在当前位置（即 gap 所在位置），移动 gap 的代价极低，接近 O(1)。

关键区分：SlotTable 是 Composition 阶段的内部数据结构，存储的是组合状态，不是 UI 节点树。UI 节点树在下一阶段由 Applier 构建。

6.1 Layout 的基本流程

Section titled “6.1 Layout 的基本流程”

measureAndLayout() 遍历所有标记为需要重新布局的 LayoutNode，对每个节点：

  1. 调用其 MeasurePolicy 的 measure() 获取 MeasureResult。
  2. 执行 MeasureResult 中的 place() 调用，确定子节点相对于父节点的 (x, y) 偏移。
  3. 更新 LayoutNode.position。

6.2 measureAndLayout() 的完整职责

Section titled “6.2 measureAndLayout() 的完整职责”

measureAndLayout() 并不是简单的布局，它实际上是一个完整的测量-布局循环：

关键点：

  • 测量与布局的耦合：某些布局（如 Box、Column）在测量阶段就需要知道子节点的位置，因此 measure 和 layout 可能交替执行。
  • 级联更新：子节点尺寸变化可能触发父节点重新测量，形成向上的传播链。
  • onGloballyPositioned：所有测量布局完成后，才会分发 onGloballyPositioned 回调，确保回调中能拿到正确的位置信息。

本例中，Text 节点被放置在 (0, 0)（相对于 root）。

8 传统 View 与 Compose 的混合渲染

Section titled “8 传统 View 与 Compose 的混合渲染”

回到我们的例子：LinearLayout 里同时放了传统 TextView 和 Compose Text。它们是如何一起渲染的？

8.1 共享同一套渲染管线

Section titled “8.1 共享同一套渲染管线”

关键事实：

  • 同一个 Canvas：Android View 系统自顶向下分发同一个 android.graphics.Canvas，传统 View 和 Compose 都在这个 Canvas 上绘制。
  • 同一个 HardwareRenderer：所有绘制指令统一由 Android 硬件渲染器处理。
  • 同一个 RenderThread：绘制指令在同一个渲染线程上执行。
  • 同一个 DisplayList：所有指令被缓存到 DisplayList，GPU 统一光栅化。

传统 View 和 Compose 的绘制指令最终合并到同一个 DisplayList，由 GPU 一起渲染上屏。它们不是两套独立的渲染系统，而是同一套管线中的两个绘制源。

8.2 为什么可以无缝共存

Section titled “8.2 为什么可以无缝共存”

ComposeView 和 AndroidComposeView 都是标准的 Android View / ViewGroup，完全遵循 Android 的 measure → layout → draw 协议。Android View 系统并不知道 ComposeView 内部用的是 Compose 渲染，它只看到一个普通的子 View，按标准流程调度即可。

9.2 智能重组

Section titled “9.2 智能重组”

Compose 的核心优势是智能重组——只有状态真正改变的 Composable 才会重新执行。这由三层机制保证：

  1. 位置记忆（Positional Memoization）：Composer 根据 Composable 在源码中的调用位置（group key）判断是否可以跳过。
  2. 状态订阅粒度：只有读取了变化 State 的 scope 才会被标记无效。
  3. SlotTable 差分：Composer 对比 SlotTable 新旧状态，只生成必要的变更操作。

10 数据结构总结

Section titled “10 数据结构总结”
数据结构所在阶段用途特点选择原因
SlotTable (Gap Buffer)Composition存储组合状态连续位置插入/删除极快组合过程大量线性扫描和当前位置插入
ChangeListComposition → Apply记录树变更操作线性表，顺序执行Applier 按顺序逐条执行变更
LayoutNode 树 (MutableVector)Apply / Measure / Layout / Draw实际 UI 节点树ArrayList，支持随机访问树遍历和小规模子节点操作频繁
DisplayListDraw → GPU缓存绘制指令硬件加速，GPU 执行避免每帧重新录制绘制指令

12 小结

Section titled “12 小结”

Compose 的渲染并非魔法，而是一条清晰的管线：

  1. Composition 是声明层，用 SlotTable (Gap Buffer) 做差分，生成最小变更集。
  2. Apply 是桥接层，用 UiApplier 将变更应用到 LayoutNode 树 (ArrayList)。
  3. Measure / Layout / Draw 是执行层，完全嵌入 Android View 系统的 onMeasure / onLayout / dispatchDraw 协议。

传统 View 和 Compose 共享同一个 Canvas、同一个 HardwareRenderer、同一个 RenderThread。它们不是两套渲染系统，而是同一管线中的两个绘制源。理解了这一点，混合使用时的行为就不难预测了。
