三、稳定性推断系统（Stability Inference）

Section titled “三、稳定性推断系统（Stability Inference）”

3.1 为什么需要稳定性？

Section titled “3.1 为什么需要稳定性？”

Compose 的跳过（Skip）机制依赖于判断参数是否变化。如果参数类型是”稳定的”（相等性含义明确且不可变），编译器就可以在参数未变时跳过重组；如果类型”不稳定”，每次都必须执行。

3.2 Stability 类型体系

Section titled “3.2 Stability 类型体系”

源码：analysis/Stability.kt

推断规则（StabilityInferencer.stabilityOf()）：

  1. 直接标记：类标注了 @Stable / @Immutable（或其父类有 @StableMarker）→ Stable
  2. 已知稳定类型：Pair、Triple、Result、枚举、object、基本类型、BigInteger 等 → Stable
  3. 外部稳定类型：通过 stabilityConfigurationPath 配置文件指定 → Stable
  4. 接口：→ Unknown（无法确定实现是否稳定）
  5. 含 var 属性的类：→ Unstable
  6. 含不稳定属性的类：递归检查所有属性 → Combined
  7. 泛型类：→ Parameter + 具体类型参数的稳定性
  8. 跨文件公开类（JVM）：→ Runtime（运行时通过 $stable 字段判断）
  9. Java 类：→ Unstable（保守策略）

3.3 已知稳定类型注册表

Section titled “3.3 已知稳定类型注册表”

源码：analysis/KnownStableConstructs.kt

3.4 ClassStabilityTransformer — 稳定性标注注入

Section titled “3.4 ClassStabilityTransformer — 稳定性标注注入”

源码：lower/ClassStabilityTransformer.kt

此 Pass 遍历所有类，推断其稳定性，然后：

  1. 在类上添加 @StabilityInferred(bits) 注解：将推断出的稳定性位掩码写入注解参数
  2. 在类中生成 $stable 静态字段：供运行时读取

四、Durable Key 生成

Section titled “四、Durable Key 生成”

4.1 为什么需要 Key？

Section titled “4.1 为什么需要 Key？”

Compose 运行时使用 SlotTable 存储组合状态，每个 Group 在 SlotTable 中通过 Key 定位。Key 必须是”持久的”（Durable）——即同一调用点在不同次执行中必须生成相同的 Key，否则运行时无法正确匹配状态。

6.1 三类 Group

Section titled “6.1 三类 Group”

Compose 运行时中有三种 Group，对应不同的语义：

Group 类型API用途
Replace GroupstartReplaceableGroup / endReplaceGroup条件分支：when/if 中只有一个分支会执行

6.4 参数比较逻辑（Skipping Preamble）

Section titled “6.4 参数比较逻辑（Skipping Preamble）”

源码：buildPreambleStatementsAndReturnIfSkippingPossible()

核心逻辑：对于每个状态为 Uncertain 的参数，调用 $composer.changed(param) 比较：

  • 0b0110 — 隔离第 0 个参数的 3-bit 状态（排除最低强制位）
  • 如果 changed() 返回 true（值变了）→ 设置为 Different(0b010)
  • 如果 changed() 返回 false（值没变）→ 设置为 Same(0b001)

6.5 跳过判断

Section titled “6.5 跳过判断”

0b1011 vs 0b1010 的含义：

  • 0b1010 = 第 0 个参数是 Same(0b001) 且强制位为 0 → 所有参数未变
  • 0b1011 = 掩码，检查所有参数是否都是”相同”状态
  • 如果 $dirty & 0b1011 != 0b1010，说明至少有一个参数变化了 → 执行
  • 或者 $composer.skipping == false → 运行时禁用了跳过 → 执行

十六、总结

Section titled “十六、总结”

Compose 编译器插件的本质是将 声明式的 Composable 函数 变换为 与运行时深度协作的状态机。核心变换包括：

  1. 参数注入：$composer、$changed、$default — 为运行时提供控制通道
  2. Group 生成：Restart / Replace / Movable — 为 SlotTable 提供结构
  3. 跳过逻辑：基于 $changed 位掩码判断参数变化 — 避免不必要的重组
  4. 比较传播：将参数状态从父函数传递给子函数 — 减少运行时开销
  5. 重组回调：updateScope — 让运行时知道如何重新执行此范围
  6. 稳定性推断：编译期确定类型稳定性 — 决定跳过策略
  7. Lambda 记忆化：缓存 Composable Lambda 实例 — 避免引用变化导致的重组

理解这些编译期变换，是理解 Compose 运行时行为的基石。
