ChangeList 生成机制详解：从首次组合到重组的完整追踪

一、核心概念：ChangeList 是什么

Section titled “一、核心概念：ChangeList 是什么”

ChangeList 是 Compose 运行时的延迟变更队列。Recomposition 过程中产生的所有结构变更（slot 更新、node 创建/移动/删除、group 插入/移动/删除）都被记录为一系列 Operation，在 recomposition 完成后由 applyChanges() 统一执行。

为什么不直接修改 SlotTable？

  1. 读写分离：Recomposition 需要 SlotReader 读取旧值进行比较，如果同时写入会破坏读取一致性
  2. 原子性：所有变更要么全部生效，要么全部丢弃（出错时安全回退）
  3. 精确排序：Applier 的回调顺序（down/up/move/remove）需要精确控制，ChangeList 提供了统一的排序点

四、首次 Composition：ChangeList 的生成

Section titled “四、首次 Composition：ChangeList 的生成”

4.1 核心原则：首次组合时 ChangeList 几乎为空

Section titled “4.1 核心原则：首次组合时 ChangeList 几乎为空”

首次 composition 时 inserting=true 贯穿全程。此时：

  • updateValue() → 直接写入 writer.update(value)（insertTable 的 SlotWriter），不产生 ChangeList Operation
  • startGroup() / endGroup() → 直接写入 writer.startGroup() / writer.endGroup()，不产生 ChangeList Operation
  • createNode() → 写入 insertFixups（FixupList），不直接产生 ChangeList Operation

唯一产生 ChangeList Operation 的时机：最外层 end() 退出时，调用 recordInsert() 将整个 insertTable 内容刷入 ChangeList。

5.4 为什么 Button 子树没有 ChangeList Operation

Section titled “5.4 为什么 Button 子树没有 ChangeList Operation”

Button 的 scope 未被 invalidate，scope.requiresRecompose=false → skipping=true → skipToGroupEnd() → 整个 Button 子树被跳过。

跳过时的 ChangeList 行为：

操作是否产生 ChangeList Operation原因
skipToGroupEnd()❌ 不产生只移动 reader 指针，不修改任何数据
reader.skipToGroupEnd()❌ 不产生reader 内部操作，writer 不参与
scopeSkipped()❌ 不产生标记 scope 状态，不影响 SlotTable

Button 子树中的 onClick lambda、content lambda、Text(“Increment”) 都不会被访问，reader 直接跳到 Button group 的末尾。
