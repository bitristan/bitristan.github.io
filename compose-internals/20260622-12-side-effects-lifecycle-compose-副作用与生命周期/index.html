一、副作用的设计理念

Section titled “一、副作用的设计理念”

Compose 的组合过程必须是纯函数——相同的输入产生相同的 UI 描述。但真实应用不可避免地需要执行副作用（网络请求、传感器监听、动画等）。Compose 通过将副作用绑定到组合生命周期来解决这个矛盾：

核心机制是 RememberObserver 接口——它是所有生命周期感知副作用的基石。

十三、关键设计决策

Section titled “十三、关键设计决策”

13.1 为什么副作用必须绑定 key？

Section titled “13.1 为什么副作用必须绑定 key？”

key 的变化触发”清理旧副作用 + 创建新副作用”。如果没有 key（Unit），副作用只在进入/离开组合时执行。如果有 key，副作用会在 key 变化时重启。

这避免了闭包捕获过时值的问题——新 key = 新闭包 = 新副作用。

13.2 为什么 rememberUpdatedState 存在？

Section titled “13.2 为什么 rememberUpdatedState 存在？”

有些副作用需要”永不重启”（key = Unit），但又需要访问最新的参数。rememberUpdatedState 提供了一个间接层：State 对象引用不变，但值始终是最新的。

13.3 为什么 SideEffect 在 onRemembered 之后执行？

Section titled “13.3 为什么 SideEffect 在 onRemembered 之后执行？”

SideEffect 通常用于同步状态到外部系统（如 analytics）。如果在 onRemembered 之前执行，副作用可能看到的是不完整的组合状态。

13.4 为什么 WindowRecomposer 使用 PausableClock？

Section titled “13.4 为什么 WindowRecomposer 使用 PausableClock？”

当 Activity 进入后台（ON_STOP），动画应该暂停而不是取消。暂停时钟使得所有 withFrameNanos 调用挂起，恢复时从暂停点继续。这比取消+重新创建动画高效得多。
