四、Animation 接口：无状态计算器

Section titled “四、Animation 接口：无状态计算器”

4.1 接口定义

Section titled “4.1 接口定义”

Animation 是纯粹的时间→值映射函数。给定播放时间，返回对应的值和速度。它不管理生命周期。

4.2 TargetBasedAnimation

Section titled “4.2 TargetBasedAnimation”

TargetBasedAnimation 包装了一个 VectorizedAnimationSpec，缓存初始值和目标值。它的 getValueFromNanos() 将调用委托给向量化规格，然后通过 typeConverter 转换回应用类型。

4.3 DecayAnimation

Section titled “4.3 DecayAnimation”

DecayAnimation 用于减速动画（如 fling 后的惯性滚动）。它总是有限的（isInfinite = false），时长由衰减规格决定。

五、AnimationState：可变状态持有器

Section titled “五、AnimationState：可变状态持有器”

AnimationState 持有动画的当前状态：

  • value: T — 由 mutableStateOf 支持，更新时触发重组
  • velocityVector: V — 当前速度
  • isRunning: Boolean — 是否正在运行
  • lastFrameTimeNanos — 最后一帧的墙钟时间
  • finishedTimeNanos — 动画完成时间

所有 setter 都是 internal set——只有动画系统内部可以修改状态。

十二、关键设计决策

Section titled “十二、关键设计决策”

12.1 为什么需要 TwoWayConverter？

Section titled “12.1 为什么需要 TwoWayConverter？”

动画引擎只需要在向量空间中做数学运算。通过 TwoWayConverter，任何类型（Dp、Color、Offset）都可以参与动画，而不需要修改动画引擎本身。这也是为什么动画系统可以独立于 UI 模块存在。

12.2 为什么 AnimationSpec 和 Animation 分离？

Section titled “12.2 为什么 AnimationSpec 和 Animation 分离？”
  • AnimationSpec 是配置（声明式），可以序列化、比较、组合
  • Animation 是执行（状态持有），管理具体的值和速度

这种分离使得相同的 AnimationSpec 可以被多个 Animation 复用。

12.3 为什么 Transition 的 transitionSpec 是 composable？

Section titled “12.3 为什么 Transition 的 transitionSpec 是 composable？”

因为不同状态转换可能需要不同的动画规格。例如，从 A 到 B 可能用 300ms 的 tween，而从 B 到 A 可能用弹簧。composable lambda 可以读取 segment 来决定使用哪个规格。

12.4 为什么用 Channel.CONFLATED？

Section titled “12.4 为什么用 Channel.CONFLATED？”

animateValueAsState 使用汇合通道确保快速连续的目标值变化不会排队——只有最新的值会被处理。这避免了动画队列积压。

12.5 为什么 Animatable 有边界强制？

Section titled “12.5 为什么 Animatable 有边界强制？”

在很多场景中（如拖拽），值需要被限制在范围内。Animatable 内置了这个能力，避免了开发者手动 coerceIn。
