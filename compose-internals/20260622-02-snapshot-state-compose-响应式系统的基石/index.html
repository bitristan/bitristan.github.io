三、Snapshot 系统：全局快照架构

Section titled “三、Snapshot 系统：全局快照架构”

3.1 核心概念

Section titled “3.1 核心概念”

Snapshot 系统的设计借鉴了数据库的 MVCC（多版本并发控制）思想。它的核心概念：

  • Global Snapshot：全局快照，代表当前所有线程可见的”最新状态”。整个系统只有一个 Global Snapshot。
  • Thread Snapshot：线程局部快照，存储在 SnapshotThreadLocal 中。当一个 Snapshot 被 enter 后，它就成为当前线程的活动快照。
  • SnapshotId：每个 Snapshot 有一个单调递增的唯一 ID。
  • Invalid Set：每个 Snapshot 维护一个”无效 ID 集合”，记录哪些 Record 对它而言是无效的。

3.2 快照类型

Section titled “3.2 快照类型”
  • ReadonlySnapshot：只能读取状态，不能修改。用于布局（layout）和绘制（draw）阶段。
  • MutableSnapshot：可以修改状态，修改对外不可见，直到 apply() 被调用。用于组合（composition）阶段。
  • TransparentObserverMutableSnapshot：一种特殊的可写快照，它不创建新的隔离环境，而是透传到父快照。Compose 用它来在不改变快照上下文的情况下附加读写观察者。

4.2 writeObserver：追踪重组期间的写入

Section titled “4.2 writeObserver：追踪重组期间的写入”

当状态被修改时，notifyWrite 触发 writeObserver。Compose 在重组期间使用 writeObserver 解决一个关键问题：如果一个 State 在同一个 RecomposeScope 中先被写入、后被读取，Composer 需要将这个写入也纳入无效化范围。

例如，在 CompositionLocalProvider 中，一个动态 CompositionLocal 的值在组合过程中被写入，随后子组件读取它。writeObserver 确保这种”先写后读”的依赖关系被正确追踪，使得子组件在下次重组时能读取到最新的值。

五、SnapshotStateObserver：连接状态与重组

Section titled “五、SnapshotStateObserver：连接状态与重组”

SnapshotStateObserver 是连接 Snapshot 状态系统和 Compose 重组系统的核心桥梁。

5.1 核心设计

Section titled “5.1 核心设计”

ObservedScopeMap 内部维护两个核心数据结构：

  • scopeToValues：Scope → Set<StateObject> — 每个 Scope（如 RecomposeScope）读取了哪些 State
  • valueToScopes：StateObject → Set<Scope> — 每个 State 被哪些 Scope 读取

这两个映射是双向的，使得：

  • 当 State 变化时，可以 O(1) 找到所有受影响的 Scope
  • 当 Scope 重组时，可以清除旧的读取记录并重新建立

5.3 线程安全

Section titled “5.3 线程安全”

SnapshotStateObserver 不是线程安全的，但它通过 onChangedExecutor 确保通知回调在指定的线程（通常是主线程）上执行。变更的收集（addChanges）使用 AtomicReference 实现无锁入队。

六、DerivedState：派生状态的惰性求值

Section titled “六、DerivedState：派生状态的惰性求值”

derivedStateOf 创建一个 DerivedSnapshotState，它不直接存储值，而是在被读取时根据依赖的状态计算值。

6.1 核心实现

Section titled “6.1 核心实现”

关键设计：

  • DerivedSnapshotState 持有一个 ResultRecord，缓存最近一次计算的结果
  • ResultRecord 记录了计算时依赖的所有 StateObject（dependencies 字段）
  • ResultRecord 还记录了计算时的 validSnapshotId 和 validSnapshotWriteCount

6.2 何时重新计算

Section titled “6.2 何时重新计算”

当你读取 derivedState.value 时：

  1. 检查当前 ResultRecord 是否对当前 Snapshot 有效
  2. 如果有效，直接返回缓存值
  3. 如果无效，重新执行 calculation lambda，更新 ResultRecord 的 dependencies

性能优势：假设 derivedStateOf { a.value + b.value } 被 100 个 UI 组件读取。当 a 或 b 变化时，derivedState 本身只会重新计算一次。下游的 100 个组件只有在计算结果确实变化时（由 policy 判断）才会触发重组。

6.3 与 applyObserver 的集成

Section titled “6.3 与 applyObserver 的集成”

DerivedSnapshotState 实现了 StateObject，因此它的 ResultRecord 会被 Snapshot 的 applyObserver 传播。SnapshotStateObserver 通过 dependencies 字段判断一个 State 变化是否真的会影响 DerivedState 的值，从而避免不必要的无效化。

八、并发控制：快照隔离与乐观合并

Section titled “八、并发控制：快照隔离与乐观合并”

8.1 快照隔离

Section titled “8.1 快照隔离”

Snapshot 系统的核心保证是快照隔离（Snapshot Isolation）：

  • 在一个 MutableSnapshot 中的修改对其他 Snapshot 不可见
  • 只有调用 apply() 后，修改才对新的 Snapshot 可见
  • 每个 Snapshot 看到的是创建时刻的一致性视图

这保证了重组过程的原子性：Composer 在一个 MutableSnapshot 中执行重组，如果重组过程中状态被其他线程修改，apply() 会失败并触发重新重组。

Section titled “8.2 乐观合并”

十、关键设计决策与权衡

Section titled “十、关键设计决策与权衡”

10.1 为什么用链表而非 Map 存储 StateRecord？

Section titled “10.1 为什么用链表而非 Map 存储 StateRecord？”

链表结构避免了 HashMap 的开销。在 Compose 中，一个 State 通常只有 1-2 个活跃的 Record（当前 Snapshot + 可能的嵌套 Snapshot），链表遍历的开销可以忽略。

10.2 为什么需要同步块？

Section titled “10.2 为什么需要同步块？”

overwritable 和 writable 操作必须在 sync {} 块中执行。这是因为 readable 函数遍历链表时不是原子的——如果另一个线程正在创建新的 Snapshot 并修改链表，可能导致读取到不一致的状态。sync {} 块确保了写入和快照创建的互斥。

10.3 为什么 SnapshotStateObserver 不是线程安全的？

Section titled “10.3 为什么 SnapshotStateObserver 不是线程安全的？”

因为 Compose 的重组是在单线程（主线程）上执行的。SnapshotStateObserver 的所有 observeReads 调用都发生在主线程，只有 applyObserver 的通知可能来自其他线程，但 addChanges 使用 AtomicReference 实现了无锁入队。
