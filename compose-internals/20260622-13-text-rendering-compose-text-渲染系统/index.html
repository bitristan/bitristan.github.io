第 13 篇：Text 渲染系统 — 文本的测量、布局与渲染管线

深入理解 Compose 的文本渲染管线：从 AnnotatedString 到 TextMeasurer，从 MultiParagraph 到 AndroidParagraph，从 TextPainter 到平台 StaticLayout。

二、AnnotatedString：带样式的文本

Section titled “二、AnnotatedString：带样式的文本”

2.1 结构

Section titled “2.1 结构”

AnnotatedString 是 Compose 文本系统的基础数据类型。它由纯文本字符串和一组注解范围（Range<Annotation>）构成。注解可以是：

  • SpanStyle（字符级样式）：字体、颜色、粗细、大小等
  • ParagraphStyle（段落级样式）：对齐、行高、断行策略等
  • StringAnnotation（自定义注解）：URL、点击处理等

2.2 构建

Section titled “2.2 构建”

Builder 在 init 中预索引 spanStylesOrNull 和 paragraphStylesOrNull，加速后续查询。

2.3 段落样式的非重叠约束

Section titled “2.3 段落样式的非重叠约束”

段落样式不能部分重叠——它们可以嵌套或完全相同，但不能交叉。这是因为段落是布局的基本单元，部分重叠会导致布局歧义。

3.1 TextStyle：双层样式

Section titled “3.1 TextStyle：双层样式”

TextStyle 同时包含字符级和段落级样式。merge() 方法用另一个 TextStyle 的非默认值填充当前样式的空缺。

关键优化：hashCodeLayoutAffectingAttributes() 只计算影响布局的属性（字体、大小、间距等），忽略颜色、阴影等纯绘制属性。TextLayoutCache 使用这个哈希值，使得颜色动画不会使布局缓存失效。

十一、关键设计决策

Section titled “十一、关键设计决策”

11.1 为什么需要 TextMeasurer 而不是直接用 Text composable？

Section titled “11.1 为什么需要 TextMeasurer 而不是直接用 Text composable？”

TextMeasurer 将文本测量从 composable 树中解耦，使得：

  • 可以在非 composable 上下文中测量文本（如 Layout 的 MeasurePolicy）
  • 可以缓存测量结果（LRU 缓存）
  • 可以在 DrawScope 中直接绘制文本（drawText 扩展）

11.2 为什么 TextLayoutCache 忽略颜色属性？

Section titled “11.2 为什么 TextLayoutCache 忽略颜色属性？”

文本颜色频繁变化（如选中高亮、动画），但不影响布局。通过 hashCodeLayoutAffectingAttributes() 分离布局属性和绘制属性，避免了大量不必要的重新布局。

11.3 为什么 MultiParagraph 而不是单个 Paragraph？

Section titled “11.3 为什么 MultiParagraph 而不是单个 Paragraph？”

不同的文本段可能有不同的段落样式（对齐、行高、断行策略）。MultiParagraph 按段落样式分割文本，每段独立布局，然后组合为统一的坐标空间。

11.4 为什么 BasicText 使用 Modifier.Node 而不是子 composable？

Section titled “11.4 为什么 BasicText 使用 Modifier.Node 而不是子 composable？”

文本渲染不需要子节点树——它是一个自包含的绘制操作。使用 Modifier.Node 避免了创建不必要的 LayoutNode，减少了树的深度和内存开销。

11.5 为什么选择系统使用 Registrar 模式？

Section titled “11.5 为什么选择系统使用 Registrar 模式？”

多个独立的文本节点可能需要参与同一个选区。SelectionRegistrar 作为中介，允许 SelectionManager 统一管理所有可选中的节点，而不需要它们有共同的父 composable。
