3.5 Speculative Hit（投机命中）

Section titled “3.5 Speculative Hit（投机命中）”

当一个节点设置了 interceptOutOfBoundsChildEvents() = true 时，即使指针不在其子节点范围内，父节点也会”投机性”地尝试命中。如果子节点有更好的命中，投机命中会被丢弃。

这用于实现类似 LazyList 的场景：滚动容器需要在子项还没进入可见区域时就开始接收事件。

  • 在子节点看到事件之前做决策

  • 按钮消费点击事件

  • 记录事件处理结果
  • 确保所有节点都看到最终状态

十一、关键设计决策

Section titled “十一、关键设计决策”

11.1 为什么有三个事件阶段？

Section titled “11.1 为什么有三个事件阶段？”

三个阶段提供了完整的事件处理灵活性：

  1. Initial：父节点可以在子节点之前预处理事件（如滚动容器准备拦截）
  2. Main：子节点优先处理（如按钮消费点击），然后父节点处理
  3. Final：所有 Main 处理完成后的后处理（如状态同步）

这种设计使得父子节点之间的事件协商成为可能。

11.2 为什么用协程处理手势？

Section titled “11.2 为什么用协程处理手势？”

手势识别本质上是跨多个事件的状态机。用协程实现的优势：

  1. 声明式：awaitPointerEvent() 读起来像同步代码
  2. 可组合：多个手势可以通过协程的结构化并发组合
  3. 可取消：手势冲突时可以自然地取消协程
  4. 状态管理：协程作用域自然地管理手势的生命周期

11.3 为什么命中测试需要优先级系统？

Section titled “11.3 为什么命中测试需要优先级系统？”

当多个节点重叠时，需要确定哪个节点”赢”了：

  • 直接命中 > 近似命中
  • 层内 > 层外
  • 较小的节点通常优先于较大的节点（因为小节点更”精确”）

这确保了按钮可以正确地在可滚动容器中被点击。

11.4 为什么键盘事件使用两阶段传播？

Section titled “11.4 为什么键盘事件使用两阶段传播？”

与触摸事件类似，键盘事件也需要父子节点之间的协商：

  • Pre-event（隧道）：祖先节点可以拦截快捷键
  • Event（冒泡）：焦点节点优先处理，然后祖先节点处理

这使得快捷键可以在任意层级被处理。
