第 1 篇：Compose 宏观架构总览

从全局视角理解 Compose 的模块划分、核心抽象和数据流，建立后续深入学习的心智模型。

一、为什么需要理解宏观架构

Section titled “一、为什么需要理解宏观架构”

Jetpack Compose 不是一个单独的库，而是一个由多个模块协同工作的系统。在深入任何单一模块之前，先理解整个系统的全貌，能帮助你在后续阅读源码时始终保持方向感——知道每个类在整个系统中扮演什么角色，以及数据是如何在模块之间流动的。

模块依赖关系

Section titled “模块依赖关系”

关键点：runtime 模块是框架的引擎，它不关心 UI 是什么——它只负责管理组合（Composition）和状态。ui 模块通过实现 Applier 接口，将运行时的抽象节点操作转换为实际的 UI 树操作。这种分离使得 Compose 的运行时理论上可以驱动任何类型的树形结构（例如 Compose for Desktop 驱动的是桌面 UI 节点）。

三、四个核心层次

Section titled “三、四个核心层次”

3.1 响应式基础层：Snapshot State

Section titled “3.1 响应式基础层：Snapshot State”

位置：compose/runtime 中的 Snapshot.kt、SnapshotState.kt

这是整个 Compose 的地基。mutableStateOf 创建的不是一个普通的变量，而是一个被全局快照系统（Snapshot System）管理的可观察状态对象。

当你读取 count.value 时，Snapshot 系统会记录”谁在读”；当你修改 count.value 时，Snapshot 系统会记录”谁被改了”。这个读写追踪机制是 Compose 能够精确知道哪些 UI 需要更新的基础。

核心类：

  • Snapshot：全局快照，管理读写追踪的上下文
  • SnapshotMutableState：mutableStateOf 返回的实际类型
  • SnapshotStateObserver：观察状态变化，将变化通知转发给 Recomposer

3.2 编译器层：Compose Compiler Plugin

Section titled “3.2 编译器层：Compose Compiler Plugin”

位置：Kotlin 仓库（plugins/compose），非 AndroidX

编译器插件的工作是将声明式的 @Composable 函数转换为运行时可以理解和高效执行的命令式代码。它做了三件关键的事：

  1. 注入 $composer 参数：每个 @Composable 函数都会隐式接收一个 Composer 实例
  2. 生成 Group 调用：在函数的入口和出口插入 startRestartGroup() / endRestartGroup()，将函数的调用映射到 SlotTable 中的 Group
  3. 生成跳过和重启代码：通过 $changed 参数和 $composer.changed() 实现参数未变时跳过执行；通过 updateScope 实现状态变化时的定点重启

3.3 运行时层：Composer + SlotTable + Applier

Section titled “3.3 运行时层：Composer + SlotTable + Applier”

这是 Compose 最核心的部分，由三个协作的组件构成。

SlotTable — 状态的存储

Section titled “SlotTable — 状态的存储”

SlotTable 是一个扁平化的数组结构（而非树），它存储了组合过程中所有的中间状态：remember 的缓存值、上次传入的参数（用于比较是否变化）、节点引用等。

它由两个平行的数组组成：

  • Group 数组：存储结构信息（Key、Size、Parent 等），描述组合树的骨架
  • Slot 数组：存储实际的值（remember 的结果、参数快照等）

SlotTable 的设计选择扁平化数组而非链表或树，是为了获得更好的缓存局部性和更少的内存分配。写入时使用 Gap Buffer 策略——在当前位置维护一个”间隙”，使得顺序写入和局部插入都能保持 O(1) 或接近 O(1) 的效率。

Composer — 过程的指挥者

Section titled “Composer — 过程的指挥者”

Composer 是整个组合/重组过程的协调者。它持有 SlotTable 的引用，在 @Composable 函数执行过程中同步地读写 SlotTable：

  • 初始组合：Composer 依次执行所有 @Composable 函数，将结果写入空的 SlotTable
  • 重组：当状态变化触发时，Composer 带着上次的 SlotTable 重新执行受影响的 @Composable 函数，通过比较新旧值来决定是跳过还是重新执行

Composer 的核心算法可以用一句话概括：用位置（在调用树中的位置）作为身份，用比较（新旧参数）决定是否更新。

Applier — 抽象到具体的桥梁

Section titled “Applier — 抽象到具体的桥梁”

Composer 管理的是抽象的组合逻辑（“应该有什么节点”），而 Applier 负责将这些抽象操作转换为对实际节点树的具体操作：

在 Compose UI 中，UiApplier 实现了这个接口，将 insertBottomUp 转换为向 LayoutNode 树中插入一个 LayoutNode 节点。

3.4 UI 框架层：LayoutNode + Modifier.Node + Coordinator

Section titled “3.4 UI 框架层：LayoutNode + Modifier.Node + Coordinator”

这是实际构建 UI 的层次。

LayoutNode — UI 树的节点

Section titled “LayoutNode — UI 树的节点”

每个可组合的 UI 元素（如 Box、Text、Button）在运行时都对应一个 LayoutNode。它是 Compose UI 树的基本单元，承担三个核心职责：

  1. 布局：通过 MeasurePolicy 进行测量和放置
  2. 绘制：通过 NodeCoordinator 进行渲染
  3. 输入：处理触摸事件和焦点

Modifier.Node — 行为的附加

Section titled “Modifier.Node — 行为的附加”

Modifier 是 Compose 中最核心的设计模式之一。它是一个不可变的链式结构，每个 Element 节点可以附加一种行为（padding、background、clickable 等）。

当组合执行到这个 Box 时，Composer 通过 Applier 将这些 Modifier 转换为 LayoutNode 内部的 NodeChain——一个由 Modifier.Node 实例组成的链表。每个 Node 知道如何执行自己的职责（测量、绘制、处理输入等）。

NodeCoordinator — 布局与绘制的桥梁

Section titled “NodeCoordinator — 布局与绘制的桥梁”

每个 LayoutNode 拥有一条 Coordinator 链（outer → inner），其中每个 Coordinator 对应 NodeChain 中的一个 Modifier.Node。最外层的 Coordinator 作为整个 LayoutNode 的入口，负责：

  • 接收父节点的约束并测量自身
  • 管理绘制顺序和图层
  • 处理坐标变换

五、Compose 与 View 系统的本质区别

Section titled “五、Compose 与 View 系统的本质区别”

5.1 声明式 vs 命令式

Section titled “5.1 声明式 vs 命令式”

View 系统是命令式的：你创建一个 View，然后在状态变化时手动更新它的属性。

Compose 是声明式的：你描述”当前状态对应的 UI 应该是什么样”，框架负责计算差异并更新。

5.2 组合 vs 继承

Section titled “5.2 组合 vs 继承”

View 系统通过继承来组合行为：AppCompatButton extends Button extends TextView extends View。这导致了菱形继承问题和深层继承链。

Compose 通过 Modifier 链来组合行为：Modifier.padding().background().clickable()。每个行为是独立的、可组合的、可复用的。

5.3 单棵树 vs 双棵树

Section titled “5.3 单棵树 vs 双棵树”

View 系统只有一棵 View 树，状态和 UI 直接绑定。

Compose 有两棵树：

  • 组合树（SlotTable 中的 Group 树）：记录组合的逻辑结构
  • UI 树（LayoutNode 树）：实际的布局和绘制节点

Composer 通过 Applier 保持两棵树的同步。这种分离使得重组可以只关注逻辑结构的变化，而不必每次都重建整棵 UI 树。

5.4 全局重绘 vs 精确更新

Section titled “5.4 全局重绘 vs 精确更新”

View 系统中，invalidate() 会导致整个 View 子树重绘。虽然有 requestLayout() 和局部刷新的优化，但粒度较粗。

Compose 中，状态变化可以精确到单个 RecomposeScope（通常是一个函数调用点）。未读取被修改状态的函数完全不会被执行。
