第 5 篇：Composer 与重组 — 组合的核心引擎

深入理解 Composer 如何协调 SlotTable、Snapshot 和 Applier，实现精确的增量重组。从 start() 函数到 ChangeList，从 RecomposeScope 到结构化差异算法。

一、Composer 的角色

Section titled “一、Composer 的角色”

Composer 是 Compose 运行时的核心引擎。它在重组期间执行以下职责：

  1. 遍历 SlotTable：读取上次组合的数据，与当前代码的输出对比
  2. 管理 Group 的生命周期：start() / end() 对应每个 Group 的开始和结束
  3. 检测参数变化：通过 changed() 判断参数是否需要更新
  4. 生成 ChangeList：将需要执行的操作（插入、删除、移动、更新）记录到 ChangeList
  5. 管理 RecomposeScope：将函数注册为可独立重组的单元
  6. 协调 Applier：最终将 ChangeList 中的操作传递给 Applier 执行

6.1 为什么需要 ChangeList？

Section titled “6.1 为什么需要 ChangeList？”

Composer 在执行 @Composable 函数时不直接操作 UI 树。相反，它将所有需要执行的操作记录到 ChangeList 中，等所有 @Composable 函数执行完毕后，再批量执行。

这种设计有两个好处：

  1. 原子性：要么所有变更都生效，要么都不生效
  2. 排序：操作可以按照最优顺序执行（如先插入再删除，避免索引偏移）

十三、关键设计决策与权衡

Section titled “十三、关键设计决策与权衡”

13.1 为什么用 ChangeList 而不是直接操作？

Section titled “13.1 为什么用 ChangeList 而不是直接操作？”

直接操作 UI 树会导致两个问题：

  1. 索引偏移：插入一个 Group 后，后续所有 Group 的索引都变了
  2. 异常安全：如果中间出错，已执行的操作无法回滚

ChangeList 将”计算变更”和”执行变更”分离，使得执行阶段可以按最优顺序批量处理。

13.2 为什么需要 Pending 模式？

Section titled “13.2 为什么需要 Pending 模式？”

简单的逐个比较在列表重排场景下效率极低。Pending 模式通过 key 匹配实现 O(n) 的差异检测，而非 O(n²) 的暴力搜索。

13.3 为什么 RecomposeScope 用 Anchor 而不是 Index？

Section titled “13.3 为什么 RecomposeScope 用 Anchor 而不是 Index？”

因为 SlotTable 的 Gap Buffer 会移动数据，Index 不稳定。Anchor 会随 Gap 的移动自动调整，确保重组闭包总能找到正确的位置。

13.4 为什么重组在主线程执行？

Section titled “13.4 为什么重组在主线程执行？”

Compose 的 UI 更新必须在主线程执行（Android 的 UI 框架要求）。虽然状态变化可能来自任意线程，但重组本身（包括 ChangeList 的生成和执行）都在主线程的协程中完成。
