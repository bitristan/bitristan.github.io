第 3 篇：Compose Compiler Plugin — 声明式到命令式的翻译器

深入理解编译器插件如何将 @Composable 函数转换为运行时可理解的高效代码，包括 Group 生成、跳过优化、重组重启等核心机制。

一、编译器插件的角色

Section titled “一、编译器插件的角色”

Compose 的声明式 UI 写法看起来像普通函数调用，但背后发生的事情远比函数调用复杂。编译器插件（Compose Compiler Plugin）是一个 Kotlin 编译器插件，在编译期将 @Composable 函数转换为携带额外逻辑的代码，使运行时（Composer）能够：

  1. 追踪函数调用的位置，为每次调用生成唯一的位置标识
  2. 管理 SlotTable 的读写，通过 Group 标记结构边界
  3. 实现跳过优化，在参数未变时跳过整个函数体的执行
  4. 支持重组重启，将函数体注册为可被精确重新执行的闭包

编译器插件的源码位于 Kotlin 仓库（jetbrains/compose-compiler），而非 AndroidX。但我们可以通过理解它生成的代码模式和它调用的运行时 API 来完整理解它的行为。

二、@Composable 注解的本质

Section titled “二、@Composable 注解的本质”

@Composable 不仅仅是一个注解——它改变了函数的签名和调用约定。

Section titled “2.1 隐式参数注入”

每个 @Composable 函数在编译后都会被注入一个隐式的 $composer 参数和一个 $changed 位掩码参数：

$composer 是 Composer 的实例，是整个运行时的核心接口。$changed 是一个位掩码，编码了每个参数在本次重组中是否发生了变化。

2.2 调用约定

Section titled “2.2 调用约定”

当一个 @Composable 函数调用另一个 @Composable 函数时，调用方负责计算被调用方的 $changed 值：

位掩码的低若干位表示参数是否”不确定”（$changed and 0b0000_0100 != 0），中间位表示参数是否”相同”（$changed and 0b0000_0001 == 0）。这种位编码避免了额外的方法调用开销。

三、Group：组合的结构骨架

Section titled “三、Group：组合的结构骨架”

3.1 Group 的概念

Section titled “3.1 Group 的概念”

Group 是 Composer 中最核心的抽象之一。它是一个逻辑分组，代表组合中的一个结构单元。每个 Group 有：

  • Key：由编译器基于源码位置生成的唯一整数
  • Slots：Group 内部存储的数据（参数、状态、节点引用等）
  • Children：嵌套的子 Group

3.2 Group 的类型

Section titled “3.2 Group 的类型”

编译器插件根据不同的代码结构生成不同类型的 Group：

Group 类型生成场景关键特性
startRestartGroup / endRestartGroup可重组的函数可被精确重新执行，支持增量更新
startReplaceableGroup / endReplaceableGroup条件分支（if/when）不能移动，只能插入/删除/替换
startReplaceGroup / endReplaceGroup条件分支（新版编译器）比 Replaceable 更严格，强制替换
startMovableGroup / endMovableGroupkey() 块可以随 key 移动

startReusableNode / endReusableNode可复用节点支持节点在不同位置复用
startDefaults / endDefaults默认参数块标记默认参数计算的边界
startReusableGroup / endReusableGroupReusableContent标记可复用内容的边界

3.3 Key 的生成

Section titled “3.3 Key 的生成”

Key 是编译器基于调用点的源码位置生成的整数。它在编译期确定，在运行期恒定不变。Key 的作用是让 Composer 在重组时能够将当前的 Group 与上次的 Group 对应起来。

对于同一个调用点，Key 始终相同。但当代码结构改变（如 if 分支切换）时，不同分支的 Key 不同，Composer 由此判断结构发生了变化。

四、RestartGroup 与重组机制

Section titled “四、RestartGroup 与重组机制”

4.1 RestartGroup 的生命周期

Section titled “4.1 RestartGroup 的生命周期”

startRestartGroup / endRestartGroup 是编译器插件生成的最重要的 Group 对。它包围整个 @Composable 函数体，使函数成为一个可独立重组的单元。

4.2 updateScope 的作用

Section titled “4.2 updateScope 的作用”

endRestartGroup() 返回一个 ScopeUpdateScope，其 updateScope 方法接受一个 lambda。这个 lambda 就是重组闭包——当某个状态变化导致这个 Scope 被 invalidate 时，Composer 会调用这个闭包来重新执行函数。

block 的签名 (Composer, Int) -> Unit 正好对应 @Composable 函数注入 $composer 和 $changed 参数后的签名。这就是为什么 Compose 可以精确地重新执行单个函数，而不是整个 UI 树。

4.3 $changed 参数的编码

Section titled “4.3 $changed 参数的编码”

$changed 参数是一个 Int 位掩码，每一位编码了函数参数的变化状态。编译器为每个参数分配若干位：

  • 第 0 位组：参数是否”确定”（definitely），即我们知道参数没有变化
  • 第 1 位组：参数是否”相同”（same），即参数值与上次相同
  • 第 2 位组：参数是否”不确定”（uncertain），即调用方无法确定参数是否变化

当一个参数被传入 remember {} 或来自 State.value 时，它的变化状态是”确定”的；当参数来自外部调用方时，它的变化状态是”不确定”的，需要通过 composer.changed() 运行时检查。

五、跳过优化（Skip Optimization）

Section titled “五、跳过优化（Skip Optimization）”

5.1 核心思想

Section titled “5.1 核心思想”

在重组过程中，如果一个 @Composable 函数的所有参数都没有变化，那么它的输出也不会变化——此时可以直接跳过函数体的执行，复用上次的结果。这是 Compose 最重要的性能优化之一。

5.3 $composer.skipping 属性

Section titled “5.3 $composer.skipping 属性”

skipping 属性反映了当前 Composer 是否允许跳过。在大多数情况下它是 true，但在某些特殊场景下会变为 false：

  • 当静态 CompositionLocal（staticCompositionLocalOf）的值发生变化时，所有消费该 Local 的函数都不能跳过
  • 当 ReusableContent 的 key 变化时，处于复用状态的函数不能跳过

5.4 skipToGroupEnd() 的实现

Section titled “5.4 skipToGroupEnd() 的实现”

当决定跳过时，skipToGroupEnd() 告诉 Composer 直接跳到当前 RestartGroup 的末尾。SlotTable 的读取器会跳过所有子 Group 的 slots，但不会清除它们——上次的值仍然保留在 SlotTable 中，供后续重组使用。

关键逻辑：

9.3 Updater.set 的差异更新

Section titled “9.3 Updater.set 的差异更新”
  • 首次创建：inserting 为 true，直接设置值并记录
  • 重组时：检查新值是否与 SlotTable 中缓存的旧值相同，只有不同才更新
  • composer.apply(value, block) 将更新操作排入队列，等组合完成后由 Applier 批量执行

10.2 Lambda 的稳定性与跳过

Section titled “10.2 Lambda 的稳定性与跳过”

编译器会分析 Lambda 是否”稳定”（Stable）。如果一个 Lambda 的所有捕获变量都是不可变的或 @Stable 的，编译器可以将其标记为稳定的，从而使包含它的函数可以被跳过。

如果 Lambda 捕获了可变状态（如 MutableState），则包含它的函数不能被跳过——因为即使函数参数不变，Lambda 的行为也可能不同。

十五、关键设计决策与权衡

Section titled “十五、关键设计决策与权衡”

15.1 为什么 Key 基于位置而非内容？

Section titled “15.1 为什么 Key 基于位置而非内容？”

基于位置的 Key 在编译期确定，无需运行时计算。它假设代码结构在运行期不变（if 分支的代码位置不会变），这在绝大多数情况下成立。对于列表等需要内容 Key 的场景，通过 key() 函数提供显式 Key。

15.2 为什么不使用 Kotlin 的默认参数？

Section titled “15.2 为什么不使用 Kotlin 的默认参数？”

$changed 和 $default 参数不能用 Kotlin 的默认参数语法，因为它们需要在调用链中被精确传递和计算。使用默认参数会破坏位掩码的编码方案。

15.3 为什么 skipToGroupEnd 而不直接 return？

Section titled “15.3 为什么 skipToGroupEnd 而不直接 return？”

skipToGroupEnd 跳过的是 SlotTable 中的 slots，而不是函数执行。Composer 需要正确推进 SlotTable 的读取位置，以便后续的 Group（如 endRestartGroup）能正确匹配。如果直接 return，SlotTable 的位置会错乱。

15.4 为什么需要 skipping 属性？

Section titled “15.4 为什么需要 skipping 属性？”

某些场景下（如静态 CompositionLocal 变化），即使函数参数没变，函数也需要重新执行。skipping 属性为 Composer 提供了一个”全部不跳过”的开关，确保这些场景的正确性。
