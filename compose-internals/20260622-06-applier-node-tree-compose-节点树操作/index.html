第 6 篇：Applier 与节点树 — 将组合结果转化为 UI 操作

理解 Applier 如何作为 Composer 与 UI 树之间的桥梁，将 ChangeList 中的抽象操作转化为具体的树结构变更。从 Applier 接口到 LayoutNode，从插入策略到节点生命周期。

一、Applier 的角色

Section titled “一、Applier 的角色”

Composer 在重组过程中不直接操作 UI 树。它将所有需要执行的操作记录到 ChangeList 中，然后由 Applier 负责将这些操作应用到实际的 UI 树上。

这种分离设计使得：

  1. Composer 专注于逻辑：只需关注 SlotTable 的读写和差异检测
  2. Applier 专注于树操作：只需关注如何高效地修改目标树结构
  3. 可替换的树类型：不同的 Applier 实现可以驱动不同类型的树（UI 树、Vector 图形树等）

2.2 insertTopDown vs insertBottomUp

Section titled “2.2 insertTopDown vs insertBottomUp”

这是 Applier 中最有趣的设计之一。同一个节点插入操作有两个时机：

Top-down（自顶向下）：先插入 B 到 R，再插入 A、C 到 B

Bottom-up（自底向上）：先构建 A、C 到 B，再将 B 插入 R

选择哪种策略取决于树的实现：

  • Bottom-up：Android 的 View 系统、LayoutNode（避免重复通知）
  • Top-down：某些需要父节点先存在的树结构

7.1 操作类型

Section titled “7.1 操作类型”

ChangeList 中的操作分为两类：

SlotTable 操作（在 SlotWriter 上执行）：

  • UpdateValue：更新 slot 值
  • InsertSlots：插入新的 slot 组
  • RemoveCurrentGroup：删除当前 Group
  • MoveCurrentGroup：移动当前 Group

Applier 操作（在 Applier 上执行）：

  • InsertNodeFixup：创建并插入节点
  • PostInsertNodeFixup：完成节点插入（bottom-up 阶段）
  • UpdateNode：更新节点属性
  • RemoveNode：移除节点
  • MoveNode：移动节点
  • Downs / Ups：遍历树

十一、关键设计决策

Section titled “十一、关键设计决策”

11.1 为什么需要两个插入方法（TopDown/BottomUp）？

Section titled “11.1 为什么需要两个插入方法（TopDown/BottomUp）？”

不同的树结构有不同的最优插入策略。Android 的 View 系统在子节点进入时会通知父节点，如果使用 top-down，通知次数是指数级的；使用 bottom-up 则是线性的。

11.2 为什么节点插入需要两阶段？

Section titled “11.2 为什么节点插入需要两阶段？”

因为 Composer 在遇到 ComposeNode 时，节点的插入位置可能还不确定（Pending 模式中的移动操作尚未完成）。两阶段设计允许先创建节点，等位置确定后再插入。

11.3 为什么 UiApplier 在 onEndChanges 中通知 Owner？

Section titled “11.3 为什么 UiApplier 在 onEndChanges 中通知 Owner？”

因为 onEndChanges 标志着所有变更已应用完毕，此时 Owner（通常是 AndroidComposeView）可以安全地触发布局和绘制。

11.4 为什么需要 OffsetApplier？

Section titled “11.4 为什么需要 OffsetApplier？”

子组合的节点需要插入到父树的特定位置。OffsetApplier 通过偏移量调整，使得子组合的节点操作能够正确定位到父树中的目标位置。
