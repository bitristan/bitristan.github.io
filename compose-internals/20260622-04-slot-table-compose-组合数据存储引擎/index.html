第 4 篇：SlotTable — 组合数据的底层存储引擎

深入理解 Compose 如何用两个数组和 Gap Buffer 高效存储整个 UI 树的结构信息、状态数据和节点引用。

一、SlotTable 的角色

Section titled “一、SlotTable 的角色”

在第 3 篇中我们看到，编译器插件为每个 @Composable 函数生成 Group 调用（startRestartGroup、startNode 等）。这些 Group 的数据存储在哪里？答案就是 SlotTable。

SlotTable 是 Composer 的持久化存储。它记录了：

  • 组合的结构（哪些 Group、嵌套关系）
  • 每个 Group 的参数值（用于比较参数是否变化）
  • remember 缓存的值
  • 节点引用（如 LayoutNode）
  • CompositionLocal 的值

在重组过程中，Composer 通过 SlotTable 的读写来判断哪些部分需要更新。

SlotTable 的核心是两个数组：

2.1 Groups 数组

Section titled “2.1 Groups 数组”

groups 是一个 IntArray，每 5 个连续元素构成一个 Group 的”字段组”：

五个字段的含义：

偏移字段含义

2.2 GroupInfo 的位编码

Section titled “2.2 GroupInfo 的位编码”

GroupInfo 是一个 32 位整数，高位用于标志位，低位用于节点计数：

节点计数的范围是 26 位（最大约 6700 万），足以覆盖任何实际的 UI 树。

2.3 Slots 数组

Section titled “2.3 Slots 数组”

slots 是一个 Array<Any?>，存储 Group 的实际数据。每个 Group 在 slots 中的布局是：

  • ObjectKey：可选，存储 key() 函数传入的对象
  • Node：可选，存储节点引用（如 LayoutNode）
  • Aux：可选，存储辅助数据（如 CompositionLocal 的 map）
  • Slot₀, Slot₁, …：存储 remember 的值、函数参数的缓存值等

一个 Group 的 slots 从 dataAnchor 开始，到下一个 Group 的 slots 起始位置结束。

三、Gap Buffer：高效的插入与删除

Section titled “三、Gap Buffer：高效的插入与删除”

3.1 为什么需要 Gap Buffer？

Section titled “3.1 为什么需要 Gap Buffer？”

SlotTable 需要频繁地在中间位置插入和删除 Group（比如 if 分支切换、列表项增删）。如果使用普通数组，每次插入/删除都需要移动后续所有元素，时间复杂度 O(n)。

Gap Buffer 的核心思想：在数组中预留一个”间隙”（gap），插入时直接写入间隙位置，删除时扩大间隙。这使得连续的插入操作（重组时的常见模式）几乎 O(1)。

3.4 Reader vs Writer 的 Gap 策略

Section titled “3.4 Reader vs Writer 的 Gap 策略”
  • Reader：Gap 始终在数组末尾。这意味着 Index == Address，读取时不需要任何地址转换，效率最高。
  • Writer：Gap 跟随写入位置移动。Writer 需要维护 groupIndexToAddress() 映射，将逻辑 Index 转换为物理 Address。

七、Anchor：稳定的位置引用

Section titled “七、Anchor：稳定的位置引用”

7.1 问题：Gap 移动导致 Index 失效

Section titled “7.1 问题：Gap 移动导致 Index 失效”

当 Gap 移动时，Gap 后面的所有 Group 的 Index 都会改变。但外部持有 Group 位置引用的代码（如 RecomposeScope）需要一个稳定的位置标识。

7.2 Anchor 的设计

Section titled “7.2 Anchor 的设计”

Anchor 是一个轻量级的位置追踪器：

当 Gap 移动时，SlotWriter 会更新所有受影响的 Anchor 的 location。当 Group 被删除时，对应的 Anchor 被标记为无效（location = Int.MIN_VALUE）。

7.3 Anchor 的优化

Section titled “7.3 Anchor 的优化”

SlotWriter 只更新 Gap 移动时”跨越”的 Anchor。如果一个 Anchor 指向的 Group 在 Gap 移动范围之外，它不需要更新。这使得 Anchor 的维护开销与 Gap 移动的距离成正比，而非 Anchor 的总数。

八、并发控制：Reader 与 Writer 的互斥

Section titled “八、并发控制：Reader 与 Writer 的互斥”

8.1 读写规则

Section titled “8.1 读写规则”
  • 可以同时有多个 Reader（重组期间的布局和绘制可以并行）
  • 同一时间只能有一个 Writer
  • Writer 活跃时不能有 Reader
  • Reader 活跃时不能创建 Writer

8.2 Version 检测

Section titled “8.2 Version 检测”

SlotTable 维护一个 version 计数器，每次创建 Writer 时递增。Reader 在创建时记录当时的 version，如果在读取过程中 version 变了，说明有并发修改，需要抛出异常。

十、LinkBuffer：新型存储实现

Section titled “十、LinkBuffer：新型存储实现”

10.1 两套实现

Section titled “10.1 两套实现”

当前代码库中存在两套 SlotTable 实现：

  • GapBuffer：传统实现（composer/gapbuffer/），4200+ 行

10.2 LinkBuffer 的设计差异

Section titled “10.2 LinkBuffer 的设计差异”

LinkBuffer 不使用连续数组 + Gap 的方式，而是使用地址空间（SlotTableAddressSpace）来管理 Group：

LinkBuffer 的 Group 使用 GroupHandle 来引用，而非直接的数组 Index。这种方式在频繁的中间插入和删除操作时可能有更好的性能特征，因为不需要移动大量数据来维护 Gap。

Section titled “10.3 选择策略”

两套实现共存表明 Compose 团队正在进行架构演进。GapBuffer 是经过验证的成熟实现，LinkBuffer 是面向未来的优化方向。

十一、关键设计决策与权衡

Section titled “十一、关键设计决策与权衡”

11.1 为什么用 IntArray 而非对象数组存储 Group 信息？

Section titled “11.1 为什么用 IntArray 而非对象数组存储 Group 信息？”

Group 的 5 个字段中，4 个是整数（Key、ParentAnchor、Size、DataAnchor），1 个是编码的标志位（GroupInfo）。使用 IntArray 避免了对象分配和 GC 压力。在 Compose 中，Group 的数量可能达到数千甚至数万，使用原始类型数组的性能优势显著。

11.2 为什么 slots 用 Array<Any?> 而非类型化数组？

Section titled “11.2 为什么 slots 用 Array<Any?> 而非类型化数组？”

Slot 存储的值类型多样（String、Int、Float、对象引用等），必须使用 Any? 类型。基本类型的装箱开销是不可避免的，但 Compose 通过 changed(Int) 等重载方法尽量减少运行时的装箱。

11.3 为什么 Reader 把 Gap 放在末尾？

Section titled “11.3 为什么 Reader 把 Gap 放在末尾？”

Reader 的主要操作是顺序遍历，Gap 在末尾意味着遍历过程中不需要任何地址转换。这是一个典型的”读优化”设计——写入时移动 Gap 的开销由 Writer 承担，读取时享受零开销。

11.4 为什么需要 Anchor 而不是直接存储 Index？

Section titled “11.4 为什么需要 Anchor 而不是直接存储 Index？”

因为 Gap 的移动会改变 Index。如果 RecomposeScope 直接存储 Group 的 Index，每次 Gap 移动都需要更新所有 Scope。Anchor 的设计使得只有跨越 Gap 边界的 Group 才需要更新，大大减少了维护开销。
