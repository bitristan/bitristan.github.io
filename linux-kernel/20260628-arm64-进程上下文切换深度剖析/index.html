1. 概述：什么是上下文切换

Section titled “1. 概述：什么是上下文切换”

在操作系统中，当调度器决定将 CPU 从进程 A 切换到进程 B 时，内核必须完整地保存 A 的执行现场，并恢复 B 的执行现场。这个过程称为上下文切换（Context Switch）。

ARM64 架构的上下文切换涉及两个层面的工作：

  1. 地址空间切换（Address Space Switch）：切换页表基地址寄存器（TTBR），使 CPU 使用新进程的虚拟地址空间映射。
  2. 执行状态切换（Execution State Switch）：保存/恢复通用寄存器、系统寄存器、浮点/SIMD 状态、安全扩展状态等。

这看似简单的操作，背后涉及的硬件机制和软件设计却极其精妙。下面我们从硬件到软件逐层展开。

2. ARM64 硬件基础

Section titled “2. ARM64 硬件基础”

2.1 异常级别模型（Exception Level）

Section titled “2.1 异常级别模型（Exception Level）”

ARM64 架构定义了四个异常级别：

  • EL0：用户态应用程序运行在此级别，权限最低。
  • EL1：操作系统内核运行在此级别，拥有完整的系统管理权限。
  • EL2：虚拟化层（Hypervisor），管理虚拟机。Linux KVM 的 hypervisor 代码运行在此级别。
  • EL3：安全监控器（Secure Monitor），管理 ARM TrustZone 的安全世界与普通世界之间的切换，拥有最高权限。

上下文切换发生在 EL1（内核态），但切换完成后，进程恢复执行的位置可能是 EL0（用户态进程）或 EL1（内核线程）。

VHE（Virtualization Host Extension） 是 ARMv8.1 引入的扩展，允许 Linux 内核直接运行在 EL2。当 VHE 启用时，内核使用的系统寄存器后缀从 _el1 变为 _el2，但上下文切换的核心逻辑不变。

2.2 寄存器架构

Section titled “2.2 寄存器架构”

ARM64 的寄存器体系是理解上下文切换的关键。

通用寄存器（General Purpose Registers）

Section titled “通用寄存器（General Purpose Registers）”

ARM64 拥有 31 个 64 位通用寄存器：x0 - x30。

根据 AArch64 调用约定（AAPCS64），它们分为两类：

类型寄存器说明
Caller-saved（调用者保存）x0-x18函数可以自由使用，调用前由调用者保存
Callee-saved（被调用者保存）x19-x28函数如果使用，必须在返回前恢复原值
帧指针x29 (FP)栈帧指针
链接寄存器x30 (LR)函数返回地址
栈指针SP栈指针

关键设计决策：由于上下文切换发生在函数调用的边界（调度器调用 __switch_to() 是一个普通 C 函数调用），编译器会自动保存 caller-saved 寄存器（x0-x18）在栈上。因此，上下文切换只需保存 callee-saved 寄存器（x19-x28, FP, SP, LR），这大幅减少了需要保存的寄存器数量。

特殊寄存器

Section titled “特殊寄存器”
寄存器用途
SP_EL0ARM64 内核用此寄存器存放当前进程的 task_struct 指针（即 current 宏的实现）
SP_EL1内核栈指针（异常处理时使用）
PC程序计数器（通过 LR/ERET 间接控制）
PSTATE处理器状态，包含条件标志、异常掩码、处理器模式等

SIMD/FP 寄存器

Section titled “SIMD/FP 寄存器”

ARM64 提供了 32 个 128 位 SIMD/FP 寄存器：q0-q31（也被称为 v0-v31），以及控制寄存器 FPSR（浮点状态）和 FPCR（浮点控制）。

总大小：32 × 16 + 8 = 520 字节。

SVE 可扩展向量寄存器

Section titled “SVE 可扩展向量寄存器”

SVE（Scalable Vector Extension）将向量寄存器扩展为可变长度（128-2048 位硬件实现）：

  • z0-z31：可扩展向量寄存器（低 128 位与 q0-q31 共享）
  • p0-p15：谓词寄存器
  • FFR：First Fault 寄存器

TTBR1_EL1内核空间页表基地址。所有进程共享同一个内核地址空间，通常不变

2.4 地址空间与页表机制

Section titled “2.4 地址空间与页表机制”

ARM64 使用分离的双页表设计：

硬件通过虚拟地址的高位自动选择使用哪个 TTBR：

  • 高位全 0 → 使用 TTBR0_EL1（用户空间映射）
  • 高位全 1 → 使用 TTBR1_EL1（内核空间映射）

这意味着切换用户地址空间只需修改 TTBR0_EL1，而内核映射始终通过 TTBR1_EL1 访问，无需修改。

2.5 ASID（Address Space ID）机制

Section titled “2.5 ASID（Address Space ID）机制”

ASID 是 ARM64 实现高效地址空间切换的核心硬件机制。

为什么需要 ASID

Section titled “为什么需要 ASID”

如果没有 ASID，每次切换进程都必须刷新整个 TLB，因为不同进程的相同虚拟地址可能映射到不同的物理地址。TLB 刷新代价极高——意味着所有后续内存访问都要重新走页表遍历，可能造成数百到数千个周期的性能损失。

ASID 为每个地址空间分配一个标签，TLB 条目中同时缓存了虚拟地址和 ASID。当进程 A（ASID=1）访问地址 0x400000 时，TLB 命中的是 {VA=0x400000, ASID=1} 的条目，不会与进程 B（ASID=2）的 {VA=0x400000, ASID=2} 冲突。

ASID 位数

Section titled “ASID 位数”

由 ID_AA64MMFR0_EL1.ASIDBits 字段决定：

  • 8-bit：256 个 ASID（ARMv8 基线）
  • 16-bit：65536 个 ASID（ARMv8.1 扩展，主流芯片均支持）

ASID 与 TTBR 的关系

Section titled “ASID 与 TTBR 的关系”

ARM64 的 ASID 存储在 TTBR 寄存器的高位：

TCR_EL1.A1 位控制 ASID 的来源：

  • A1=1：ASID 来自 TTBR1_EL1（Linux 默认配置）
  • A1=0：ASID 来自 TTBR0_EL1

Linux 设置 A1=1，即 ASID 由 TTBR1_EL1 指定。因此在 cpu_do_switch_mm() 中，必须同时更新 TTBR1_EL1 和 TTBR0_EL1 中的 ASID 值。

2.6 TLB 与缓存一致性

Section titled “2.6 TLB 与缓存一致性”

ARM64 提供了丰富的 TLB 无效化指令，关键指令包括：

指令说明
TLBI VMALLE1IS无效化 EL1 所有 TLB 条目（Inner Shareable 广播）
TLBI ASIDE1IS, val按 ASID 无效化 EL1 TLB 条目
TLBI VAE1IS, val按虚拟地址 + ASID 无效化
TLBI RVAE1IS, val按范围无效化（ARMv8.4-TLBIRANGE 扩展）

IS 后缀表示 Inner Shareable——操作会广播到同一 Inner Shareable 域中的所有 CPU 核心，硬件自动完成跨核 TLB 一致性维护，无需软件发送 IPI。

DSB ISH 指令确保 TLB 无效化操作在所有核心上完成后才继续执行。

3.1 task_struct 与 thread_struct

Section titled “3.1 task_struct 与 thread_struct”

每个进程/线程在内核中由 task_struct 描述。其中与架构相关的部分嵌套在 thread_struct 中。

3.2 cpu_context：硬件寄存器快照

Section titled “3.2 cpu_context：硬件寄存器快照”

这是上下文切换的核心数据结构。它恰好保存了 AAPCS64 规定的 callee-saved 寄存器。注意它不保存 x0-x18、PSTATE、FPSIMD 等——这些要么由编译器在函数调用时自动管理（x0-x18），要么由单独的子系统负责切换（FPSIMD）。

cpu_context 是 thread_struct 的第一个成员，所以 offsetof(struct task_struct, thread.cpu_context) 就是汇编中使用的 THREAD_CPU_CONTEXT 常量。

5.3 指针认证（Pointer Authentication）

Section titled “5.3 指针认证（Pointer Authentication）”

ARMv8.3 引入的 PAC（Pointer Authentication Code）机制在函数返回地址中嵌入加密签名，防止 ROP（Return-Oriented Programming）攻击。

上下文切换涉及两组密钥：

  • 内核密钥（APIA）：在 cpu_switch_to 汇编中切换，保护内核栈
  • 用户密钥：在 ptrauth_thread_switch_user() 中切换，保护用户栈

通过 ARM64 的 alternative 机制，如果 CPU 不支持 PAC，这些指令会被替换为 NOP，零开销。

5.4 影子调用栈（Shadow Call Stack）

Section titled “5.4 影子调用栈（Shadow Call Stack）”

影子调用栈（SCS）利用 ARM64 预留的 x18 寄存器作为指向影子栈的指针。影子栈只存放返回地址，与普通栈分离，攻击者无法通过栈缓冲区溢出来篡改影子栈上的返回地址。

这也是 x18 在 ARM64 Linux 中被 ABI 保留的原因——用户态编译器不会使用 x18，它始终是 SCS 指针。

5.5 MTE（内存标签扩展）

Section titled “5.5 MTE（内存标签扩展）”

MTE（Memory Tagging Extension）是 ARMv8.5 引入的硬件内存安全特性。它为每个 16 字节的内存颗粒分配一个 4 位标签，指针的高位也携带标签。访问时硬件自动检查标签是否匹配。

MTE 切换必须在 DSB（数据同步屏障）之后执行，因为 DSB 保证了所有之前的异步标签故障已经被记录到 TFSR_EL1 寄存器中。

5.6 GCS（Guarded Control Stack）

Section titled “5.6 GCS（Guarded Control Stack）”

GCS（FEAT_GCS）是 ARMv9.2 引入的可选硬件影子调用栈特性（通过 ID_AA64PFR1_EL1.GCS 字段检测）。与软件 SCS 不同，GCS 由硬件自动管理——每次 BL（函数调用）和 RET（返回）指令都会自动在 GCS 上压入/弹出返回地址。

6. KPTI 与安全机制对上下文切换的影响

Section titled “6. KPTI 与安全机制对上下文切换的影响”

KPTI（Kernel Page Table Isolation）

Section titled “KPTI（Kernel Page Table Isolation）”

KPTI（CVE-2017-5754 / Meltdown 缓解）将内核地址空间从用户态页表中移除，仅保留少量”蹦床”代码的映射。

对上下文切换的影响：

  1. ASID 分配：KPTI 启用时，每个地址空间使用两个 ASID——偶数号保留给内核映射，奇数号用于用户映射。check_and_switch_context() 中的 set_kpti_asid_bits() 函数通过 memset(map, 0xaa, len) 在 ASID 位图中标记所有偶数 ASID 为已占用（0xaa = 二进制 10101010，所有偶数位为 1）。

  2. 异常入口/出口：从用户态进入内核时，蹦床代码必须先切换到完整的内核页表（通过修改 TTBR1）：

  1. ASID 格式：KPTI 下的 ASID 最低位区分内核/用户：
    • ASID | 0 = 内核态映射
    • ASID | 1 = 用户态映射
