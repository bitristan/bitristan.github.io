1. 概述与设计动机

Section titled “1. 概述与设计动机”

在多处理器（SMP）系统中，多个 CPU 可能同时访问共享数据，这通常需要使用锁机制（如自旋锁、互斥锁）来保证一致性。然而，锁会引入竞争（contention），降低系统的可扩展性。

Per-CPU 变量的核心思想极其简单：为每个 CPU 提供一份数据的独立副本。由于每个 CPU 只访问自己的副本，不需要任何锁，天然避免了竞争条件。这是一种以空间换时间的策略。

Per-CPU 机制的典型应用场景包括：

  • 统计计数器：如网络收发包计数、进程调度统计，每个 CPU 独立更新，需要汇总时再遍历所有 CPU
  • CPU 本地缓存：如 slab 分配器的 per-CPU 缓存、页面分配器的 per-CPU 页面集
  • 当前任务指针：current 宏在 x86_64 上就是通过 per-CPU 变量 pcpu_hot.current_task 实现的
  • 中断统计：每个 CPU 独立维护自己的中断计数

Per-CPU 变量的主要优势：

  • 无锁访问：本 CPU 访问自己的副本无需任何同步
  • 缓存友好：每个 CPU 的副本在自己的缓存行中，避免 false sharing
  • 可扩展性：CPU 越多，并发度越高，优势越明显

2. 基本原理

Section titled “2. 基本原理”

Per-CPU 变量的核心原理可以用一句话概括：同一变量名，不同 CPU 看到不同的内存地址。

在编译时，per-CPU 变量被放置到特殊的链接器段 .data..percpu 中，其地址是一个基准地址（base address）。在运行时，每个 CPU 通过加上自己的**偏移量（offset）**来定位自己的副本。

关键数据结构：

  • __per_cpu_offset[]：每个 CPU 的偏移量数组（声明大小为 NR_CPUS，实际使用 nr_cpu_ids 个条目，nr_cpu_ids ≤ NR_CPUS）
  • pcpu_base_addr：第一个 chunk 的基地址（对应链接器符号 __per_cpu_start 的运行时地址）
  • __per_cpu_start / __per_cpu_end：链接器定义的静态 per-CPU 段起止符号

在代码中，per-CPU 指针带有 __percpu 标记（通过 sparse 工具进行地址空间检查），以防止误用：

当使用 sparse 静态分析工具时（__CHECKER__ 被定义），__percpu 展开为 address_space(__percpu) 注解，让 sparse 能够检测到直接解引用 per-CPU 指针（而不经过 per-CPU 访问接口）等错误。在正常编译路径下，__percpu 展开为 BTF_TYPE_TAG(percpu)，为 BPF Type Format（BTF）提供类型标记，使 BPF 验证器和调试工具能够识别 per-CPU 指针。

3.3 不同变体

Section titled “3.3 不同变体”

根据不同的需求，内核提供了多种 per-CPU 变量变体：

DEFINE_PER_CPU_SHARED_ALIGNED

Section titled “DEFINE_PER_CPU_SHARED_ALIGNED”

在 SMP 配置下，变量被放置在 .data..percpu..shared_aligned 子段，并且强制 cacheline 对齐。这用于虽然每个 CPU 有独立副本，但可能被其他 CPU 读取的场景（如 /proc 统计信息的汇总），避免不同 CPU 的数据共享同一缓存行导致的 false sharing。

DEFINE_PER_CPU_ALIGNED

Section titled “DEFINE_PER_CPU_ALIGNED”

变量强制 cacheline 对齐，无论是否为 SMP 配置。适用于对齐要求严格的场景。

DEFINE_PER_CPU_PAGE_ALIGNED

Section titled “DEFINE_PER_CPU_PAGE_ALIGNED”

变量页对齐，放置在 .data..percpu..page_aligned 子段。适用于需要页对齐的数据结构（如某些页表项）。

DEFINE_PER_CPU_READ_MOSTLY

Section titled “DEFINE_PER_CPU_READ_MOSTLY”

放置在 .data..percpu..read_mostly 子段，标记为”主要读取”。内核可以将这些变量放在更适合读取的内存区域。

DEFINE_PER_CPU_FIRST

Section titled “DEFINE_PER_CPU_FIRST”

放置在 .data..percpu..first 子段，确保在所有 per-CPU 变量之前。

DEFINE_PER_CPU_DECRYPTED

Section titled “DEFINE_PER_CPU_DECRYPTED”

用于 AMD 内存加密（SME/SEV）环境，标记变量在 guest 中需要以解密方式访问。

4.2 Per-CPU 分配器架构

Section titled “4.2 Per-CPU 分配器架构”

Per-CPU 分配器的设计灵感来自 slab 分配器，但专门针对 per-CPU 的特殊需求进行了优化。

核心概念

Section titled “核心概念”
  • Chunk（块）：per-CPU 分配器的基本管理单元，包含一组连续的内存区域
  • Unit（单元）：chunk 内划分的等大小区域，每个 unit 对应一个 possible CPU
  • Group（组）：按 NUMA 节点关系将 units 分组

First Chunk 的特殊结构

Section titled “First Chunk 的特殊结构”

第一个 chunk 始终存在，具有特殊的内部布局：

  • Static 区域：存放内核编译时定义的 .data..percpu 段内容
  • Reserved 区域：为模块的静态 per-CPU 变量预留空间
  • Dynamic 区域：用于运行时动态分配

分配策略

Section titled “分配策略”

分配器将 chunks 组织到不同的**槽位（slot）**链表中，按最大连续空闲区域大小排序（完全空闲的 chunks 放入 pcpu_free_slot）。分配时从最满的 chunk 开始尝试（类似 slab 的思路），以减少内存碎片。完整的分配流程详见 4.4 节。

Memcg 感知分配

Section titled “Memcg 感知分配”

分配器支持通过 __GFP_ACCOUNT 标志进行内存控制组（memcg）感知分配。分配器维护两组独立的 chunk 链表：

  • 已记账（accounted）链表：存放带有 __GFP_ACCOUNT 标志的分配

并发控制

Section titled “并发控制”

分配器使用两级锁保护内部数据结构：

  • pcpu_lock（自旋锁）：保护轻量级元数据操作，包括 slot 链表的增删移动、alloc_map/bound_map 位图更新、块元数据维护。通常以 irq-safe 方式获取（分配/释放路径使用 spin_lock_irqsave，balance 工作使用 spin_lock_irq），因此分配和释放操作可以在中断上下文中安全执行。单次临界区较短（主要是位图扫描和标记），实际锁竞争通常有限。
  • pcpu_alloc_mutex（互斥锁）：保护重量级操作，包括 chunk 的创建与销毁、页面的 populate/depulate、以及 map 扩展。这些操作可能涉及内存分配（vmalloc/memblock）和睡眠，因此使用 mutex 而非 spinlock。由于这些操作不在分配热路径上（通常由后台 balance 工作或首次分配触发），对整体性能影响较小。

这种两级锁设计将高频的元数据操作与低频的结构变更解耦，避免了重操作阻塞轻操作。

this_cpu_* — 中断安全

Section titled “this_cpu_* — 中断安全”

提供中断安全性。在通用实现中，this_cpu_* 的写和 RMW 操作通过 raw_local_irq_save/restore 禁用本地中断来保证不被中断处理程序干扰（raw_ 前缀版本不触发 lockdep/irq 追踪等调试钩子，开销更小）。但读操作有两条路径（this_cpu_generic_read）：

  • 当变量大小是原生字长（__native_word(pcp) 为真，如 4 字节或 8 字节）时，仅使用 preempt_disable_notrace() 禁止抢占，无需关中断——因为对原生字长的单次加载本身就是原子的，即使中断发生并修改了该变量，加载指令读到的仍然是中断前或中断后的完整值，不会出现”撕裂读”
  • 当变量大小非原生字长时，使用 raw_local_irq_save/restore 禁用本地中断，防止多字加载被中断打断导致数据不一致

在 x86 等有架构优化的平台上，可能使用单条原子指令实现，无需禁用中断。

6.4 UP 配置的退化

Section titled “6.4 UP 配置的退化”

在单处理器（!CONFIG_SMP）配置下，per-CPU 机制会大幅退化，所有访问退化为对普通全局变量的操作：

  • per_cpu_ptr(ptr, cpu) 忽略 cpu 参数，直接返回原始指针
  • this_cpu_ptr 和 raw_cpu_ptr 等价，无任何额外开销
  • this_cpu_* / raw_cpu_* / __this_cpu_* 的通用回退实现退化为普通的变量读写和算术操作
  • 不存在偏移量数组、段寄存器等机制，整个 per-CPU 区域只有一份数据

这意味着在 UP 内核上，per-CPU 变量本质上就是普通全局变量。代码中仍然需要正确使用 per-CPU 访问接口，以保证在 SMP 和 UP 配置下都能编译通过。

值得注意的是，percpu_counter 在 UP 下退化更为彻底——整个结构体缩减为只有 s64 count 字段（无 spinlock、无 per-CPU counters），所有操作退化为简单的 local_irq_save 保护下的标量加减法：

9.1 Per-CPU 计数器（percpu_counter）

Section titled “9.1 Per-CPU 计数器（percpu_counter）”

percpu_counter 是 per-CPU 机制的经典应用，提供了一种近似计数器机制：

工作原理：

  • 每次 percpu_counter_add() 时，将增量加到当前 CPU 的 *counters 上（无锁、快速）
  • 当某个 CPU 的计数器超过 batch 阈值时，将其值累加到主 count 并清零
  • batch 参数控制近似程度：值越大，写操作越快（刷新频率低），但 percpu_counter_read() 读取的误差越大
  • percpu_counter_read() 只读 count，结果可能是近似值
  • percpu_counter_sum() 遍历所有 CPU 的计数器并求和，结果精确但较慢

注意这里使用 local_irq_save/restore（而非 preempt_disable/enable），因为该函数需要在中断上下文中也能安全使用。刷入主计数器时使用 __this_cpu_sub(*fbc->counters, count - amount) 而非简单的清零——因为 count（即 旧本地累积值 + amount）已整体刷入 fbc->count，本地计数器只应减去已刷入的旧累积值（count - amount），保留本次调用的 amount 增量在本地继续累积。

percpu_counter_add_local() 使用 PERCPU_COUNTER_LOCAL_BATCH（即 INT_MAX）作为 batch，这意味着本地计数器几乎永远不会溢出到主计数器，写操作极其高效，但读取时必须使用 percpu_counter_sum() 而非 percpu_counter_read()。

9.2 Per-CPU 引用计数（percpu_ref）

Section titled “9.2 Per-CPU 引用计数（percpu_ref）”

percpu_ref 实现了一个可以在per-CPU 模式和原子模式之间切换的引用计数：

工作原理：

  1. Per-CPU 模式：percpu_ref_get/put 操作当前 CPU 的本地计数器，无锁高效
  2. Kill 操作：percpu_ref_kill() 将引用计数从 per-CPU 模式切换到原子模式
    • 汇总所有 CPU 的计数到 atomic_long_t count
    • 设置 __PERCPU_REF_ATOMIC 和 __PERCPU_REF_DEAD 标志
  3. 原子模式：percpu_ref_get/put 操作 atomic_long_t count
  4. Reinit 操作：percpu_ref_reinit() 可以将原子模式切回 per-CPU 模式

这种模式切换机制在对象生命周期管理中非常有用：正常操作时使用 per-CPU 模式获得高性能，销毁时切换到原子模式获得精确的零检测。

PERCPU_REF_ALLOW_REINIT 标志控制是否允许从原子模式切回 per-CPU 模式。kill 操作设置 __PERCPU_REF_DEAD 标志后，普通的 switch_to_percpu 将被拒绝，只有 reinit 或 resurrect 才能恢复。两者的区别在于：resurrect 仅清除 __PERCPU_REF_DEAD 标志，不重置计数值也不切换模式；reinit 则完整重置引用计数并切回 per-CPU 模式。

RCU 保护的必要性：percpu_ref_get/put 中的 rcu_read_lock() 并非用于保护数据，而是保证模式切换的安全性。percpu_ref_kill() 在汇总所有 CPU 计数并设置 __PERCPU_REF_ATOMIC 标志后，需要通过 synchronize_rcu() 等待所有在 RCU 读侧临界区中的 __ref_is_percpu() 检查完成。这确保了：一旦 kill 返回，后续所有 get/put 都会走原子路径，不会有任何调用者在 kill 完成后仍误入 per-CPU 路径。

Section titled “9.3 Per-CPU 读写信号量（percpu_rw_semaphore）”

percpu_rw_semaphore 利用 per-CPU 计数器实现了一种读操作极其高效的读写信号量：

读路径（极快）：

might_sleep() 标记此函数为可睡眠函数——如果在原子上下文（如持有自旋锁时）调用会触发警告。rwsem_acquire_read() 是 lockdep 的读锁获取注解，用于死锁和锁序检测。

没有写者时，读操作仅执行 this_cpu_inc()，无需任何原子操作或锁。这里的关键是 rcu_sync 机制：它维护一个”是否有人在慢速路径”的状态。当没有任何写者时，rcu_sync_is_idle() 返回 true，读者可以直接走快速路径，无需检查任何全局状态。只有当写者调用 percpu_down_write() 时，rcu_sync 才通过 rcu_sync_enter() 进入非空闲状态，后续读者才会进入慢速路径。写者释放锁后，rcu_sync_exit() 经过至少一个 RCU-sched 宽限期才恢复空闲状态，确保读者快速路径重新可用时能看见写者临界区的所有修改。

写路径：

释放时（percpu_up_write）依次执行：atomic_set_release(&sem->block, 0) 释放写锁 → __wake_up() 唤醒等待者 → rcu_sync_exit() 经过至少一个 RCU-sched 宽限期后恢复读者快速路径。

写路径的详细流程：

  1. rcu_sync_enter()：通知 rcu_sync 状态机进入非空闲状态，后续读者的 rcu_sync_is_idle() 检查将返回 false，被迫走慢速路径
  2. __percpu_down_write_trylock()：通过 atomic_xchg(&sem->block, 1) 原子地尝试获取写锁。如果已有写者（block 非零），则进入 percpu_rwsem_wait() 等待
  3. readers_active_check()：在 rcuwait_wait_event 中反复检查 per_cpu_sum(*sem->read_count) 是否为零——即所有快速路径读者是否已退出。由于 block 已设置，新读者不会进入快速路径，因此该计数只会递减
  4. 释放时：rcu_sync_exit() 不是立即恢复快速路径，而是经过至少一个 RCU-sched 宽限期后才恢复，保证新读者能看到写者临界区内的所有修改

11.3 Per-CPU 变量与中断安全

Section titled “11.3 Per-CPU 变量与中断安全”

三种操作级别的中断安全性差异在 6.2 节已详细阐述，此处强调 x86 架构下的一个关键细节：

在 x86 上，this_cpu_add 使用 add %gs:offset 单条指令完成。虽然普通 add 指令（无 lock 前缀）不保证多核原子性，但相对于本 CPU 的中断是安全的——因为中断只能发生在指令边界，一条 add 指令不会被中断打断为读和写两步。通用架构的 this_cpu_add 则需要 raw_local_irq_save/restore 来禁用本地中断以达到同样的效果。

11.4 内存回收与碎片整理

Section titled “11.4 内存回收与碎片整理”

Per-CPU 分配器有自动的平衡机制（pcpu_balance_workfn）：

  • 维持 PCPU_EMPTY_POP_PAGES_LOW（2）到 PCPU_EMPTY_POP_PAGES_HIGH（4）个空页面供原子分配使用
  • 当存在多个完全空闲的 chunk 时，回收多余的 chunk
  • 当 chunk 中有大量空闲页面时，depulate（释放物理页）以节省内存

12. 总结

Section titled “12. 总结”

Linux 内核的 per-CPU 机制是一个精心设计的子系统，从编译时的段布局到运行时的动态分配，从通用的偏移量数组到 x86 的段寄存器优化，每一层都有明确的设计考量和性能优化。

核心要点回顾：

  1. 设计哲学：以空间换时间，每个 CPU 持有数据副本，消除锁竞争
  2. 两类变量：静态（编译时定义，.data..percpu 段布局管理）和动态（运行时分配，chunk/unit/group 管理）
  3. 地址转换：通用实现用 __per_cpu_offset[] 偏移量数组，x86 用 GS/FS 段寄存器 + this_cpu_off 实现单指令寻址
  4. 三级操作：raw_cpu_*（无保护，调用者自负安全）、__this_cpu_*（preempt 调试检查）、this_cpu_*（中断安全，通用实现通过关中断或架构通过原子指令保证）
  5. 并发控制：动态分配器采用 spinlock + mutex 两级锁设计，将高频元数据操作与低频结构变更解耦
  6. 架构优化：x86 利用段前缀寻址、inc/dec 指令优化、cmpxchg 无锁 CAS、Named Address Space 等手段层层递进
  7. 初始化：启动时通过 embed（连续内存，TLB 友好）或 page（NUMA 友好）策略构建 first chunk，CPU hotplug 时复用预分配区域
  8. 高级应用：percpu_counter（batch 阈值控制的近似计数器）、percpu_ref（per-CPU/原子模式切换的引用计数）、percpu_rw_semaphore（rcu_sync 驱动的读写信号量，读者快速路径仅一条 per-CPU 加法）
  9. UP 退化：在单处理器配置下，整个 per-CPU 机制退化为普通全局变量访问，零额外开销

选择正确的操作级别（raw_cpu_* vs this_cpu_* vs get_cpu_var）是内核开发中的常见决策——遵循最小充分保护原则，既保证正确性又避免不必要的开销。Per-CPU 机制是 Linux 内核实现高可扩展性的基石之一，理解其原理对内核开发至关重要。
