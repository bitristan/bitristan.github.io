ARM64 Linux 启动原理（三）：head.S 汇编入口

基于 Linux v6.10 源码，arch/arm64/kernel/head.S

1. 概述

Section titled “1. 概述”

head.S 是 ARM64 内核的第一个执行点。从 primary_entry 开始，CPU 在最小环境（MMU 和 D-cache 可能开启也可能关闭，取决于固件）下完成以下工作：

  1. 检测 MMU 状态、修复字节序
  2. 保存 bootloader 传递的参数
  3. 构建初始身份映射页表
  4. 确定并设置正确的异常级别
  5. 配置 CPU 寄存器（MAIR、TCR）
  6. 开启 MMU
  7. 映射内核到虚拟地址
  8. 跳转到 start_kernel()

整个过程跨越了从”无虚拟地址”到”完整内核虚拟地址空间”的关键转变。

2. 寄存器约定

Section titled “2. 寄存器约定”

启动过程中使用 callee-saved 寄存器保存关键状态，因为此时没有栈可用：

寄存器生命周期用途
x19primary_entry() → start_kernel()MMU 是否开启（0=关闭，非0=开启）
x20init_kernel_el() → __primary_switched()CPU 启动模式（BOOT_CPU_MODE_EL1 或 _EL2，含 E2H 标志）
x21primary_entry() → start_kernel()FDT 物理地址

3. 镜像头部

Section titled “3. 镜像头部”

这个头部同时满足 Linux bootloader 和 UEFI 固件的要求。Direct Boot 通过 b primary_entry 进入，EFI Boot 通过 PE 头的 AddressOfEntryPoint 进入。

4. primary_entry：主入口

Section titled “4. primary_entry：主入口”

以下逐函数分析。

5. record_mmu_state：检测 MMU 状态

Section titled “5. record_mmu_state：检测 MMU 状态”

逻辑分析：

  1. 读取当前 EL 的 SCTLR 寄存器
  2. 检查字节序（EE 位）是否与编译目标一致。如果不一致，翻转 EE 位并关闭 MMU
  3. 检查 D-cache（C 位）：如果 D-cache 关闭，说明是 Direct Boot 路径，清除 x19
  4. 最终 x19 = 0 表示 MMU 关闭，x19 != 0 表示 MMU 开启

为什么需要检查 D-cache：如果 D-cache 关闭但 M 位为 1，说明固件配置异常，按 MMU 关闭处理更安全。

6. preserve_boot_args：保存启动参数

Section titled “6. preserve_boot_args：保存启动参数”

关键点：

  • x21 = x0：FDT 地址被保存到 x21，在整个启动过程中一直保留
  • boot_args[0] = x21 (FDT)，boot_args[1-3] = x1-x3（x0 在取 boot_args 地址时已被覆盖）
  • MMU 关闭时需要手动失效缓存行，确保写入对后续代码可见
  • MMU 开启时记录 mmu_enabled_at_boot，后续 setup_arch() 会检查

7. __pi_create_init_idmap：创建初始身份映射

Section titled “7. __pi_create_init_idmap：创建初始身份映射”

在 primary_entry 中设置好早期栈后，调用位置无关函数 __pi_create_init_idmap 创建初始身份映射页表：

身份映射（Identity Map）的含义：虚拟地址 == 物理地址。这是开启 MMU 的前提——CPU 在开启 MMU 的瞬间，PC 寄存器中的物理地址必须能被正确翻译到同一条指令。

映射内容（map_range.c:99-102）：

  • _stext 到 __initdata_begin：映射为 PAGE_KERNEL_ROX（只读可执行），覆盖 .text、.rodata、.rodata.text（含 .idmap.text）、页表目录、init 代码段
  • __initdata_begin 到 _end：映射为 PAGE_KERNEL（读写），覆盖 init 数据段、.data、.bss、init 页表

缓存维护（primary_entry 第 99-116 行）：

  • MMU 关闭时：失效页表区域的缓存行（防止投机加载的脏数据）
  • MMU 开启时（EFI 路径）：清理身份映射代码到 PoC（确保 MMU 关闭后仍能执行）

8. init_kernel_el：异常级别切换

Section titled “8. init_kernel_el：异常级别切换”

EL1 路径

Section titled “EL1 路径”

已经在 EL1，只需：

  1. 设置 SCTLR_EL1 为 INIT_SCTLR_EL1_MMU_OFF（禁用 MMU/I-cache/D-cache，使能对齐检查 LSMAOE/nTLSMD、异常入口同步 EIS、线程上下文同步 TSCXT 等）
  2. 设置 SPSR_EL1 和 ELR_EL1 为返回地址
  3. ERET 切换到 SP_EL1/PSTATE_EL1 上下文（EL1 路径下实际效果是重置 PSTATE 并返回调用者）

EL2 路径

Section titled “EL2 路径”

从 EL2 降到 EL1，需要：

  1. 关闭 EL2 MMU
  2. 配置 HCR_EL2（nVHE 模式标志）
  3. 检查 VHE-only CPU（ID_AA64MMFR4_EL1.E2H0 < 0），设置 HCR_E2H
  4. init_el2_state：初始化定时器、调试、GIC、Stage-2 等 EL2 寄存器
  5. 安装 __hyp_stub_vectors（处理 HVC 调用的 stub）
  6. 设置 SCTLR_EL1
  7. ERET 到 EL1

返回值：x0 = BOOT_CPU_MODE_EL1 或 BOOT_CPU_MODE_EL2（可能附带 BOOT_CPU_FLAG_E2H 标志）

9. __cpu_setup：CPU 寄存器配置

Section titled “9. __cpu_setup：CPU 寄存器配置”

MAIR_EL1：内存属性

Section titled “MAIR_EL1：内存属性”

MAIR_EL1_SET 定义了 8 种内存类型编码：

索引属性用途
0DEVICE_nGnRnE (0x00)设备内存，最严格
1DEVICE_nGnRE (0x04)设备内存，允许早期写确认
2NORMAL_NC (0x44)普通内存，非缓存
3NORMAL_TAGGED (0xf0)普通内存，MTE 标记
4NORMAL (0xff)普通内存，可缓存

页表项中的 AttrIndx 字段引用这些索引。

TCR_EL1：翻译控制

Section titled “TCR_EL1：翻译控制”

关键字段：

  • T0SZ：TTBR0 地址空间大小（身份映射用）
  • T1SZ：TTBR1 地址空间大小（内核用，64 - VA_BITS）
  • TG0/TG1：页粒度（4K/16K/64K）
  • IPS：物理地址大小
  • SH0/SH1：共享属性
  • IRGN0/IRGN1、ORGN0/ORGN1：缓存属性
  • HA：硬件访问标志更新

INIT_SCTLR_EL1_MMU_ON

Section titled “INIT_SCTLR_EL1_MMU_ON”

__cpu_setup 返回的 SCTLR 值，包含：

  • M：MMU 使能
  • C：D-cache 使能
  • I：I-cache 使能
  • SA/SA0：栈对齐检查
  • IESB：隐式异常同步
  • SPAN：特权访问永远不会

10. __primary_switch：开启 MMU

Section titled “10. __primary_switch：开启 MMU”

__enable_mmu：MMU 开启的精确时序

Section titled “__enable_mmu：MMU 开启的精确时序”

开启 MMU 的瞬间：

  • TTBR0_EL1 → init_idmap_pg_dir：身份映射，VA == PA
  • TTBR1_EL1 → reserved_pg_dir：空页表
  • SCTLR_EL1.M = 1：MMU 开启

此时 CPU 正在执行 __primary_switch 中的代码，PC 中是物理地址。由于身份映射的存在，这个物理地址被翻译为相同的虚拟地址，CPU 继续执行同一条指令流。

__pi_early_map_kernel：映射内核

Section titled “__pi_early_map_kernel：映射内核”

MMU 开启后，调用位置无关函数完成内核映射：

完成后的状态：

  • TTBR0_EL1 → init_idmap_pg_dir（身份映射仍然存在）
  • TTBR1_EL1 → swapper_pg_dir（内核虚拟地址映射）
  • BSS 已清零
  • KASLR 偏移已计算

跳转到 __primary_switched

Section titled “跳转到 __primary_switched”

br x8 使用绝对地址跳转，此时 x8 中是 __primary_switched 的虚拟地址。由于 TTBR1_EL1 已指向 swapper_pg_dir，这次跳转完成了从物理地址到虚拟地址的过渡。

11. __primary_switched：进入 C 世界

Section titled “11. __primary_switched：进入 C 世界”

init_cpu_task 宏

Section titled “init_cpu_task 宏”

finalise_el2：VHE 使能

Section titled “finalise_el2：VHE 使能”

如果 CPU 支持 VHE（Virtualization Host Extensions），finalise_el2 会将内核上下文从 nVHE 升级到 VHE，使内核可以在 EL2 运行。

12. 从核启动路径（简述）

Section titled “12. 从核启动路径（简述）”

从核（Secondary CPU）有两条入口：

spin-table 方式

Section titled “spin-table 方式”

从核在 holding pen 中自旋，等待主核写入其 MPIDR 到 secondary_holding_pen_release。

PSCI 方式

Section titled “PSCI 方式”

从核由 PSCI 固件唤醒，直接进入 secondary_startup。

通用路径

Section titled “通用路径”

从核使用 swapper_pg_dir（主核已创建好）和 idmap_pg_dir（运行时身份映射），无需重新创建页表。

13. 代码段属性

Section titled “13. 代码段属性”

head.S 中的代码分布在多个特殊段中：

段属性用途
.head.textHEAD_TEXT镜像头部，不缓存
.idmap.text身份映射可执行MMU 开启前后都能执行的代码
__INIT初始化代码仅启动时使用
.text普通代码正常内核代码

.idmap.text 段的代码被同时映射到身份映射和内核虚拟地址空间，确保 MMU 开启/关闭切换时不会丢失执行流。
