Linux 设备驱动模型：总线、设备、驱动的关系

一、核心思想：一个红娘模型

Section titled “一、核心思想：一个红娘模型”

Linux 设备驱动模型的核心可以用一句话概括：总线是红娘，负责把设备和驱动撮合到一起。

想象一个场景：

  • 设备说：“我是谁谁谁，谁能来管我？”
  • 驱动说：“我能管什么样的设备，谁需要我？”
  • 总线说：“别急，我来帮你们配对。”

三者中，总线是核心枢纽。设备和驱动都必须挂载在某条总线上，由总线负责匹配（match）。匹配成功后，驱动的 probe 函数被调用，驱动开始真正操控设备。

二、三个核心结构体

Section titled “二、三个核心结构体”

2.1 struct bus_type —— 总线

Section titled “2.1 struct bus_type —— 总线”

最关键的字段是 match。每条总线必须实现 match 函数，它决定了”什么样的设备和什么样的驱动能配对”。

2.2 struct device —— 设备

Section titled “2.2 struct device —— 设备”

关键字段：

  • bus：指向设备所属的总线，匹配时要用到
  • driver：匹配成功后，指向绑定的驱动
  • of_node：设备树节点，用于设备树匹配

2.3 struct device_driver —— 驱动

Section titled “2.3 struct device_driver —— 驱动”

关键字段：

  • bus：驱动注册到哪条总线
  • of_match_table：设备树兼容性匹配表，这是最常用的匹配方式
  • probe：匹配成功后被调用，初始化设备

三、三者的关系图

Section titled “三、三者的关系图”
  • 每个 device 通过 bus 字段指向所属总线
  • 每个 device_driver 通过 bus 字段指向所属总线
  • 匹配成功后，device->driver 指向绑定的驱动
  • 总线内部维护两个链表：已注册的设备链表和已注册的驱动链表

5.2 platform_match —— 四种匹配策略

Section titled “5.2 platform_match —— 四种匹配策略”

匹配优先级从高到低：

  1. driver_override：设备强制指定驱动名
  2. 设备树匹配：比较 compatible 属性和驱动的 of_match_table
  3. ACPI 匹配：x86 平台使用
  4. ID 表匹配：驱动提供 id_table，逐项比较设备名
  5. 名称匹配：设备名 == 驱动名

八、总结

Section titled “八、总结”
概念角色关键结构关键操作
总线红娘/管理者bus_typematch() 配对，probe() 探测
设备被管理者device注册时触发匹配
驱动管理者device_driver注册时触发匹配

核心流程一句话：设备和驱动分别注册到总线上，总线用 match 函数判断能否配对，配对成功则调用 probe 完成绑定。

谁后注册，谁主动去找对方——设备后注册就遍历驱动链表，驱动后注册就遍历设备链表。这种双向查找机制确保了无论谁先到，都能找到对方。
