ARM64 Linux 启动原理（二）：EFI Stub

ARM64 Linux 启动原理（二）：EFI Stub — UEFI 到内核的桥梁

Section titled “ARM64 Linux 启动原理（二）：EFI Stub — UEFI 到内核的桥梁”

基于 Linux v6.10 源码，arch/arm64/ 架构

1. 概述

Section titled “1. 概述”

EFI Stub 是嵌入在内核镜像中的代码，使 Linux 能够直接作为 UEFI 应用程序启动，无需额外的 bootloader。内核镜像同时是一个合法的 PE/COFF 可执行文件，UEFI 固件可以直接加载并执行。

与 Direct Boot 相比，EFI 路径的最大差异是：进入 primary_entry 时 MMU 和 D-cache 可能处于开启状态（取决于 UEFI 固件），内核需要通过 record_mmu_state 检测并处理。

这条 ccmp 指令在执行时无害，但其操作码的前两个字节恰好是 ASCII “M”（0x4D）和 “Z”（0x5A），满足 PE/COFF 的签名要求。

3.2 PE 头部关键字段

Section titled “3.2 PE 头部关键字段”

efi-header.S 中的 __EFI_PE_HEADER 宏定义了完整的 PE/COFF 头部。关键字段：

字段值含义
MachineIMAGE_FILE_MACHINE_ARM64AArch64 架构

SizeOfImage_end - _text整个镜像大小

PE 头部定义了两个段：.text（_text 到 __initdata_begin，可执行）和 .data（__initdata_begin 到 _edata，可读写）。

5. FDT 处理与 ExitBootServices

Section titled “5. FDT 处理与 ExitBootServices”

5.1 FDT 获取与修改

Section titled “5.1 FDT 获取与修改”

efi_boot_kernel() 调用 allocate_new_fdt_and_exit_boot() 完成 FDT 处理：

  1. 从 UEFI 配置表获取固件提供的 FDT（DEVICE_TREE_GUID）；如果固件未提供，则从命令行 dtb= 参数指定的文件加载
  2. 分配新 FDT 内存，复制原始 FDT
  3. 添加/修改属性：
    • /chosen/bootargs：内核命令行
    • /chosen/linux,initrd-start、linux,initrd-end：initrd 位置
    • /chosen/linux,uefi-mmap-start 等：UEFI 内存映射
    • /chosen/linux,uefi-system-table：UEFI 系统表地址
    • /chosen/linux,uefi-secure-boot：安全启动状态
  4. 安装 FDT 到 UEFI 配置表

FDT 准备完毕后，调用 ExitBootServices()：

这是 EFI Stub 最关键的一步：此后 Boot Services 不再可用，内核完全接管硬件。只剩 Runtime Services（GetTime、SetVirtualAddressMap 等）可供后续使用。

5.3 最终跳转

Section titled “5.3 最终跳转”

primary_entry_offset() 的必要性：UEFI 固件加载镜像时会跳过文件头，直接加载 .text 段。因此不能通过镜像头的分支指令跳转，而需要计算 primary_entry 相对于 _text 的偏移量。

6. EFI 进入 primary_entry 后的状态

Section titled “6. EFI 进入 primary_entry 后的状态”

record_mmu_state（head.S:133）检测 SCTLR_ELx 的 M 位和 C 位，将结果存入 x19。后续代码根据 x19 决定是否需要缓存维护操作。

7. EFI Runtime Services

Section titled “7. EFI Runtime Services”

内核启动后，通过 efi_init()（在 setup_arch() 中调用）初始化 EFI 子系统：

  • 解析 FDT 中的 UEFI 系统表地址
  • 获取 UEFI 内存映射
  • 根据 efi_novamap 决定是否调用 SetVirtualAddressMap()
    • efi_novamap = true（VA_BITS >= 48 且无 workaround）：直接使用 1:1 映射
    • efi_novamap = false：调用 SetVirtualAddressMap() 将运行时服务映射到虚拟地址

内核运行时可通过 /sys/firmware/efi/ 查看 EFI 信息。

8. 设计原理

Section titled “8. 设计原理”

为什么 EFI Stub 嵌入内核？

Section titled “为什么 EFI Stub 嵌入内核？”
  • 简化部署：直接将内核文件复制到 EFI 系统分区即可启动
  • 减少依赖：不依赖特定 bootloader 的实现
  • 统一入口：内核和 EFI Stub 共享同一镜像

为什么 KASLR 在 EFI Stub 中处理？

Section titled “为什么 KASLR 在 EFI Stub 中处理？”
  • EFI Stub 在 ExitBootServices() 之前运行，可以使用 UEFI 内存分配器
  • 内核运行时的内存分配器尚未初始化
  • EFI Stub 可以获取 EFI_RNG_PROTOCOL 的随机数

为什么需要 primary_entry_offset()？

Section titled “为什么需要 primary_entry_offset()？”

PE/COFF 头部是文件格式的一部分，不是内存镜像的一部分。UEFI 固件加载时会跳过文件头，因此需要通过偏移量计算实际入口。
