基于 Linux v6.10 源码，arch/arm64/ 架构

1. 概述

Section titled “1. 概述”

本文从硬件到软件、从启动到运行，完整讲解 ARM64 页表的方方面面：

  • 硬件基础：PTE 格式、TLB 组织、页表遍历器、TTBR/ASID/CNP

2.2 不同级别的描述符类型

Section titled “2.2 不同级别的描述符类型”
级别表描述符块描述符映射大小（4K 页）
PGD✅ (type=11)❌—
P4D✅ (type=11)✅ (type=01)512 GB（LPA2）
PUD✅ (type=11)✅ (type=01)1 GB

块描述符允许在 PUD 或 PMD 级别直接映射大块物理内存，跳过下级页表，减少 TLB 压力。

2.3 连续映射（Contiguous Mapping）

Section titled “2.3 连续映射（Contiguous Mapping）”

ARM64 支持将多个连续的 PTE 或 PMD 条目标记为 PTE_CONT，TLB 可以将它们合并为一个条目：

标记连续条目数合并大小（4K 页）
PTE_CONT1664 KB
PMD_CONT1632 MB

要求：地址必须对齐到合并后的大小边界。连续映射是提示性的（hint），TLB 可以忽略。

2.4 访问权限（AP 位）

Section titled “2.4 访问权限（AP 位）”
AP[2:1]EL1EL0含义
00RW无访问内核读写
01RWRW用户读写
10RO无访问内核只读
11RORO用户只读

AP[1] 也称为 P bit（User accessible bit）。当 P=0 时，EL0 访问该页会触发 Permission fault。这是内核与用户空间隔离的硬件基础。

3. 特权级别与页表访问控制

Section titled “3. 特权级别与页表访问控制”

3.1 异常级别（Exception Level）

Section titled “3.1 异常级别（Exception Level）”

ARM64 用 EL0~EL3 四个特权级别区分代码权限：

CPU 通过 PSTATE 中的 CurrentEL 字段知道当前处于哪个级别。EL0 不能执行特权指令（如 MSR/MRS 访问系统寄存器），也不能直接修改 MMU 配置。

3.2 EL0 如何进入 EL1

Section titled “3.2 EL0 如何进入 EL1”

EL0 不能自行跳转到 EL1。唯一的通道是异常：

  • SVC 指令（系统调用）
  • 缺页异常（Translation/Permission Fault）
  • 中断（IRQ/FIQ）
  • 非法指令（Undefined Instruction）

异常发生时，CPU 硬件自动切换到 EL1，跳转到 VBAR_EL1 指向的异常向量表。

3.3 TTBR0/TTBR1 与 EL 的交互

Section titled “3.3 TTBR0/TTBR1 与 EL 的交互”

ARM64 有两个页表基址寄存器，将虚拟地址空间分为两半：

硬件根据 VA 高位选择 TTBR，不看当前 EL 级别。但软件约定是：

  • TTBR0 映射用户空间（PTE 的 AP[1]=1，EL0 可访问）
  • TTBR1 映射内核空间（PTE 的 AP[1]=0，仅 EL1 可访问）

EL0 即使知道内核地址，MMU 也会因 P bit 拒绝访问 → Permission fault → CPU 自动进入 EL1。

4.3 ASID（Address Space Identifier）

Section titled “4.3 ASID（Address Space Identifier）”

ASID 是 8-bit 或 16-bit 的标识符（由 TCR_EL1.AS 控制），用于区分不同进程的 TLB 条目：

  • 不同 ASID 的 TLB 条目可以共存于 TLB 中
  • 进程切换时只需切换 ASID，无需刷新整个 TLB
  • ASID 0 是保留值（内核使用）

4.4 CNP（Common Not Private）

Section titled “4.4 CNP（Common Not Private）”

当 TCR_EL1.CNP = 1 时，同一物理页表在所有核上共享同一个 TLB 条目，减少 TLB 压力。

5.6 页表页与 struct page

Section titled “5.6 页表页与 struct page”

页表页是普通的物理页，有对应的 struct page。它们被标记为 __GFP_MOVABLE 排除（不可迁移），因为页表页的物理地址被上级页表项直接引用，迁移会导致引用失效。

6. 启动阶段：页表从零到稳定态

Section titled “6. 启动阶段：页表从零到稳定态”

这是 ARM64 页表系统最精妙的部分。内核需要在 MMU 关闭的状态下建立第一张页表，打开 MMU，然后逐步切换到最终的稳定态页表。

6.1 核心问题

Section titled “6.1 核心问题”

bootloader 跳转到内核时，MMU 处于关闭状态。内核代码在物理地址上运行。要打开 MMU，必须先有页表；但页表本身也需要 MMU 才能方便地访问。这是一个”鸡生蛋”问题。

解决方案分三步：

  1. MMU 关闭：在物理地址上建一张恒等映射页表（VA == PA）
  2. 打开 MMU：CPU 的 PC 仍然是物理地址，经过恒等映射翻译后指向自身
  3. MMU 已开：在此基础上建立真正的高半区虚拟映射，切换过去

6.2 关键页表一览

Section titled “6.2 关键页表一览”

在深入代码前，先认识启动过程中涉及的所有页表：

页表位置生命周期用途
init_idmap_pg_dir.init.data 段free_initmem() 释放初始恒等映射（TTBR0）
init_pg_dirBSS 之后不释放（死副本+活引用）建表工作区（临时 PGD）

关键布局约束：

6.3 __pi_ 前缀的含义

Section titled “6.3 __pi_ 前缀的含义”

启动阶段的部分代码使用 __pi_ 前缀（如 __pi_create_init_idmap）。这来自 arch/arm64/kernel/pi/Makefile：

pi/ 目录下的代码以 -fpie（Position Independent）编译，通过 objcopy 给所有符号加 __pi_ 前缀，使其可在 MMU 关闭时以任意物理地址运行。

关键设计点：

  1. 恒等映射：start == pa，虚拟地址等于物理地址。MMU 打开后 CPU 的 PC 仍然是物理地址，经过 MMU 翻译后仍指向自身。
  2. 为什么放在 TTBR0：内核镜像被 bootloader 加载在低物理地址，VA 高位为 0，属于低半区，硬件规定低半区走 TTBR0。
  3. IDMAP_ROOT_LEVEL：4 - IDMAP_LEVELS。4K 页 + 48 位 VA = 4 级，root level = 0 (PGD)。
  4. 只映射内核镜像：不映射 FDT 和其他 DRAM。FDT 稍后由 map_fdt 单独追加。

init_pg_dir 的内部布局（bump allocator 的效果）：

为什么不只用 swapper_pg_dir？

  1. swapper_pg_dir 只有 1 页（4KB），而构建完整页表需要 PGD + PUD + PMD + PTE 多页。需要一个足够大的工作区。
  2. swapper_pg_dir 必须与 reserved_pg_dir 紧邻（固定偏移 PAGE_SIZE），以满足 KPTI/SW PAN 的快速切换要求。这个位置约束无法容纳多页。
  3. init_pg_dir 在 BSS 之后，大小由 INIT_DIR_SIZE 宏计算，专作临时工作区。

memcpy 中的 va_offset：swapper_pg_dir 的符号值是链接器指定的高半区虚拟地址，但此时代码运行在恒等映射（TTBR0 idmap）的”庇护”下，PC 是物理地址。va_offset 将虚拟地址转换为恒等映射能访问的地址，确保 memcpy 写入正确位置。

为什么只拷贝 PGD 页？ 拷贝后，swapper_pg_dir 的 PGD entry 仍然指向 init_pg_dir 中的下级页表页（PUD/PMD/PTE）。init_pg_dir 不在 __init_begin..__init_end 范围内，不会被 free_initmem() 释放，所以下级页表页一直有效。第一页（原始 PGD）则成为死副本，浪费 4KB。

7. KPTI 与 SW PAN：运行时安全机制

Section titled “7. KPTI 与 SW PAN：运行时安全机制”

7.1 KPTI 的动机：Meltdown 攻击

Section titled “7.1 KPTI 的动机：Meltdown 攻击”

CPU 的投机执行会在权限检查完成前执行指令。攻击者可以：

  1. 用户态程序投机执行一条”读内核地址”的指令
  2. CPU 在权限检查前开始翻译，缓存中留下痕迹
  3. 攻击者通过缓存侧信道恢复出内核数据

KPTI 的防御思路：在用户态运行时，让 TTBR1 不包含内核的完整映射，投机执行就无法翻译出内核地址。

tramp_pg_dir 只映射一小段 trampoline 代码。用户态投机执行时，TTBR1 能翻译到的地址只有这段代码，内核数据完全不可见。

7.4 reserved_pg_dir 的用途

Section titled “7.4 reserved_pg_dir 的用途”

reserved_pg_dir 是一张全零的 PGD 页。全零意味着每个 entry 的 Valid 位为 0，硬件会将其视为”无映射”（Translation fault）。这是合法状态，不是错误。

它有三个用途：

  1. 启动时占位：__enable_mmu 打开 MMU 时，TTBR1 必须有合法值但不能有映射（避免高半区地址冲突）
  2. SW PAN 保护：内核不需要访问用户内存时，将 TTBR0 设为空表
  3. 内核线程：内核线程没有用户地址空间，TTBR0 指向 reserved_pg_dir
