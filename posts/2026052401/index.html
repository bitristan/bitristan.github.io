    三、稳定性推断系统（Stability Inference）

    3.1 为什么需要稳定性？

    Compose 的跳过（Skip）机制依赖于判断参数是否变化。如果参数类型是"稳定的"（相等性含义明确且不可变），编译器就可以在参数未变时跳过重组；如果类型"不稳定"，每次都必须执行。

    3.2 Stability 类型体系

    sealed class Stability {
    // 编译期可确定：稳定（true）或不稳定（false）
    class Certain(val stable: Boolean) : Stability()

    // 运行时才能确定：需要读取 $stable 位掩码
    class Runtime(val declaration: IrClass) : Stability()

    // 无法确定：如接口类型，其实现可能不稳定
    class Unknown(val declaration: IrClass) : Stability()

    // 泛型参数：取决于泛型实参的稳定性
    class Parameter(val parameter: IrTypeParameter) : Stability()

    // 组合：多个稳定性条件的组合
    class Combined(val elements: List<Stability>) : Stability()
}

    推断规则（StabilityInferencer.stabilityOf()）：

    1. 直接标记：类标注了 @Stable / @Immutable（或其父类有 @StableMarker）→ Stable
    2. 已知稳定类型：Pair、Triple、Result、枚举、object、基本类型、BigInteger 等 → Stable
    3. 外部稳定类型：通过 stabilityConfigurationPath 配置文件指定 → Stable
    4. 接口：→ Unknown（无法确定实现是否稳定）
    5. 含 var 属性的类：→ Unstable
    6. 含不稳定属性的类：递归检查所有属性 → Combined
    7. 泛型类：→ Parameter + 具体类型参数的稳定性
    8. 跨文件公开类（JVM）：→ Runtime（运行时通过 $stable 字段判断）
    9. Java 类：→ Unstable（保守策略）

    四、Durable Key 生成

    4.1 为什么需要 Key？

    Compose 运行时使用 SlotTable 存储组合状态，每个 Group 在 SlotTable 中通过 Key 定位。Key 必须是"持久的"（Durable）——即同一调用点在不同次执行中必须生成相同的 Key，否则运行时无法正确匹配状态。

    4.2 DurableKeyTransformer

    源码：lower/DurableKeyTransformer.kt

    此 Transform 基于 AST 结构为每个调用点构建路径式 Key：

    1
2
3
4
5

    pkg-com.example.app
  └─ file-MainActivity.kt
       └─ class-MainActivity
            └─ call-Greeting
                 └─ arg-0

    关键机制：

    • enter(key) — 进入一个作用域，Key 路径追加该段
    • siblings(key) — 同级节点用 : 分隔，避免同名冲突
    • root(keys) — 重置 Key 路径
    • buildKey() — 构建最终的 Key 字符串

    对于 when 表达式中的同级分支，使用 siblings 确保每个分支有唯一的 Key。

    4.3 DurableFunctionKeyTransformer

    源码：lower/DurableFunctionKeyTransformer.kt

    在 DurableKeyTransformer 基础上，为每个 Composable 函数调用生成语义化的 Key（包含函数名和参数信息），并可选地生成 @FunctionKeyMeta 注解供工具使用。

    5.2 $changed 位掩码详解

    源码：ComposableFunctionBodyTransformer.kt 中的 ParamState 枚举

    每个参数在 $changed 中占用 3 个 bit（最低位是特殊强制执行位）：

    1
2
3
4
5
6
7
8

    $changed 布局（每个 Int 存 10 个参数，超出则追加 $changed1, $changed2...）

  bit:  [31] ... [4] [3] [2] [1] [0]
                                ↑ 强制执行位
                          ↑↑↑ 第 0 个参数的 3-bit 状态
                    ↑↑↑ 第 1 个参数的 3-bit 状态
              ↑↑↑ 第 2 个参数的 3-bit 状态
        ...

    每个参数的 3-bit 状态值（ParamState）：

    值名称含义
    0b000Uncertain不确定，需要调用 equals() 比较
    0b001Same已知与上次相同
    0b010Different已知与上次不同
    0b011Static永远不会变化（如字面量）

fun changedParamCount(realValueParams: Int, thisParams: Int): Int {

    六、ComposableFunctionBodyTransformer — 函数体核心变换

    源码：lower/ComposableFunctionBodyTransformer.kt （最复杂、最重要的文件，约 4600 行）

    这是整个编译器插件最核心的变换，负责 6 大任务：

    1. 控制流 Group 生成
    2. 默认参数处理
    3. Composable 函数跳过（Skipping）
    4. 比较传播（Comparison Propagation）
    5. 可重组性（Recomposability）
    6. 源码位置信息

    6.1 三类 Group

    Compose 运行时中有三种 Group，对应不同的语义：

    Group 类型API用途
    Replace GroupstartReplaceableGroup / endReplaceGroup条件分支：when/if 中只有一个分支会执行
    Movable GroupstartMovableGroup / endMovableGroup可移动内容：位置可能变化但状态保留
    Restart GroupstartRestartGroup / endRestartGroup可重启范围：重组的最小单位

    6.2 函数分类与变换策略

    visitFunctionInScope() 根据函数特征分三类处理：

    1
2
3
4
5
6
7
8

    return when {
    isLambda && isTracked ->
        visitComposableLambda(...)        // Composable Lambda
    restartable && isTracked ->
        visitRestartableComposableFunction(...)  // 可重启的 Composable 函数
    else ->
        visitNonRestartableComposableFunction(...)  // 不可重启的 Composable 函数
}

    判断是否可重启（restartable）的条件：

    • 函数标注了 @Composable
    • 没有 @NonRestartableComposable 注解
    • 不是 inline lambda

    判断是否可跳过（skippable）的条件：

    • 所有参数类型都是稳定的（或启用了 Strong Skipping）
    • 函数返回 Unit
    • 没有 @NonSkippableComposable 注解

    fun Greeting(name: String, $composer: Composer<*>, $changed: Int) {
    // ① 开始 Restart Group
    $composer.startRestartGroup(<key>)

    // ② 源码位置信息（可选）
    sourceInformation("Greeting(name)")

    // ③ 创建 $dirty 变量，复制 $changed
    var $dirty = $changed

    // ④ 参数比较逻辑（只在 Uncertain 时调用 composer.changed）
    if ($dirty and 0b0110 == 0) {  // 第 0 个参数状态为 Uncertain
        $dirty = $dirty or if ($composer.changed(name)) 0b0010 else 0b0100
    }

    // ⑤ 跳过判断
    if ($dirty and 0b1011 != 0b1010 || !$composer.skipping) {
        // ⑥ 执行函数体
        Text("Hello, $name!", $composer, 0b110)  // 传播比较信息
    } else {
        // ⑦ 跳过执行
        $composer.skipToGroupEnd()
    }

    // ⑧ 结束 Restart Group 并注册重组回调
    $composer.endRestartGroup()?.updateScope { next ->
        Greeting(name, next, $changed or 0b1)
    }
}

    6.4 参数比较逻辑（Skipping Preamble）

    源码：buildPreambleStatementsAndReturnIfSkippingPossible()

    核心逻辑：对于每个状态为 Uncertain 的参数，调用 $composer.changed(param) 比较：

    1
2
3
4

    if ($dirty and 0b0110 == 0) {
    // 参数状态是 Uncertain，需要运行时比较
    $dirty = $dirty or if ($composer.changed(name)) 0b0010 else 0b0100
}
    • 0b0110 — 隔离第 0 个参数的 3-bit 状态（排除最低强制位）
    • 如果 changed() 返回 true（值变了）→ 设置为 Different(0b010)
    • 如果 changed() 返回 false（值没变）→ 设置为 Same(0b001)

    6.5 跳过判断

    1
2
3
4
5

    if ($dirty and 0b1011 != 0b1010 || !$composer.skipping) {
    // 执行函数体
} else {
    $composer.skipToGroupEnd()
}

    0b1011 vs 0b1010 的含义：

    • 0b1010 = 第 0 个参数是 Same(0b001) 且强制位为 0 → 所有参数未变
    • 0b1011 = 掩码，检查所有参数是否都是"相同"状态
    • 如果 $dirty & 0b1011 != 0b1010，说明至少有一个参数变化了 → 执行
    • 或者 $composer.skipping == false → 运行时禁用了跳过 → 执行

    七、Lambda 记忆化（Memoization）

    7.1 问题

    在 Composable 函数中传递 Lambda 时，如果每次重组都创建新实例，即使 Lambda 内容相同，引用也不同，会导致子 Composable 不必要地重组。

    7.3 捕获分析

    CaptureCollector 分析 Lambda 捕获了哪些外部变量。如果 Lambda 不捕获任何变量（即纯函数），可以全局复用同一实例；如果捕获了变量，则根据捕获值的变化决定是否需要新实例。

    八、默认参数处理

    1. 默认表达式中可能包含 Composable 调用，需要在正确的 Group 中执行
    2. 需要传播 Composer 参数

    fun ProfileCard(
    user: User,
    modifier: Modifier,
    $composer: Composer<*>,
    $changed: Int,
    $default: Int
) {
    // ── 默认参数处理 ──
    $composer.startDefaults()
    val modifier = if ($default and 0b1 != 0) {
        Modifier
    } else {
        modifier
    }
    $composer.endDefaults()

    // ── 开始 Restart Group ──
    $composer.startRestartGroup(<key>)
    sourceInformation("ProfileCard(User, Modifier)")

    // ── 参数比较 ──
    var $dirty = $changed
    if ($dirty and 0b0110 == 0) {
        $dirty = $dirty or if ($composer.changed(user)) 0b0010 else 0b0100
    }
    if ($dirty and 0b011000 == 0) {
        $dirty = $dirty or if ($composer.changed(modifier)) 0b010000 else 0b100000
    }

    // ── 跳过判断 ──
    if ($dirty and 0b10101011 != 0b10101010 || !$composer.skipping) {
        // ── 执行函数体 ──
        Column(
            modifier = modifier,
            $composer = $composer,
            $changed = 0b110 or ($dirty and 0b110000 shr 2)
        )
        // Replace Group: Column 的内容
        $composer.startReplaceableGroup(<key1>)
        Text(user.name, $composer, 0b110 or ($dirty and 0b110 shr 0))
        $composer.endReplaceGroup()

        $composer.startReplaceableGroup(<key2>)
        Text(user.email, $composer, 0b110 or ($dirty and 0b110 shr 0))
        $composer.endReplaceGroup()
    } else {
        $composer.skipToGroupEnd()
    }

    // ── 结束 Restart Group 并注册重组回调 ──
    $composer.endRestartGroup()?.updateScope { next ->
        ProfileCard(user, modifier, next, $changed or 0b1, $default)
    }
}

    十二、关键注解对编译的影响

    注解位置效果
    @Composable函数/属性触发整个编译器变换流水线
    @Stable类/函数标记为稳定类型，影响跳过判断
    @Immutable类标记为不可变类型，所有属性视为稳定
    @NonRestartableComposable函数不生成 Restart Group 和 updateScope
    @NonSkippableComposable函数不生成跳过逻辑，总是执行
    @ReadOnlyComposable函数只读 Composable，不生成 Group
    @ExplicitGroupsComposable函数显式管理 Group，编译器不自动插入
    @DisallowComposableCallsLambda禁止在 Lambda 体中调用 Composable
    @NoLiveLiterals文件/函数禁用 Live Literal 变换
    @DontMemoize函数禁止 Lambda 记忆化

    13.2 OptimizeNonSkippingGroups

    Feature Flag: OptimizeNonSkippingGroups（默认开启）

    对于不可跳过的 Composable 函数，省略 Replace Group，减少 SlotTable 开销。

    13.3 Intrinsic Remember

    Feature Flag: IntrinsicRemember（默认开启）

    将 remember 作为编译器内联操作处理，避免额外的 Group 和 Slot 开销。

    13.4 比较传播优化

    通过将参数状态从父函数传播到子函数，避免子函数重复调用 equals() 比较。

    十五、调试与诊断

    15.1 构建指标

    通过 metricsDestination 和 reportsDestination 选项，编译器会输出详细的指标：

    • 每个函数是否 Composable / Restartable / Skippable
    • 每个参数的稳定性
    • Group 数量
    • 不稳定类列表

    15.2 源码信息

    sourceInformation 选项（默认开启）在生成的代码中嵌入源码位置信息，供 Android Studio 的 Layout Inspector 使用。

    15.3 Trace Markers

    traceMarkersEnabled 选项在生成的代码中插入追踪标记，用于性能分析工具识别 Composable 函数的执行。

    十六、总结

    Compose 编译器插件的本质是将 声明式的 Composable 函数 变换为 与运行时深度协作的状态机。核心变换包括：

    1. 参数注入：$composer、$changed、$default — 为运行时提供控制通道
    2. Group 生成：Restart / Replace / Movable — 为 SlotTable 提供结构
    3. 跳过逻辑：基于 $changed 位掩码判断参数变化 — 避免不必要的重组
    4. 比较传播：将参数状态从父函数传递给子函数 — 减少运行时开销
    5. 重组回调：updateScope — 让运行时知道如何重新执行此范围
    6. 稳定性推断：编译期确定类型稳定性 — 决定跳过策略
    7. Lambda 记忆化：缓存 Composable Lambda 实例 — 避免引用变化导致的重组

    理解这些编译期变换，是理解 Compose 运行时行为的基石。
