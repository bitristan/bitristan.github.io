    Compose 框架中的 Group Anchor 索引机制详解

    一、背景：Compose 如何存储 UI？

    在 Compose 中，所有 UI 结构都存储在 SlotTable 中。

    可以把 SlotTable 想象成一个巨大的数组，每个元素是一个 group（UI 单元）：

    1

    组合函数执行时：

Text("Hello")      → 写入一个 group
Button { ... }     → 写入一个 group
Column { ... }     → 写入一个 group（包含子 groups）

SlotTable 内容：
[Text group] [Button group] [Column group] [Text group] ...
     0             1              2             3

    问题来了： 如果在中间插入或删除元素，后面的所有元素位置都会改变。

    初始状态：
[Button] [Text] [Image] [Column]
  0       1       2        3

插入 Spacer 后：
[Button] [Spacer] [Text] [Image] [Column]
  0        1       2       3        4

Text 的索引从 1 变成 2
Image 的索引从 2 变成 3
Column 的索引从 3 变成 4

    核心问题： 如果代码中直接用索引引用某个 group（比如"索引 1"），一旦数组改变，这个引用就失效了。

    解决方案： 需要一个稳定的引用机制——Anchor（锚点），它能自动跟踪 group 的位置变化。

    二、核心概念：三个层次的位置表示

    在深入 Anchor 之前，需要先理解 SlotTable 中位置的三个层次表示。

    2.1 Address（地址）

    定义： 数组中的绝对物理偏移量，忽略 Gap 的存在（源码注释：“an absolute offset into the array ignoring its gap”）。

    1
2
3
4

    物理数组：[A] [B] [C] │ GAP │ [D] [E] [F]
Address:   0   1   2          3   4   5
                         ↑
                    Gap 被忽略

    关键特性：

    • Address 和 Index 在 Gap 位于末尾时相等
    • Gap 在中间时，Address 和 Index 不同
    • SlotReader 利用这一特性优化访问（Gap 在末尾时直接用 Address）

    这是底层的实现细节，用户不应直接使用。

    2.2 Index（索引）

    定义： group 的逻辑位置，Gap 不计入索引。

    1
2
3
4

    逻辑视图：[A] [B] [C] │ GAP │ [D] [E] [F]
Index:     0   1   2          3   4   5
                         ↑
                    Gap 不计入索引

    这是我们通常理解的"第几个元素"。

    2.3 Anchor（锚点）

    定义： Index 的编码形式，自动维护有效性。

    1
2
3
4

    Anchor 存储值：
- A, B, C: location = 0, 1, 2    (正数：在 Gap 前)
- D:       location = 3           (正数：index == groupGapStart，<= 约定归为正数)
- E, F:    location = -2, -1      (负数：在 Gap 后，相对末尾位置)

    三者的关系：

    概念稳定性用途示例
    Address最不稳定内部实现细节Gap 移动就失效
    Index中等稳定当前逻辑位置插入元素后可能变化
    Anchor最稳定长期引用自动跟踪位置变化

    三、技术原理

    3.1 Gap Buffer 原理

    直观理解：

    想象你在填表格，每行一个数据。突然发现第 2 行需要插入一条新数据。

    普通做法： 把第 2 行及后面的内容全部下移一行 → 慢，需要移动很多数据

    Gap Buffer 做法： 预留一些空行（Gap），插入时先移动空行到目标位置 → 快

    具体示例：

    1
2
3
4
5
6
7
8

    普通数组插入（插入 X 到位置 1）：
[A] [B] [C] [D]           → 插入 X 到位置 1
[A] [X] [B] [C] [D]       ← 移动 B, C, D（3 个元素）

Gap Buffer 插入（Gap 在位置 3，插入 X 到位置 1）：
[A] [B] [C] │ 空 │ [D]    → Step 1: 移动 Gap 到位置 1
[A] │ 空 │ [B] [C] [D]    ← B, C 被搬移到 Gap 原位置（2 个元素，顺序不变）
[A] [X] [B] [C] [D]       ← Step 2: 在 Gap 处写入 X

    等等，这看起来没省多少？ 单次插入确实如此——Gap 搬移了 B、C（2 个），普通数组搬移了 B、C、D（3 个）。差距不大。

    真正的威力在连续操作：

    普通数组：连续在位置 1 插入 3 个元素
插入 X:  移动 B,C,D（3次）
插入 Y:  移动 X,B,C,D（4次）
插入 Z:  移动 Y,X,B,C,D（5次）
总计: 3 + 4 + 5 = 12 次搬移  → O(n²)

Gap Buffer：连续在位置 1 插入 3 个元素（Gap 初始在位置 1）
插入 X:  直接写入 Gap（0次搬移），Gap 右移到位置 2
插入 Y:  直接写入 Gap（0次搬移），Gap 右移到位置 3
插入 Z:  直接写入 Gap（0次搬移），Gap 右移到位置 4
总计: 0 次搬移  → O(n)

    Gap Buffer 的核心优势：

    • Gap 移到位后，连续插入/删除零开销——这是最大的性能来源
    • 单次远距离插入，Gap 搬移成本与普通数组相当，并无优势
    • 适合顺序执行、局部修改的模式（Compose 重组正是如此）

    适用场景：

    场景普通数组Gap Buffer说明
    连续插入到同一位置O(n²)O(n)Gap Buffer ✅ Gap 移一次，后续免费
    顺序追加O(1) 摊销O(1)相当，Gap 在末尾直接写入
    单次远距离插入O(n)O(n)相当，Gap Buffer 无优势
    随机访问O(1)O(1)相当

    3.2 双模式存储

    Anchor 用一个 Int 值存储位置，但有两种含义：

    • 正数模式（location >= 0）：从数组开头计数
    • 负数模式（location < 0）：从数组末尾倒数

    为什么用两种模式？

    类比： 在图书馆找书

    • 正数模式： 从入口开始数（适合前面的书）
    • 负数模式： 从出口开始数（适合后面的书）

    如果有人在中间插了新书：

    • 前面的书：从入口数的距离不变 ✓
    • 后面的书：从出口数的距离不变 ✓
    • 中间的书：需要重新计算

    这就是 Gap Buffer 的智慧： Gap 前后的元素相对固定，只有跨越 Gap 的才需要更新！

    Gap Buffer 结构：

┌───── Gap 前 ─────┐ ┌── Gap ──┐ ┌───── Gap 后 ─────┐
│  A   B   C       │ │         │ │  D   E   F       │
│  ↓   ↓   ↓       │ │         │ │  ↓   ↓   ↓       │
│ Index: 0, 1, 2   │ │   GAP   │ │ Index: 3, 4, 5   │
│  ↓   ↓   ↓       │ │         │ │  ↓   ↓   ↓       │
│ Loc:   0, 1, 2   │ │         │ │ Loc:   3,-2,-1   │
└──────────────────┘ └─────────┘ └──────────────────┘
     正数模式                          混合模式

关键：Gap 后的元素相对末尾位置稳定，负数模式不需要更新！

注意：D 的 index=3 恰好等于 groupGapStart，根据源码的 <= 约定，
它属于正数模式（location=3），而不是负数。详见第 4.1 节的边界条件分析。

    // SlotWriter 版本（有活跃 Writer 时使用，Gap 可能存在）
fun anchorIndex(anchor: GapAnchor): Int {
    val location = anchor.location
    return if (location >= 0) {
        location           // 正数：直接返回
    } else {
        size + location    // 负数：从末尾倒数
    }
}

// SlotTable 版本（无活跃 Writer 时使用，Gap 已移到末尾，location 全为正数）
fun anchorIndex(anchor: GapAnchor): Int {
    return anchor.location  // 直接返回，无需计算
}

    说明： 以下示例的 Gap 位置是假设的初始状态，实际位置取决于操作历史。

    4.1 插入操作

    场景：在位置 1 插入新 group

    关于边界条件： 源码中 anchor 正/负的判断条件是 index <= groupGapStart（包含等于）。这意味着恰好在 groupGapStart 位置的元素也属于"Gap 前"，使用正数 location。

┌───────────────────────────────────────────────┐
│ Index:     0   1   2   │ GAP │  3   4   5   6 │
│ Element:   A   B   C   │     │  D   E   F   G │
│ Anchor:    0   1   2   │     │  3  -3  -2  -1 │
└───────────────────────────────────────────────┘
              ↑                   ↑
          正数模式            正/负数模式

    Anchor 验证（源码：if (index <= groupGapStart) index else -(size - index)，size=7）：

    • A(0): 0 ≤ 3 → location=0 ✓
    • B(1): 1 ≤ 3 → location=1 ✓
    • C(2): 2 ≤ 3 → location=2 ✓
    • D(3): 3 ≤ 3 → location=3（正数！<= 使 D 也属于"Gap 前"）✓
    • E(4): 4 > 3 → location=-(7-4)=-3 ✓
    • F(5): 5 > 3 → location=-(7-5)=-2 ✓
    • G(6): 6 > 3 → location=-(7-6)=-1 ✓

    注意： D 在物理上位于 Gap 之后，但源码的 <= 约定让它使用正数编码。这是因为 anchorIndex 的正确性不依赖正/负选择——正数 3 和负数 -(7-3)=-4 都能解析到 index=3。源码统一将 index <= groupGapStart 归为正数，简化了边界处理。

    步骤 2：更新受影响的 Anchors

    为什么要更新？ Gap 移动后，原来在 Gap 前（正数模式）的 B、C、D 现在在 Gap 后，需要切换到负数模式。

    // Gap 向下移动（previousGapStart > newGapStart）的情况
var index = anchors.locationOf(newGapStart, size)
while (index < anchors.size) {
    val anchor = anchors[index]
    val location = anchor.location
    if (location >= 0) {
        anchor.location = -(size - location)
        index++
    } else break
}

    关键点：

    1. anchors.locationOf(newGapStart, size) 定位到解析位置 >= newGapStart 的第一个 anchor
    2. 遍历所有 location >= 0 的 anchor，切换为负数
    3. 遇到第一个 location < 0 的 anchor 时停止

    为什么没有范围上限检查？ 源码依赖 anchor 列表按解析位置有序的特性：

    • 在 newGapStart 之前的 anchor 应该已经是负数（之前 Gap 移动时转换的）
    • 在 previousGapStart 之后的 anchor 本来就是负数（初始创建时就是负数）
    • 因此只需遍历连续的正数 anchor 即可

    具体更新：

    // 切换公式：负数 location = -(currentSize - 原索引)
// currentSize 是移动 Gap 时的逻辑元素总数（插入前）
B: location = 1 → -(7 - 1) = -6   // size=7（A~G 共 7 个 group）
C: location = 2 → -(7 - 2) = -5
D: location = 3 → -(7 - 3) = -4

    注意： D 也会被切换为负数。源码的 <= 约定只在创建 anchor 时使用；Gap 移动后，所有原来在 Gap 前但现在在 Gap 后的 anchor 都需要切换，无论它们之前是否恰好在边界上。

    为什么插入前 size=7 算出的值，插入后（size=8）仍然正确？

    因为负数 location 表示"从末尾倒数"，而 Gap 移动时 B、C、D 的位置相对于末尾并未改变—— 插入前：末尾是 G（第 7 个），B 是倒数第 6 个 → location = -6 插入后：末尾仍是 G（第 8 个），B 还是倒数第 6 个 → location 仍然是 -6

    这正是负数模式的精髓：相对末尾的位置在插入操作中天然稳定。

    // ⚠️ 以下代码仅供理解原理，实际无法编译（internal API）
import androidx.compose.runtime.composer.gapbuffer.SlotTable
import androidx.compose.runtime.composer.gapbuffer.GapAnchor
import androidx.compose.runtime.Composer

fun main() {
    val table = SlotTable()

    // ===== 1. 创建初始结构 =====
    // 写入 7 个 groups，key 分别为 100-106
    table.write { writer ->
        repeat(7) { index ->
            writer.startGroup(100 + index, Composer.Empty)
            writer.endGroup()
        }
    }
    println("初始 groups 数量: ${table.groupsSize}")
    // 输出: 初始 groups 数量: 7

    // ===== 2. 为 G2（索引 2，key=102）创建 Anchor =====
    // 创建 anchor 后，即使数组改变，也能找到 G2
    val anchorForG2 = table.anchor(2)
    println("初始: location=${anchorForG2.location}, valid=${anchorForG2.valid}")
    // 输出: location=2, valid=true
    // 此时 G2 的索引是 2，anchor 存储的是正数 2

    // ===== 3. 验证初始位置 =====
    table.read { reader ->
        // anchorIndex() 根据 location 计算当前索引
        // 无活跃 Writer 时，location 始终为正数，直接返回即可
        val index = table.anchorIndex(anchorForG2)
        // 跳到目标 group
        repeat(index) { reader.skipGroup() }
        println("G2 索引: $index, key: ${reader.groupKey}")
        // 输出: G2 索引: 2, key: 102  ← 确实找到了 G2
    }

    // ===== 4. 在位置 1 插入新 group =====
    table.write { writer ->
        // 定位到索引 1（SlotWriter 没有 moveTo(index) 方法，
        // 实际中通过 startGroup/endGroup 自然推进，此处用 skipGroup 简化演示）
        repeat(1) { writer.skipGroup() }
        writer.beginInsert()
        writer.startGroup(999, Composer.Empty)
        writer.endGroup()
        writer.endInsert()
    }
    println("插入后 groups 数量: ${table.groupsSize}")
    println("插入后: location=${anchorForG2.location}")
    // 输出: location=3
    // Writer 关闭时会将 Gap 移到末尾（moveGroupGapTo(size)），
    // 此时所有 anchor 的 location 被转回正数。
    // G2 的新索引是 3，所以 location=3。

    // ===== 5. 验证 G2 的新位置（关键测试！）=====
    table.read { reader ->
        val newIndex = table.anchorIndex(anchorForG2)
        repeat(newIndex) { reader.skipGroup() }
        println("G2 新索引: $newIndex, key: ${reader.groupKey}")
        // 输出: G2 新索引: 3, key: 102  ← 仍然是同一个 group！
        // Writer 关闭后 location 已是正数，anchorIndex 直接返回
    }

    // ===== 6. 删除索引 1 的 group =====
    table.write { writer ->
        repeat(1) { writer.skipGroup() }
        writer.removeGroup()
    }
    println("删除后 groups 数量: ${table.groupsSize}")

    // ===== 7. 验证 G2 的最终位置 =====
    table.read { reader ->
        val finalIndex = table.anchorIndex(anchorForG2)
        repeat(finalIndex) { reader.skipGroup() }
        println("G2 最终索引: $finalIndex, key: ${reader.groupKey}")
        // 输出: G2 最终索引: 2, key: 102  ← 还是同一个 group！
        // 经过插入和删除，anchor 仍然正确工作
    }

    // ===== 8. 测试失效情况 =====
    // 创建一个会被删除的 group 的 anchor
    // 注意：在 write 块内必须使用 writer.anchor()，
    // 不能用 table.anchor()（会抛异常：use active SlotWriter to create an anchor）
    val tempAnchor = table.write { writer ->
        repeat(1) { writer.skipGroup() }
        writer.beginInsert()
        writer.startGroup(888, Composer.Empty)
        writer.endGroup()
        writer.endInsert()
        writer.anchor(1)
    }
    println("Temp anchor 初始有效: ${tempAnchor.valid}")
    // 输出: true

    // 删除这个 group
    table.write { writer ->
        repeat(1) { writer.skipGroup() }
        writer.removeGroup()
    }

    println("Temp anchor 删除后有效: ${tempAnchor.valid}")
    // 输出: false（group 被删除后，anchor 失效）

    // 使用失效的 anchor 会抛出异常
    try {
        table.anchorIndex(tempAnchor)
    } catch (e: IllegalArgumentException) {
        println("正确捕获异常: anchor 已失效")
    }
}

    六、失效处理

    6.1 何时失效？

    Anchor 会在以下情况下失效：

    1. 对应的 group 被删除
    2. 父 group 被删除（级联删除）
    3. SlotTable 被清空

    七、性能特点

    操作复杂度说明
    查询O(1)简单数学计算：if (loc >= 0) loc else size + loc
    创建O(log n)插入有序列表
    更新O(n) 最坏Gap 跨越时更新

    内存占用： 每个 anchor 对象 ≈ 12-20 字节（JVM 对象头 8-16 字节 + 一个 Int 字段 4 字节），不含 ArrayList 中的引用开销

    适用场景：

    • ✅ 长期引用：RecomposeScope、MovableContent
    • ❌ 一次性访问：直接用 index 更高效

    总结

    核心机制

    1. Gap Buffer：在数组中维护间隙，高效插入/删除
    2. 双模式存储：正数从前数，负数从后数
    3. 自动维护：Gap 移动时自动切换模式

    关键优势

    • Gap 后的元素负数模式不需要更新
    • 查询 O(1)，大部分更新免费
    • 内存占用小，适合大规模使用

    设计精髓

    1
2
3

    Gap 后的元素相对末尾稳定 → 负数模式不变
Gap 前的元素相对开头稳定 → 正数模式不变
只有跨越 Gap 的元素才需要切换模式

    这是 Compose 高性能运行时的重要基础设施，在稳定性和效率之间取得了完美平衡。

    FAQ

    Q1: Anchor 会占用多少内存？ A: 对象本身 ≈ 12-20 字节（JVM 对象头 + 一个 Int 字段），不含列表引用开销

    Q2: 如何判断 anchor 是否有效？
    A: 使用 anchor.valid 属性

    Q3: Gap 移动会影响性能吗？
    A: 会，需要更新跨越 Gap 的 anchors。但 Compose 的顺序执行特性使这种情况较少。

    Q4: Anchor 和 Index 的使用场景？
    A: 长期引用用 Anchor，一次性访问用 Index

    Q5: 为什么不直接用对象引用？
    A: 对象引用增加 GC 压力，且需要每个 group 维护位置信息。Anchor 更轻量。
