    1. 概述

    1.1 ART GC 的设计理念

    Android Runtime (ART) 的垃圾回收器设计遵循以下核心原则：

    • 低延迟优先：移动设备对交互响应敏感，GC 暂停时间直接影响用户体验
    • 内存效率：移动设备内存有限，需要高效利用每一字节
    • 并发执行：尽可能让 GC 与应用线程并行执行，减少暂停时间
    • 分代收集：利用"弱分代假说"——大多数对象朝生夕死，优化收集效率

    2.2 各空间详解

    2.2.1 Image Space (镜像空间)

    • 用途：存储预加载的类和对象，来自 boot.art 镜像文件
    • 特点：内存映射文件，只读，永不回收
    • 生命周期：从 Zygote 进程启动到系统关闭

    2.2.2 Zygote Space (Zygote 空间)

    • 用途：Zygote fork 后共享的只读空间
    • 特点：在 Zygote fork 前创建，fork 后变为只读
    • 回收策略：仅 Full GC 时考虑回收

    2.2.3 Malloc Space (RosAlloc/DlMalloc 空间)

    • 用途：普通对象分配
    • 分配器：
      • RosAlloc (Runs-of-Slots Allocator)：线程本地分配，减少锁竞争
      • DlMalloc：Doug Lea’s malloc，传统分配器
    • 特点：支持非移动收集器 (MS/CMS)

    2.2.4 Bump Pointer Space (指针碰撞空间)

    • 用途：Semi-Space 和 Mark-Compact 收集器使用
    • 特点：分配极快（只需移动指针），但需要整块连续内存
    • 适用场景：年轻代对象、移动式 GC

    2.2.5 Region Space (区域空间)

    • 用途：Concurrent Copying (CC) 收集器使用
    • 特点：将堆划分为多个固定大小的 Region，支持对象移动和压缩
    • 优势：减少内存碎片，支持并发整理

    2.2.6 Large Object Space (大对象空间)

    • 用途：存储大对象（默认 > 12KB）
    • 特点：直接使用 mmap 分配，避免拷贝开销
    • 实现：LargeObjectMapSpace（离散空间）或 FreeListSpace

    // heap.cc: GrowForUtilization
void Heap::GrowForUtilization(collector::GarbageCollector* collector_ran, ...) {
  if (gc_type != collector::kGcTypeSticky) {
    // 非 Sticky GC 后，下次做 Sticky GC
    next_gc_type_ = collector::kGcTypeSticky;
  } else {
    // Sticky GC 后，根据吞吐量决定下次 GC 类型
    if (current_gc_iteration_.GetEstimatedThroughput() * adjustment >=
        non_sticky_collector->GetEstimatedMeanThroughput()) {
      // Sticky GC 效率足够高，继续 Sticky
      next_gc_type_ = collector::kGcTypeSticky;
    } else {
      // 否则升级到 Partial 或 Full
      next_gc_type_ = non_sticky_gc_type;
    }
  }
}

    4.2 Mark-Sweep (MS/CMS)

    4.2.1 算法原理

    Mark-Sweep 分为两个阶段：

    1. 标记阶段 (Mark)：从 GC Root 出发，遍历对象图，标记所有可达对象
    2. 清除阶段 (Sweep)：遍历堆，回收未被标记的对象

    4.3 Semi-Space (SS)

    4.3.1 算法原理

    Semi-Space 将堆分为两个半空间：From-Space 和 To-Space

    1. 新对象分配在 To-Space
    2. GC 时，将存活对象从 From-Space 复制到 To-Space
    3. 交换两个空间的角色

    4.3.2 优缺点

    优点：

    • 分配极快（指针碰撞）
    • 自动整理内存，无碎片
    • 适合年轻代对象（存活率低）

    缺点：

    • 内存利用率低（只有一半可用）
    • 需要复制存活对象（存活率高时开销大）

    // 读取对象引用时的屏障伪代码
Object* ReadBarrier(Object* ref) {
  if (ref == nullptr) return nullptr;
  
  // 检查对象是否在 From-Space（需要移动）
  if (IsInFromSpace(ref)) {
    // 获取或创建转发地址
    Object* forwarded = GetForwardingAddress(ref);
    if (forwarded == nullptr) {
      // 首次访问，复制对象
      forwarded = CopyToToSpace(ref);
      SetForwardingAddress(ref, forwarded);
    }
    return forwarded;
  }
  return ref;
}

    4.5 Concurrent Mark-Compact (CMC)

    4.5.1 算法原理

    CMC 结合了 Mark-Sweep 和 Compaction 的优点：

    1. 并发标记：与应用线程并发执行标记
    2. 并发压缩：使用 userfaultfd 实现并发对象移动
    3. 滑动压缩：对象向一端滑动，保持分配顺序

void Heap::ChangeCollector(CollectorType collector_type) {
  collector_type_ = collector_type;
  gc_plan_.clear();
  
  switch (collector_type) {
    case kCollectorTypeCC:
      if (use_generational_gc_) {

      gc_plan_.push_back(collector::kGcTypeFull);
      break;

      gc_plan_.push_back(collector::kGcTypeSticky);
      gc_plan_.push_back(collector::kGcTypePartial);
      gc_plan_.push_back(collector::kGcTypeFull);
      break;
      
    case kCollectorTypeSS:
      gc_plan_.push_back(collector::kGcTypeFull);
      break;
  }
}

template <bool kPrecise>
void Thread::VisitRoots(RootVisitor* visitor) {
  const uint32_t thread_id = GetThreadId();
  
  // 1. 线程对象本身
  visitor->VisitRootIfNonNull(&tlsPtr_.opeer, RootInfo(kRootThreadObject, thread_id));
  visitor->VisitRootIfNonNull(&tlsPtr_.current_peer, RootInfo(kRootThreadObject, thread_id));
  
  // 2. 异常对象
  visitor->VisitRoot(&tlsPtr_.exception, RootInfo(kRootNativeStack, thread_id));
  
  // 3. Monitor 对象
  visitor->VisitRootIfNonNull(&tlsPtr_.monitor_enter_object, ...);
  
  // 4. JNI 局部引用
  tlsPtr_.jni_env->VisitJniLocalRoots(visitor, RootInfo(kRootJNILocal, thread_id));
  
  // 5. JNI Monitor
  tlsPtr_.jni_env->VisitMonitorRoots(visitor, RootInfo(kRootJNIMonitor, thread_id));
  
  // 6. Handle Scope (编译器生成的句柄)
  HandleScopeVisitRoots(visitor, thread_id);
  
  // 7. 栈帧引用 (通过 ReferenceMapVisitor 遍历)
  ReferenceMapVisitor<...> mapper(this, &context, visitor_to_callback);
  mapper.WalkStack<...>();
}

    // mark_sweep.cc
class CheckpointMarkThreadRoots : public Closure, public RootVisitor {
  void Run(Thread* thread) override {
    // 在目标线程上执行
    thread->VisitRoots(this, kVisitRootFlagAllRoots);
    // 通过屏障
    mark_sweep_->GetBarrier().Pass(self);
  }
  
  void VisitRoots(mirror::Object*** roots, size_t count, const RootInfo& info) override {
    for (size_t i = 0; i < count; ++i) {
      mark_sweep_->MarkObjectNonNullParallel(*roots[i]);
    }
  }
};

void MarkSweep::MarkRootsCheckpoint(Thread* self, bool revoke_ros_alloc...) {
  CheckpointMarkThreadRoots check_point(this, revoke_ros_alloc...);
  ThreadList* thread_list = Runtime::Current()->GetThreadList();
  
  // 在所有线程上运行 checkpoint
  size_t barrier_count = thread_list->RunCheckpoint(&check_point);
  
  // 等待所有线程完成
  if (barrier_count > 0) {
    gc_barrier_->Increment(self, barrier_count);
  }
}

    // heap.cc: CollectGarbageInternal
collector::GcType Heap::CollectGarbageInternal(collector::GcType gc_type,
                                               GcCause gc_cause, ...) {
  // 1. 等待其他 GC 完成
  WaitForGcToCompleteLocked(gc_cause, self);
  
  // 2. 选择收集器
  collector::GarbageCollector* collector = FindCollectorByGcType(gc_type);
  
  // 3. 运行 GC
  collector->Run(gc_cause, clear_soft_references);
  
  // 4. 处理引用
  SelfDeletingTask* clear = reference_processor_->CollectClearedReferences(self);
  
  // 5. 调整堆大小
  GrowForUtilization(collector, bytes_allocated_before_gc);
  
  // 6. 完成 GC
  FinishGC(self, gc_type);
}

    // mark_sweep.cc: RunPhases
void MarkSweep::RunPhases() {
  Thread* self = Thread::Current();
  
  // ===== 阶段 1: 初始化 =====
  InitializePhase();
  
  if (IsConcurrent()) {
    // ===== 阶段 2: 并发标记 (CMS) =====
    {
      ReaderMutexLock mu(self, *Locks::mutator_lock_);
      MarkingPhase();
    }
    
    // ===== 阶段 3: 暂停阶段 =====
    ScopedPause pause(this);
    PausePhase();
    RevokeAllThreadLocalBuffers();
  } else {
    // 非并发 MS：全部在暂停中执行
    ScopedPause pause(this);
    MarkingPhase();
    PausePhase();
    RevokeAllThreadLocalBuffers();
  }
  
  // ===== 阶段 4: 并发回收 =====
  {
    ReaderMutexLock mu(self, *Locks::mutator_lock_);
    ReclaimPhase();
  }
  
  // ===== 阶段 5: 完成 =====
  FinishPhase();
}

    void MarkSweep::MarkingPhase() {
  Thread* self = Thread::Current();
  
  // 1. 绑定免疫空间的 bitmap
  BindBitmaps();
  
  // 2. 处理 dirty cards
  heap_->ProcessCards(GetTimings(), false, true, GetGcType() != kGcTypeSticky);
  
  // 3. 标记根集
  MarkRoots(self);
  
  // 4. 递归标记可达对象
  MarkReachableObjects();
  
  // 5. 预清理 dirty cards（减少暂停时间）
  PreCleanCards();
}

void MarkSweep::MarkRoots(Thread* self) {
  if (Locks::mutator_lock_->IsExclusiveHeld(self)) {
    // 暂停中：直接访问所有根
    Runtime::Current()->VisitRoots(this);
  } else {
    // 并发：使用 checkpoint 机制
    MarkRootsCheckpoint(self, kRevokeRosAllocThreadLocalBuffersAtCheckpoint);
    MarkNonThreadRoots();
    MarkConcurrentRoots(kVisitRootFlagAllRoots | kVisitRootFlagStartLoggingNewRoots);
  }
}

    void MarkSweep::PausePhase() {
  Thread* self = Thread::Current();
  
  if (IsConcurrent()) {
    // 1. 重新标记并发期间修改的根
    ReMarkRoots();
    
    // 2. 扫描 dirty objects
    RecursiveMarkDirtyObjects(true, accounting::CardTable::kCardDirty);
  }
  
  // 3. 交换 live/allocation stack
  heap_->SwapStacks();
  
  // 4. 撤销线程本地分配栈
  RevokeAllThreadLocalAllocationStacks(self);
  
  // 5. 设置引用处理器
  ReferenceProcessor* rp = GetHeap()->GetReferenceProcessor();
  rp->Setup(self, this, true, GetCurrentIteration()->GetClearSoftReferences());
  rp->EnableSlowPath();
}

    void MarkSweep::ReclaimPhase() {

}

    7. 引用处理机制

    7.1 Java 引用类型

    ART 支持四种 Java 引用类型：

    类型用途回收时机
    StrongReference普通引用永不回收（直到不可达）
    SoftReference软引用内存不足时回收
    WeakReference弱引用下次 GC 时回收
    PhantomReference虚引用随时可能回收，get() 永远返回 null

    此外还有 FinalizerReference，用于实现 Object.finalize() 方法。

}

    RequestCollectorTransition(foreground_collector_type_, ...);

    // 后台：更激进的内存回收
    RequestCollectorTransition(background_collector_type_, ...);
  }
}

    状态收集器特点
    前台CC/CMS低延迟，并发执行
    后台SS/CC Background高吞吐量，积极压缩

    void MarkSweep::PreCleanCards() {
  if (kPreCleanCards && IsConcurrent()) {
    // 处理 dirty cards
    heap_->ProcessCards(GetTimings(), false, true, false);
    
    // 标记线程根（减少暂停时的工作量）
    MarkRootsCheckpoint(self, false);

    9. 总结

    9.1 ART GC 核心特性

    特性描述
    并发收集GC 与应用线程并行执行，最小化暂停时间
    分代收集Sticky/Partial/Full 三级收集，优化年轻代对象
    多种收集器CC (默认)、CMS、SS、CMC，适应不同场景
    Read BarrierCC 使用读屏障实现并发对象移动
    Region-basedCC 使用 Region Space，减少碎片
    动态调整根据吞吐量自动调整 GC 类型和堆大小

    9.2 收集器选择建议

    场景推荐收集器原因
    Android 8+ 默认CC低延迟，自动压缩
    低内存设备CMS/SS内存开销小
    后台应用SS/CC Background积极释放内存
    Zygote ForkSS压缩 Zygote 空间
