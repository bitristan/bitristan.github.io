    Android Input ANR 机制深度解析：从三个典型案例理解超时检测原理

    一、引言

    ANR（Application Not Responding）是 Android 开发中常见的问题，指应用程序在规定时间内未能响应系统请求。常见的 ANR 类型包括：

    • Service ANR：前台服务 5 秒未执行完成，后台服务 200 秒
    • Broadcast ANR：前台广播 10 秒，后台广播 60 秒
    • ContentProvider ANR：内容提供者 10 秒未响应
    • Input ANR：输入事件 5 秒未处理完成

    本文聚焦于 Input ANR，它是最常见也最容易误解的 ANR 类型。通过三个典型案例的分析，深入理解 Input ANR 的触发机制。

    为什么选择 Socket 而非 Binder？

    输入事件分发使用 socketpair 而非 Binder 进行进程间通信，原因如下：

    特性SocketBinder
    通信方式全双工，可双向独立发送需要握手机制
    可阻塞状态可检测对端阻塞状态无法检测
    死锁风险低，可通过 poll 检测高，可能导致系统死锁
    适用场景高频输入事件一般 RPC 调用

    关键优势：InputDispatcher 可以通过 poll 检测 Socket 对端（应用进程）是否阻塞，从而判断事件是否能被及时处理。

    InputDispatcher 维护三个关键队列：

    sleep 结束T=15s空没有事件在等待

    // frameworks/native/services/inputflinger/dispatcher/InputDispatcher.cpp
class AnrTracker {
    // 使用最小堆存储超时时间
    std::map<nsecs_t, sp<Connection>> mAnrTimeouts;
    
    void insert(nsecs_t timeoutTime, const sp<Connection>& connection) {
        mAnrTimeouts.insert({timeoutTime, connection});
    }
    
    nsecs_t firstTimeout() const {
        if (mAnrTimeouts.empty()) {
            return LONG_LONG_MAX;
        }
        return mAnrTimeouts.begin()->first;
    }
};

    // frameworks/native/services/inputflinger/dispatcher/InputDispatcher.cpp
void InputDispatcher::dispatchOnce() {
    nsecs_t nextWakeupTime = LONG_LONG_MAX;
    
    // 获取最近的 ANR 检查时间
    const nsecs_t nextAnrCheck = processAnrsLocked();
    nextWakeupTime = std::min(nextWakeupTime, nextAnrCheck);
    
    // 计算超时时间
    nsecs_t currentTime = now();
    int timeoutMillis = toMillisecondTimeoutDelay(currentTime, nextWakeupTime);
    
    // epoll_wait 阻塞，直到有事件或超时
    mLooper->pollOnce(timeoutMillis);
}

    nsecs_t InputDispatcher::processAnrsLocked() {
    const nsecs_t currentTime = now();
    
    // 检查 waitQueue 中是否有超时的事件
    while (!mAnrTracker.empty()) {
        nsecs_t timeoutTime = mAnrTracker.firstTimeout();
        if (timeoutTime > currentTime) {
            return timeoutTime;  // 还没超时，返回下次检查时间
        }
        
        // 超时了，触发 ANR
        sp<Connection> connection = mAnrTracker.firstTimeoutConnection();
        onAnrLocked(connection);
    }
    
    return LONG_LONG_MAX;
}

    六、总结

    Input ANR 的本质

    Input ANR 的触发需要同时满足三个条件：

    1. 输入事件产生：用户触摸或按键
    2. 事件等待确认：事件在 waitQueue 中
    3. 超时未确认：超过 5 秒未收到 FINISHED 消息

    onClick 异步执行的意义

    View 框架将 onClick 设计为异步执行是一个精妙的设计：

    • 用户体验：让视觉状态先更新，再执行点击逻辑
    • ANR 避免：事件确认在 onClick 阻塞之前完成，避免误判
    • 灵活性：应用可以在 onClick 中执行耗时操作而不触发 ANR（前提是没有后续输入事件）

    实际开发注意事项

    1. 避免在 onClick 中执行耗时操作：虽然单次点击不会触发 ANR，但如果用户连续点击，仍会触发 ANR

    2. 理解异步执行的边界：onClick 是异步的，但 onTouch、onLongPress 等回调可能不是

    3. 监控输入事件处理时间：即使不触发 ANR，长时间阻塞也会影响用户体验

    4. ANR 弹窗的显示机制：ANR 弹窗由 system_server 显示，即使应用主线程完全阻塞，弹窗也能正常显示
