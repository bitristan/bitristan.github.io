    Compose 重组机制深度解析

    一、引言：什么是重组？

    1.1 重组的定义

    在 Jetpack Compose 中，重组（Recomposition） 是响应式编程模型的核心机制。当应用状态发生变化时，Compose 会重新执行相关的可组合函数（@Composable 函数），以更新 UI 树来反映最新的状态。

    这是一个声明式的过程：你描述"UI 应该是什么样子"，Compose 会自动处理"如何从当前状态转换到目标状态"。

    1.2 重组的特点

    智能性（Intelligent）

    Compose 只重组受影响的部分，而不是整个 UI 树。这通过以下机制实现：

    • 作用域追踪：可重启的可组合函数（restartable composable）会有一个 RecomposeScope，记录该函数读取了哪些状态
    • 增量更新：基于现有 UI 树进行局部更新，而不是重新构建整棵树
    • 智能跳过：如果输入参数未变化，跳过重组

    增量性（Incremental）

    重组是基于现有状态进行的增量更新：

    • 首次组合（Initial Composition）：从零开始构建 UI 树
    • 重组（Recomposition）：在现有基础上更新 UI 树

    可控性（Controllable）

    开发者可以通过多种方式控制重组的范围和行为：

    • remember：缓存值，避免在重组时重新计算
    • key：标识列表项，优化列表重组
    • derivedStateOf：创建派生状态，减少不必要的重组
    • @Stable / @Immutable：标记类型稳定性，帮助编译器优化

override fun setContent(content: @Composable () -> Unit) {
    owner.setOnReadyForComposition { composeViewContext ->
        if (!disposed) {
            val lifecycle = composeViewContext.lifecycleOwner.lifecycle
            lastContent = content
            if (addedToLifecycle == null) {
                // 添加生命周期观察者

    }

    // Recomposer.kt:1158
internal override fun composeInitial(
    composition: ControlledComposition,
    content: @Composable () -> Unit,
) {
    val composerWasComposing = composition.isComposing
    
    // 1. 注册 composition
    val newComposition = synchronized(stateLock) {
        if (_state.value > State.ShuttingDown) {
            val new = composition !in knownCompositionsLocked()
            if (new) {
                registerCompositionLocked(composition)
            }
            new
        } else {
            true
        }
    }
    
    try {
        // 2. 执行组合
        composing(composition, null) { composition.composeContent(content) }
    } catch (e: Throwable) {
        if (newComposition) {
            synchronized(stateLock) { unregisterCompositionLocked(composition) }
        }
        processCompositionError(e, composition, recoverable = true)
        return
    }
    
    // 3. 处理 movable content
    try {
        performInitialMovableContentInserts(composition)
    } catch (e: Throwable) {
        processCompositionError(e, composition, recoverable = true)
        return
    }
    
    // 4. 应用变更
    try {
        composition.applyChanges()
        composition.applyLateChanges()
    } catch (e: Throwable) {
        processCompositionError(e)
        return
    }
}

    关键步骤详解：

    步骤 1：注册 composition

    将 composition 添加到 Recomposer 管理的列表中，这样当状态变化时，Recomposer 可以通知它进行重组。

    步骤 2：执行组合

    1

    composing(composition, null) { composition.composeContent(content) }

    这会调用 CompositionImpl.composeContent()，最终调用 GapComposer.composeContent()。

    步骤 3：处理 movable content

    如果组合中有 movableContentOf，需要在新位置插入内容。

    步骤 4：应用变更

    将生成的变更应用到 UI 树。

    // GapComposer.kt (简化版)
internal class GapComposer(
    private val applier: Applier<*>,
    private val parentContext: CompositionContext,
    private val slotTable: SlotTable,
    private val abandonSet: MutableSet<RememberObserver>,
    private val changes: Changes,
    private val lateChanges: Changes,
    private val composition: CompositionImpl,
    private val observerHolder: CompositionObserverHolder,
) : InternalComposer {

    override fun composeContent(
        invalidationsRequested: ScopeMap<RecomposeScopeImpl, Any>,
        content: @Composable () -> Unit,
        shouldPause: ShouldPauseCallback?
    ) {
        runtimeCheck(changes.isEmpty()) { "Expected applyChanges() to have been called" }
        this.shouldPauseCallback = shouldPause
        try {
            // 核心逻辑委托给 doCompose() 执行
            doCompose(invalidationsRequested, content)
        } finally {
            this.shouldPauseCallback = null
        }
    }
}

    doCompose() 方法内部流程：

    doCompose() 是组合执行的核心方法，它负责：

    1. 更新失效信息（updateComposerInvalidations）
    2. 启动根组合（startRoot）
    3. 执行可组合函数
    4. 结束组合（endRoot）

@Composable
fun MyComposable(text: String) {

// 编译后的代码（简化）
@Composable
fun MyComposable(text: String, $composer: Composer, $changed: Int) {
    // startRestartGroup 创建一个可重启的 scope
    $composer.startRestartGroup(key)
    
    // $changed 是位掩码，用于追踪哪些参数发生了变化
    // 编译器会生成参数比较逻辑，如果参数未变化则跳过重组
    
    if ($changed and 0b0001 != 0 || !$composer.skipping) {
        // 参数变化或强制重组时执行
        Text(text, $composer, 0)
    } else {
        // 跳过重组
        $composer.skipToGroupEnd()
    }
    
    // endRestartGroup 返回 ScopeUpdateScope，用于更新重组回调
    $composer.endRestartGroup()?.updateScope { next ->
        MyComposable(text, next, $changed or 0b0001)
    }
}

    $changed 参数的作用：

    $changed 是一个位掩码（bitmask），用于追踪参数的变化状态：

    • 每个参数占用 2 个 bit：一个表示"是否可能变化"，一个表示"是否确实变化"
    • 编译器通过比较当前值和缓存的值来确定参数是否变化
    • 如果所有参数都未变化且没有强制 invalidation，则跳过重组

    Group 的作用：

    • 标识：每个 group 有一个唯一的 key，标识对应的可组合函数
    • 层级：group 可以嵌套，形成树结构，对应 UI 树的层级
    • 状态存储：group 的 slots 可以存储 remember 的值、节点引用等

    // UiApplier.android.kt
internal class UiApplier(root: LayoutNode) : AbstractApplier<LayoutNode>(root) {

    override fun insertTopDown(index: Int, instance: LayoutNode) {
        // 忽略。实际的插入在 insertBottomUp 中执行
        // 这样可以避免子节点进入树时收到重复通知
    }

    override fun insertBottomUp(index: Int, instance: LayoutNode) {
        // 从下到上插入节点（子节点先于父节点被插入）
        current.insertAt(index, instance)
    }

    override fun remove(index: Int, count: Int) {
        // 移除节点
        current.removeAt(index, count)
    }

    override fun move(from: Int, to: Int, count: Int) {
        // 移动节点
        current.move(from, to, count)
    }
}

    insertTopDown 与 insertBottomUp 的使用场景差异：

    • insertTopDown：父节点先于子节点被插入。适用于大多数情况，特别是当父节点不需要知道子节点信息就能完成初始化时。
    • insertBottomUp：子节点先于父节点被插入。适用于父节点需要依赖子节点信息的场景，例如某些布局需要先测量子节点才能确定自身尺寸。

    UiApplier 选择 BottomUp 策略的原因：

    在 UiApplier 中，insertTopDown 是空实现，实际插入只在 insertBottomUp 中执行。这样做是为了避免子节点进入树时收到重复通知——如果使用 top-down 策略，父节点先插入会触发通知，然后每个子节点插入时又会向上传播通知，导致指数级的通知数量。使用 bottom-up 策略，通知数量与节点数量成线性关系。

    四、重组的触发机制

    重组的触发是 Compose 响应式编程模型的核心。当状态发生变化时，Compose 需要精确地知道哪些可组合函数需要重新执行。这个过程涉及多个组件的协作。

    4.1 State 与 Snapshot 系统

    MutableState 的实现原理

    • StateObjectImpl：抽象类，实现 StateObject，提供读取者类型追踪功能
    • SnapshotMutableStateImpl：具体实现类，继承 StateObjectImpl，提供 value 的 getter/setter

public interface StateObject {
    // 状态记录链表的头部
    public val firstStateRecord: StateRecord

// StateObjectImpl.kt - 抽象基类，提供读取者追踪
internal abstract class StateObjectImpl internal constructor() : StateObject {

    private val readerKind = AtomicInt(0)

// SnapshotState.kt - MutableState 的具体实现
internal open class SnapshotMutableStateImpl<T>(
    value: T,
    override val policy: SnapshotMutationPolicy<T>,
) : StateObjectImpl(), SnapshotMutableState<T> {

    // 内部维护的状态记录链表头
    // StateStateRecord 存储 (snapshotId, value) 对
    private var next: StateStateRecord<T> = StateStateRecord(currentSnapshot().snapshotId, value)

    // firstStateRecord 是一个 getter，返回链表头
    override val firstStateRecord: StateRecord
        get() = next

    @Suppress("UNCHECKED_CAST")
    override var value: T
        get() = next.readable(this).value
        set(value) = next.withCurrent {
            if (!policy.equivalent(it.value, value)) {
                next.overwritable(this, it) { this.value = value }
            }
        }
}

    StateRecord 的作用

    每个 MutableState 内部维护一个 StateRecord，它记录了：

    • 当前值
    • 所属的 Snapshot ID
    • 版本信息

    Snapshot 的读写跟踪机制

    Snapshot 系统是 Compose 状态管理的基础。当读取或修改状态时，Snapshot 系统会记录这些操作，并在合适的时机触发重组。

    当读取 state.value 时，Snapshot 系统会记录"在当前 snapshot 中读取了 state"，并返回对应 snapshot 版本的值。

    当修改 state.value = newValue 时，Snapshot 系统会创建新的 StateRecord，记录"在当前 snapshot 中修改了 state"，并标记相关的 RecomposeScope 为 invalid。

    4.2 如何记录依赖关系

    在组合过程中，当读取状态时，Composer 会记录这个依赖关系。CompositionImpl 维护了一个 observations 映射表，记录了哪些 RecomposeScope 读取了哪些状态。

    依赖建立的流程：

    1. Composer 执行到某个 composable 函数
    2. 函数内部读取 state.value
    3. Snapshot 系统检测到状态读取
    4. 调用 composer.recordReadOf(state)
    5. 更新 observations: observations[state] += currentRecomposeScope

    4.3 RecomposeScope 的作用

    可重启的可组合函数（restartable composable）会有一个 RecomposeScope，它跟踪该函数需要重组的条件。

    什么是可重启的 Composable？

    编译器会为符合以下条件的 composable 函数生成 startRestartGroup() 和 endRestartGroup() 调用：

    • 函数体直接使用了状态（读取 state.value）
    • 函数有参数，且参数类型不是全不可变的

    内联的 composable lambda 或某些特殊情况下不会创建 scope。

    Scope 记录的信息：

    • anchor：指向 SlotTable 中的 group，重组时从这里开始
    • block：重组时需要执行的代码（编译器生成）
    • trackedInstances：该 scope 读取的所有状态实例
    • owner：负责管理这个 scope 的 Composition

    RecomposeScope 的生命周期：

    • 首次组合：startRestartGroup() 创建 RecomposeScope → 执行 composable 函数并记录状态依赖 → endRestartGroup() 返回 scope
    • 状态变化：state.value 被修改 → scope.invalidate() 被调用 → 标记为需要重组
    • 重组：recompose() 通过 anchor 定位到 group → 执行 scope.block() 重新执行 composable 函数

    4.4 状态变化如何触发重组

    现在我们来看完整的状态变化触发重组的流程。

    完整流程：

    1. 修改 state.value：用户代码修改状态，调用 SnapshotMutableStateImpl 的 setter
    2. Snapshot 写入观察者触发：setter 通过 next.overwritable() 写入新值，触发 Snapshot.writeObserver
    3. 通知状态观察者：Snapshot 系统通知所有观察该状态的组件（DerivedState、RecomposeScope）
    4. 标记 RecomposeScope 为 invalid：查找到读取该状态的所有 scope，调用 scope.invalidate()
    5. Recomposer 收集 invalidations：将需要重组的 scope 收集起来
    6. 帧调度重组：Recomposer 在下一帧执行重组（与屏幕刷新对齐）
    7. 执行重组：调用 recompose()，通过 anchor 定位并重新执行对应的 composable 函数

    五、重组执行流程（Recomposition）

    当状态变化被检测到，相关的 RecomposeScope 被标记为 invalid 后，Recomposer 会在合适的时机执行重组。

    5.1 Recomposer 的调度机制

    重组循环

    Recomposer 维护一个协程驱动的重组循环，不断地等待状态变化，然后执行重组。

    帧调度

    Compose 的重组与屏幕刷新帧对齐。这样做的好处：

    • 性能优化：避免在单帧内多次重组
    • 一致性：重组结果与屏幕刷新同步
    • 节流：多次状态修改只触发一次重组

    5.2 识别需要重组的 Scopes

    在执行重组前，需要先收集所有被标记为 invalid 的 scopes。对于 DerivedState，采用条件性 invalidation，只有在派生值真正变化时才触发重组。

    5.3 执行重组：Composer 的角色

    与首次组合的区别

    方面首次组合重组
    SlotTable空的已有数据
    执行范围所有 composable只执行 invalid scopes
    性能优化无可以跳过未变化的 group
    Anchor需要创建已存在，用于定位

    Composer 的智能跳过

    在重组时，Composer 会智能地跳过不需要重组的部分。只有当一个 scope 在 invalidations 列表中时，才会重新执行，否则直接跳过。

    5.4 智能重组：如何避免不必要的工作

    比较机制

    Composer 使用多种机制来判断是否需要重组：

    1. Key 比较：每个 group 都有一个 key，如果 key 不匹配，说明组合结构发生了变化
    2. $changed 位掩码：编译器生成的参数变化追踪，用于判断参数是否需要重新比较
    3. Invalidation 标记：scope 是否被标记为 invalid（状态变化触发）

    何时跳过重组，何时执行重组

    跳过重组的条件（全部满足）：

    1. group key 匹配
    2. 参数未变化
    3. 没有强制 invalidation
    4. 没有父 scope invalidation

    执行重组的条件（任一满足）：

    1. group key 不匹配（结构变化）
    2. 参数发生变化
    3. scope 被 invalidation
    4. 父 scope 需要重组

    5.5 增量更新 SlotTable

    重组时，SlotTable 需要增量更新，而不是完全重建。

    • 更新现有的 group：直接修改 slots 中的值
    • 插入新的 group：使用 beginInsert/endInsert 批量操作
    • 删除不再需要的 group：使用 removeGroup

    在插入和删除过程中，Anchor 自动维护其指向的位置。

    5.6 应用重组结果

    重组完成后，需要将变更应用到 UI 树。

    • applyChanges()：应用主要变更
    • applyLateChanges()：应用延迟变更（如 MovableContent）

    六、重组的输入与输出

    6.1 重组的输入

    重组需要以下输入：

    状态（State）

    • MutableState.value 的当前值
    • CompositionLocal 的值
    • 环境值（如 LocalContext）

    组合函数

    • @Composable 函数的代码逻辑
    • 参数值
    • $changed 标记（参数是否变化）

    上下文环境

    • 父 composition 的状态
    • SlotTable 的现有数据（重组时）
    • RecomposeScope 信息

    6.2 重组的输出

    重组产生以下输出：

    SlotTable

    • 更新后的 group 树
    • 新的 anchor 引用
    • remember 的值

    Change List

    • 对 UI 树的操作指令
    • insert/remove/move 等

    副作用（Side Effects）

    • LaunchedEffect：启动协程
    • DisposableEffect：管理生命周期
    • SideEffect：在组合完成后执行

    7.1 CompositionContext

    CompositionContext 是连接 Composition 与 Recomposer 的桥梁。

    核心职责：

    • 触发重组：当状态变化时，通知 Recomposer 调度重组

    7.3 DerivedState 与条件重组

    derivedStateOf 创建派生状态，实现高效的"条件性重组"。

    使用场景：

    1
2
3
4
5

    val items by remember { mutableStateOf(listOf(1, 2, 3, 4, 5)) }
val hasManyItems by remember { derivedStateOf { items.size > 3 } }

// 只有当 items.size 跨越 3 的阈值时，才会触发重组
// 而不是每次 items 变化都触发重组

    实现原理：

    • 内部维护一个"依赖快照"，记录计算结果
    • 只有当依赖的状态变化导致计算结果真正改变时，才触发 invalidation
    • 这是一种"选择性订阅"机制，减少不必要的重组

    7.4 MovableContent

    MovableContent 允许内容在 composition tree 中移动，同时保留其内部状态。

    典型使用场景：

    • 列表项重新排序时保留状态
    • 条件渲染导致的位置变化
    • 动画过程中的内容移动

    实现机制：

    • 使用 movableContentOf 或 movableContentWithReceiverOf 创建
    • Compose 在 apply 阶段特殊处理移动操作
    • 使用 applyLateChanges() 延迟应用移动相关的变更

    7.6 Remember 机制

    remember 是 Compose 状态持久化的基础 API。

    核心变体：

    • remember：在重组过程中保持值不变
    • rememberSaveable：额外支持配置更改（屏幕旋转、进程重建）后恢复
    • remember { mutableStateOf() }：创建带持久化的状态

    工作原理：

    • 值存储在 SlotTable 的 slots 数组中
    • 通过 group 的 anchor 定位，重组时能找回之前的值
    • rememberSaveable 使用 SavedStateRegistry 保存到 Bundle

    7.7 SideEffect 与 LaunchedEffect

    副作用 API 用于处理组合之外的"副作用"操作。

    SideEffect：

    • 在组合成功完成后执行
    • 用于同步状态给非 Compose 代码
    • 每次重组都会执行

    LaunchedEffect：

    • 在组合时启动协程
    • 当 key 变化时取消并重新启动
    • 离开 composition 时自动取消

    DisposableEffect：

    • 需要手动管理生命周期
    • 提供 onDispose 回调用于清理资源
    • 适用于注册/取消注册监听器等场景

    八、总结与最佳实践

    8.1 重组机制总结

    输入 → 处理 → 输出

    • 输入：State + Composable + SlotTable
    • 处理：Composer 执行组合，生成变更
    • 输出：更新后的 SlotTable + Changes + Side Effects

    流程

    1. State 变化
    2. Scope invalidate
    3. Recomposer 调度
    4. Composer 执行
    5. Applier 应用

    8.2 性能优化建议

    1. 合理使用 remember：缓存计算结果和对象
    2. 避免不必要的重组：使用 derivedStateOf、key 等优化
    3. 使用 key 优化列表：为列表项提供稳定的 key
    4. 稳定性的重要性：标记 @Stable 或 @Immutable 帮助编译器优化

    8.3 调试技巧

    1. 查看重组次数：使用 Layout Inspector
    2. 日志跟踪：添加日志观察重组频率
    3. 性能分析：使用 Profiler 分析性能瓶颈
