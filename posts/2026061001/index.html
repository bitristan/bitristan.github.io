    Jetpack Compose Snapshot 系统：架构设计与实现原理深度解析

    Jetpack Compose Snapshot 系统：架构设计与实现原理深度解析

    Jetpack Compose 的声明式 UI 模型建立在一个精巧的状态管理基础设施之上 —— Snapshot（快照）系统。它借鉴了数据库领域成熟的 MVCC（多版本并发控制）思想，在不依赖显式锁的前提下，为 Compose 的重组机制提供了线程安全的状态读写、精确的变更追踪以及高效的失效通知能力。

    本文将从源码层面深入剖析 Snapshot 系统的核心数据结构、MVCC 实现细节、快照生命周期、冲突检测与解决机制，以及它如何与 Composition 和 SnapshotStateObserver 协作完成从"状态变更"到"UI 重组"的完整闭环。

    一、为什么需要 Snapshot？

    在传统的 Android View 体系中，状态与 UI 的同步通常依赖手动调用 invalidate() 或 notifyDataSetChanged()。Compose 采用了声明式模型：UI 是状态的函数，状态变化时框架自动重新计算差异并更新视图树。这要求运行时解决三个核心问题：

    状态隔离 —— 多个 Composable 函数可能在同一帧内并发读取和修改状态，需要保证每个 Composable 看到的是一致性的状态视图，而非被其他并发操作中途修改的"脏"数据。

    变更追踪 —— 运行时必须精确知道哪些状态被哪些 Composable 读取了，才能在状态变化时只重新执行受影响的作用域，而非整棵组合树。

    线程安全 —— Compose 允许从任意线程修改状态（例如网络回调更新 mutableStateOf），但组合和布局通常在主线程执行，两者之间需要无锁协调。

    Snapshot 系统用一个统一的抽象同时解决了这三个问题。它的核心思想是：每次对状态的读写都发生在一个"快照"上下文中，快照看到的是一致性的状态快照（类似数据库事务的快照隔离），写操作被缓冲在当前快照中直到显式提交，提交时通过三方合并检测冲突。

    二、核心数据结构

    2.1 StateRecord 链表 —— 多版本的物理存储

    Snapshot 系统的基石是 StateRecord 链表。每个 StateObject（例如 MutableState<T> 的内部实现）维护一条由 StateRecord 节点组成的链表，每个节点代表该状态对象在某个快照版本中的值。

    // Snapshot.kt
public abstract class StateRecord(internal var snapshotId: SnapshotId) {
    internal var next: StateRecord? = null
    public abstract fun assign(value: StateRecord)
    public abstract fun create(): StateRecord
    public open fun create(snapshotId: SnapshotId): StateRecord
}

    这是一个典型的 MVCC 版本链：新版本被插入链表头部，旧版本保留在链表中供仍然打开的旧快照读取。不同快照根据各自的 snapshotId 和 invalid 集合，在链表中找到对自己"可见"的那个版本。

    3.1 可见性判定 —— valid() 函数

    private fun valid(
    currentSnapshot: SnapshotId,
    candidateSnapshot: SnapshotId,
    invalid: SnapshotIdSet,
): Boolean {
    return candidateSnapshot != INVALID_SNAPSHOT &&     // 不是哨兵值 0
        candidateSnapshot <= currentSnapshot &&          // 不能读"未来"的版本
        !invalid.get(candidateSnapshot)                  // 不在无效集合中
}

    直觉上：一个快照只能看到在它创建之前已经提交的版本，看不到其他尚未提交的并发快照写入的版本。

    private fun <T : StateRecord> readable(r: T, id: SnapshotId, invalid: SnapshotIdSet): T? {
    var current: StateRecord? = r
    var candidate: StateRecord? = null
    while (current != null) {
        if (valid(current, id, invalid)) {
            candidate =
                if (candidate == null) current
                else if (candidate.snapshotId < current.snapshotId) current else candidate
        }
        current = current.next
    }
    return candidate as T?
}

    public fun <T : StateRecord> T.readable(state: StateObject): T {
    val snapshot = Snapshot.current
    snapshot.readObserver?.invoke(state)       // 通知观察者：某个状态被读取了
    return readable(this, snapshot.snapshotId, snapshot.invalid)

                     syncSnapshot.snapshotId, syncSnapshot.invalid)
                ?: readError()
        }

    internal fun <T : StateRecord> T.writableRecord(state: StateObject, snapshot: Snapshot): T {
    val id = snapshot.snapshotId
    val readData = readable(this, id, snapshot.invalid) ?: sync { /* fallback */ }

    // 如果可读记录就是当前快照创建的，直接可写
    if (readData.snapshotId == snapshot.snapshotId) return readData

    // 否则，创建一个新的副本，挂到链表头部
    val newData = sync {
        val newReadData = readable(state.firstStateRecord, id, snapshot.invalid) ?: readError()
        if (newReadData.snapshotId == id) newReadData
        else newReadData.newWritableRecordLocked(state, snapshot)
    } as T

    // 仅当记录不属于预存快照时才标记为已修改
    if (readData.snapshotId != Snapshot.PreexistingSnapshotId.toSnapshotId()) {
        snapshot.recordModified(state)
    }
    return newData
}

    关键设计：如果当前快照已经拥有该状态的最新版本（即之前已经写过），那么可以直接原地修改；否则创建一个新的 StateRecord 副本，snapshotId 设为当前快照的 ID，插入链表头部。被修改的状态对象被记录到快照的 modified 集合中，等待 apply() 时统一提交。注意这里有一个保护条件：如果可读记录属于 PreexistingSnapshotId（快照系统初始化之前就存在的状态记录），则不标记为"已修改"，避免将历史遗留状态误报给 applyObserver。

    public fun takeMutableSnapshot(
    readObserver: ((Any) -> Unit)? = null,
    writeObserver: ((Any) -> Unit)? = null,
): MutableSnapshot =
    (currentSnapshot() as? MutableSnapshot)
        ?.takeNestedMutableSnapshot(readObserver, writeObserver)

    创建过程首先推进全局快照（刷新待处理的变更），然后分配新的 snapshotId，将当前所有打开的快照 ID 集合作为新快照的 invalid 集合。这意味着新快照看不到任何与它并发的、尚未提交的写入。

    对于嵌套的可变快照，子快照的 invalid 集合被设置为从父快照 ID + 1 到新 ID 的区间，确保子快照看不到在父快照和自身之间创建的任何快照。

    internal fun innerApplyLocked(

    //             —— 全局当前看到的值（"theirs"）
    // 2. previous = readable(first, this.snapshotId, start)
    //             —— 本快照创建时看到的值（"common ancestor"）

    //             —— 本快照写入的值（"ours"）

    // 解决方案：调用 state.mergeRecords(previous, current, applied)
    //           如果返回 null，则提交失败（SnapshotApplyResult.Failure）

    如果全局状态在本快照执行期间没有被其他快照修改过（current == previous），则直接接受本快照的写入，没有冲突。如果全局状态确实被修改了，则调用 SnapshotMutationPolicy.merge() 尝试解决冲突。

    五、冲突解决 —— SnapshotMutationPolicy

    SnapshotMutationPolicy 定义了状态值的相等性判断和冲突合并策略：

    1
2
3
4

    public interface SnapshotMutationPolicy<T> {
    public fun equivalent(a: T, b: T): Boolean
    public fun merge(previous: T, current: T, applied: T): T? = null
}

    Compose 提供了三种内置策略：

    structuralEqualityPolicy() 使用 ==（结构相等）判断，是 mutableStateOf 的默认策略。当新值与旧值 equals() 为 true 时，认为状态没有变化，不触发重组。

    referentialEqualityPolicy() 使用 ===（引用相等）判断，适用于需要区分相同值的不同实例的场景。

    neverEqualPolicy() 始终返回 false，意味着每次赋值都视为状态变化。

    merge() 方法默认返回 null（表示冲突无法解决，apply() 失败）。但开发者可以实现自定义的合并逻辑，实现类似 CRDT（无冲突复制数据类型）的效果 —— 例如一个计数器状态可以将并发增量相加而非覆盖。

    override fun mergeRecords(
    previous: StateRecord,
    current: StateRecord,
    applied: StateRecord,
): StateRecord? {
    val previousRecord = previous as StateStateRecord<T>
    val currentRecord = current as StateStateRecord<T>
    val appliedRecord = applied as StateStateRecord<T>
    return if (policy.equivalent(currentRecord.value, appliedRecord.value)) current
    else {
        val merged = policy.merge(
            previousRecord.value,
            currentRecord.value,
            appliedRecord.value
        )
        if (merged != null) {
            appliedRecord.create(appliedRecord.snapshotId).also { it.value = merged }
        } else null
    }
}

    六、线程安全机制

    Snapshot 系统使用了多层线程安全策略的组合：

    全局互斥锁 —— 所有共享的元数据（openSnapshots、nextSnapshotId、applyObservers、pinningTable 等）由一个单一的 SynchronizedObject 保护：

    线程本地快照 —— SnapshotThreadLocal<Snapshot> 确保每个线程独立持有自己的当前快照引用，读写操作无需跨线程竞争。

    不可变数据结构 —— SnapshotIdSet 的所有修改操作返回新实例，不存在共享可变状态。

    乐观锁优化 —— apply() 的三方合并预计算在锁外执行（optimisticMerges），只在最终验证和写入时短暂持有锁，显著减小了临界区。

    原子引用 —— SnapshotStateObserver 的变更队列使用 AtomicReference + CAS 循环实现无锁入队。

    private fun addChanges(set: Set<Any>) {
    while (true) {
        val old = pendingChanges.get()
        val new = when (old) {
            null -> set                     // 0 → 1: 直接存储
            is Set<*> -> listOf(old, set)   // 1 → 2: 包装为列表
            is List<*> -> old + listOf(set) // N → N+1: 追加
            else -> report()
        }
        if (pendingChanges.compareAndSet(old, new)) break
    }
}

    这种渐进式的数据结构升级（null → Set → List）在低并发场景下避免了不必要的集合分配。

    fun recordInvalidation(changes: Set<Any>): Boolean {
    changes.fastForEach { value ->
        // 跳过从未在 SnapshotStateObserver 中读取过的状态对象
        if (value is StateObjectImpl && !value.isReadIn(ReaderKind.SnapshotStateObserver))
            return@fastForEach

        // 检查是否影响 DerivedState
        if (!readingDerivedStates && value in dependencyToDerivedStates) {
            dependencyToDerivedStates.forEachScopeOf(value) { derivedState ->
                val previousValue = recordedDerivedStateValues[derivedState]
                val policy = derivedState.policy ?: structuralEqualityPolicy()
                if (!policy.equivalent(derivedState.currentRecord.currentValue, previousValue)) {
                    // DerivedState 的值确实变了 → 使其读者也失效
                    valueToScopes.forEachScopeOf(derivedState) { scope ->
                        invalidated.add(scope)
                    }
                }
            }
        }

        // 直接失效
        valueToScopes.forEachScopeOf(value) { scope ->
            invalidated.add(scope)
        }
    }
    return hasValues
}

    这里有一个精妙的优化：DerivedState 的依赖会被递归展开。如果一个底层状态变化了，但它所派生的 DerivedState 值实际上没有变（由 SnapshotMutationPolicy.equivalent() 判断），那么读取该 DerivedState 的作用域不会被失效。这避免了大量不必要的重组。

        readObserverOf(composition),
        writeObserverOf(composition, modifiedValues),
    )
    try {
        return snapshot.enter(block)
    } finally {
        applyAndCheck(snapshot)
    }
}

    override fun recordReadOf(value: Any) {
    if (!areChildrenComposing) {
        composer.currentRecomposeScope?.let { scope ->
            scope.used = true
            val alreadyRead = scope.recordRead(value)
            if (!alreadyRead) {
                if (value is StateObjectImpl) {
                    value.recordReadIn(ReaderKind.Composition)
                }
                observations.add(value, scope)
                // DerivedState 的依赖也一并追踪
                if (value is DerivedState<*>) {
                    val record = value.currentRecord
                    derivedStates.removeScope(value)
                    record.dependencies.forEachKey { dependency ->
                        dependency.recordReadIn(ReaderKind.Composition)
                        derivedStates.add(dependency, value)
                    }
                    scope.recordDerivedStateValue(value, record.currentValue)
                }
            }
        }
    }
}

private fun invalidateScopeOfLocked(value: Any) {
    observations.forEachScopeOf(value) { scope ->
        if (scope.invalidateForResult(value) == InvalidationResult.IMMINENT) {
            observationsProcessed.add(value, scope)
        }
    }

    这些被收集的失效在 recordComposerModifications() 中被分发到各个 Composition，最终触发 performRecompose() 的执行。

    这里有一个值得注意的优化：StateObjectImpl.recordReadIn(ReaderKind) 使用原子操作记录状态对象被哪种读者（Composition 还是 SnapshotStateObserver）读取过。如果一个状态从未在组合中被读取（例如它只被布局或绘制代码使用），Recomposer 的全局 apply observer 可以直接跳过它，避免不必要的重组调度。

    九、两条独立的观察路径

    理解 Snapshot 系统的一个关键架构洞察是：组合失效和布局/绘制失效是两条独立的观察路径。

    Recomposer 通过全局 registerApplyObserver 监听快照提交事件，将变更的状态对象分发给组合层，驱动 Composable 函数的重新执行。这是"重组"意义上的失效。

    SnapshotStateObserver 通过自己的 registerApplyObserver 独立监听同样的事件，但它服务于布局和绘制阶段。当一个 Composable 函数在布局（Layout）或绘制（DrawScope）过程中读取了某个状态，SnapshotStateObserver 负责在状态变化时只使对应的布局节点或绘制层失效，而非重新执行整个 Composable。

    这两条路径共享底层的 MVCC 快照基础设施，但各自维护独立的 ScopeMap 和失效队列，互不干扰。

    第一次调用确保在组合阶段创建的状态对象（通过 remember { mutableStateOf(...) }）被全局快照看到。第二次调用确保在应用变更阶段创建的状态对象也被追踪。这两次通知使得后续的状态变更都能正确触发重组。

    TransparentObserverMutableSnapshot 是另一种值得关注的快照类型。它不引入新的隔离边界，而是在父快照之上叠加额外的 readObserver / writeObserver。这在需要追踪读取但不想创建新事务边界的场景中使用。

    从架构角度看，Snapshot 系统做了几件非常精巧的事情。它将数据库级别的 MVCC 事务隔离引入到 UI 框架中，用版本链替代了传统的互斥锁，实现了真正的无阻塞并发读写。三方合并算法确保了并发写入的正确性，而 SnapshotMutationPolicy 提供了灵活的冲突解决策略。两条独立的观察路径（组合级和布局级）共享同一个底层基础设施，却各自维护精确的依赖追踪，使得失效通知的粒度可以做到非常精细。

    对于日常开发而言，理解这些底层机制有助于我们写出更高效的 Compose 代码 —— 例如为什么 @Stable 注解能帮助跳过重组、为什么 DerivedState 能减少不必要的重计算、为什么从非 UI 线程修改 mutableStateOf 是安全的。这些上层行为的正确性，都建立在 Snapshot 系统坚实的理论基础之上。
