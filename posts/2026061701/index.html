    ClassLinker::InitFromBootImage 详细源码分析

    概述

    InitFromBootImage 是 Android Runtime (ART) 启动时从预编译的 Boot Image 初始化 ClassLinker 的核心函数。

    背景知识

    什么是 Boot Image？

    Boot Image 是一个预先编译好的内存镜像，包含了 Java 核心类（如 java.lang.Object, java.lang.Class, java.lang.String 等）的运行时表示。它由 dex2oat 编译工具在系统构建时生成，主要作用是：

    1. 加速启动：避免运行时解析和加载数百个核心 Java 类
    2. 内存共享：所有应用进程通过 fork 继承 Boot Image，实现内存共享
    3. 预编译优化：核心类的机器码已提前编译，无需解释执行

    Boot Image vs App Image

    特性Boot ImageApp Image
    内容Java 核心类库应用特定的类
    生成时机系统构建时应用安装或后台优化时
    共享范围所有进程共享仅限单个应用
    存储位置/system/framework/*.art/data/app/*/oat/

    Boot Image 的组成

    Boot Image 通常由多个文件组成：

    • boot.art / boot.oat：核心类（Object, Class, String 等）
    • boot-framework.art / boot-framework.oat：Android 框架类
    • boot-ext.art / boot-ext.oat：扩展类库

    每个 .art 文件对应一个 ImageSpace，包含预编译的对象和类结构；.oat 文件包含编译后的机器码。

    实现原理

    这些是 ART 运行时的特殊 ArtMethod，不是真正的 Java 方法，而是运行时内部使用的占位符。

    ResolutionMethod (kResolutionMethod)

    • 用途: 方法调用的惰性解析
    • 原理: 当一个方法尚未解析时，其入口点指向 ResolutionMethod
    • 流程:
      1. 调用 ResolutionMethod 的入口点
      2. 触发 artQuickResolutionTrampoline
      3. 解析目标方法，更新入口点
      4. 跳转到真正的方法代码

    ImtConflictMethod (kImtConflictMethod)

    • 用途: 接口方法表 (Interface Method Table, IMT) 的冲突处理
    • 原理:
      • IMT 是一个固定大小的哈希表，用于快速查找接口方法
      • 当多个接口方法哈希到同一个槽位时，该槽位指向 ImtConflictMethod
      • ImtConflictMethod 会遍历冲突链表找到正确的方法

    ImtUnimplementedMethod (kImtUnimplementedMethod)

    • 用途: 未实现的接口方法
    • 原理: 当类没有实现某个接口方法时，IMT 槽位指向此方法，调用时会抛出 AbstractMethodError

    CalleeSaveMethod (多种变体)

    • 用途: 调用者保存寄存器的模板方法
    • 原理:
      • ART 使用寄存器传递参数和返回值
      • 某些调用约定需要保存部分寄存器（如只保存引用、保存所有寄存器等）
      • 这些方法包含保存/恢复寄存器的机器码模板

    变体说明:

    变体用途
    kSaveAllCalleeSaves保存所有被调用者保存寄存器
    kSaveRefsOnly只保存引用类型寄存器
    kSaveRefsAndArgs保存引用和参数寄存器
    kSaveEverything保存所有寄存器（用于 GC 安全点）
    kSaveEverythingForClinit用于类初始化
    kSaveEverythingForSuspendCheck用于线程挂起检查

    std::vector<const OatFile*> OatFileManager::RegisterImageOatFiles(
    const std::vector<gc::space::ImageSpace*>& spaces) {
  std::vector<const OatFile*> oat_files;
  oat_files.reserve(spaces.size());
  for (gc::space::ImageSpace* space : spaces) {
    // The oat file was generated in memory if the image space has a profile.
    bool in_memory = !space->GetProfileFiles().empty();
    oat_files.push_back(RegisterOatFile(space->ReleaseOatFile(), in_memory));
  }
  return oat_files;
}
    • 从 ImageSpace 获取并释放其拥有的 OatFile
    • 注册到 OatFileManager 的全局列表中
    • in_memory 标记是否是内存中生成的（如 JIT 编译产物）

    Trampoline 详解

    Trampoline 是一小段跳板代码，用于在不同执行模式间切换：

    Trampoline用途
    jni_dlsym_lookup_trampoline_JNI 方法动态查找（通过 dlsym 查找 native 方法）
    jni_dlsym_lookup_critical_trampoline_Critical JNI 方法查找（快速 JNI 调用）
    quick_resolution_trampoline_Quick 方法解析（未解析方法的首次调用）
    quick_imt_conflict_trampoline_IMT 冲突处理（接口方法冲突）
    quick_generic_jni_trampoline_通用 JNI 调用桥接
    quick_to_interpreter_bridge_trampoline_从 Quick 代码跳转到解释器
    nterp_trampoline_Nterp 解释器入口（优化的字节码解释器）

    这些 Trampoline 的机器码存储在 OAT Header 中，因为它们需要在 Boot Image 加载前就可用。

    bool ClassLinker::AddImageSpaces(ArrayRef<gc::space::ImageSpace*> spaces,
                                 Handle<mirror::ClassLoader> class_loader,
                                 ClassLoaderContext* context,
                                 /*out*/ std::vector<std::unique_ptr<const DexFile>>* dex_files,
                                 /*out*/ std::string* error_msg) {
  // 第一步：打开所有 Image 的 Dex 文件
  std::vector<std::vector<std::unique_ptr<const DexFile>>> dex_files_by_space_index;
  for (const gc::space::ImageSpace* space : spaces) {
    std::vector<std::unique_ptr<const DexFile>> space_dex_files;
    if (!OpenAndInitImageDexFiles(space, class_loader, /*out*/ &space_dex_files, error_msg)) {
      return false;
    }
    dex_files_by_space_index.push_back(std::move(space_dex_files));
  }

  // 第二步：添加每个 Image Space（必须在所有 Dex 文件打开后）
  for (size_t i = 0u, size = spaces.size(); i != size; ++i) {
    std::vector<std::unique_ptr<const DexFile>>& space_dex_files = dex_files_by_space_index[i];
    if (!AddImageSpace(spaces[i], class_loader, context, space_dex_files, error_msg)) {
      return false;
    }
    std::move(space_dex_files.begin(), space_dex_files.end(), std::back_inserter(*dex_files));
  }
  return true;
}

    两阶段处理的原因：Image Space 之间可能存在跨引用（如 boot-framework.art 中的类引用 boot.art 中的类），必须先打开所有 Dex 文件，确保引用可以解析。

    bool ClassLinker::OpenAndInitImageDexFiles(
    const gc::space::ImageSpace* space,
    Handle<mirror::ClassLoader> class_loader,
    std::vector<std::unique_ptr<const DexFile>>* out_dex_files,
    std::string* error_msg) {
  const ImageHeader& header = space->GetImageHeader();
  ObjPtr<mirror::Object> dex_caches_object = header.GetImageRoot(ImageHeader::kDexCaches);
  ObjPtr<mirror::ObjectArray<mirror::DexCache>> dex_caches =
      dex_caches_object->AsObjectArray<mirror::DexCache>();
  const OatFile* oat_file = space->GetOatFile();

  // 验证 DexCache 数量与 OAT 文件中的 Dex 文件数量匹配
  if (oat_file->GetOatHeader().GetDexFileCount() !=
      static_cast<uint32_t>(dex_caches->GetLength())) {
    *error_msg = "Dex cache count and dex file count mismatch...";
    return false;
  }

  // 打开每个 Dex 文件并关联到 DexCache
  for (auto dex_cache : dex_caches->Iterate()) {
    std::string dex_file_location(dex_cache->GetLocation()->ToModifiedUtf8());
    std::unique_ptr<const DexFile> dex_file = OpenOatDexFile(oat_file,
                                                             dex_file_location.c_str(),
                                                             error_msg);
    dex_cache->SetDexFile(dex_file.get());
    out_dex_files->push_back(std::move(dex_file));
  }
  return true;
}

    DexCache 的作用：每个 Dex 文件对应一个 DexCache，缓存字符串、类型、方法、字段的解析结果，避免重复解析，加速类加载。

    void ClassLinker::FinishInit(Thread* self) {
  VLOG(startup) << "ClassLinker::FinishInit entering";

  // 1. 建立 String.<init> 到 StringFactory 的绑定
  CreateStringInitBindings(self, this);

  // 2. 设置 java.lang.ref.Reference 的字段偏移量
  //    这些偏移量被 GC 直接使用
  StackHandleScope<3> hs(self);
  Handle<mirror::Class> java_lang_ref_Reference =
      hs.NewHandle(GetClassRoot<mirror::Reference>(this));

  ArtField* pendingNext = java_lang_ref_Reference->GetInstanceField(0);
  CHECK_STREQ(pendingNext->GetName(), "pendingNext");
  ArtField* queue = java_lang_ref_Reference->GetInstanceField(1);
  CHECK_STREQ(queue->GetName(), "queue");
  ArtField* referent = java_lang_ref_Reference->GetInstanceField(3);
  CHECK_STREQ(referent->GetName(), "referent");

  // 3. 验证所有 ClassRoots 已初始化
  for (size_t i = 0; i < static_cast<size_t>(ClassRoot::kMax); i++) {
    ClassRoot class_root = static_cast<ClassRoot>(i);
    ObjPtr<mirror::Class> klass = GetClassRoot(class_root);
    CHECK(klass != nullptr);
    DCHECK(klass->IsArrayClass() || klass->IsPrimitive() || klass->GetDexCache() != nullptr);
  }

  // 4. 标记初始化完成
  init_done_ = true;

  // 5. 在内存工具（如 ASan）下，预初始化 StackOverflowError
  if (kMemoryToolIsAvailable && !Runtime::Current()->IsAotCompiler()) {
    ObjPtr<mirror::Class> soe_klass = FindSystemClass(self, "Ljava/lang/StackOverflowError;");
    if (soe_klass == nullptr || !EnsureInitialized(self, hs.NewHandle(soe_klass), true, true)) {
      LOG(WARNING) << "Could not prepare StackOverflowError.";
      self->ClearException();
    }
  }
}

    关键点

    要点说明
    String 初始化绑定Java 中 new String(...) 实际调用 StringFactory 的静态方法，需要建立构造函数到工厂方法的映射
    Reference 字段偏移GC 需要直接访问 Reference 对象的字段，硬编码字段索引比每次查找更高效
    init_done_ 标志设置后，FindClass 等方法使用快速路径；之前使用慢路径处理未初始化的情况

    性能优化意义

    这个函数是 Android 启动性能优化的关键——通过预编译的 Boot Image，避免了运行时解析和加载数百个核心 Java 类的开销，使得 Zygote 可以在几十毫秒内完成初始化。

    优化点说明
    预编译的类结构所有核心类的 Class 对象、方法表、字段布局都已在编译时确定
    预编译的机器码方法的 AOT 编译结果直接可用
    预分配的对象异常对象、Sentinel 等在 Image 中预分配，避免运行时分配
    内存映射加载Boot Image 通过 mmap 加载，无需解析和构建数据结构
    Zygote 共享所有应用进程通过 fork 继承 Boot Image，实现内存共享
