    ART 虚拟机内存分配器深度解析

    摘要

    ART（Android Runtime）虚拟机为了应对不同场景的内存分配需求，设计了多种内存分配器。本文深入分析 DlMallocSpace、RosAllocSpace、BumpPointerSpace、RegionSpace 和 LargeObjectSpace 五种分配器的设计特点、应用场景及其选择策略，帮助读者理解 ART 内存管理的核心机制。

    核心问题：为什么需要多种分配器？它们各自适合什么场景？

    第一章：为什么需要多种分配器

    1.1 对象的生命周期差异

    Java 堆中的对象具有显著不同的生命周期特征：

    对象类型生命周期典型示例
    短期对象毫秒级方法内的临时变量、循环计数器
    中期对象分钟级Activity 生命周期内的对象
    长期对象进程级Application、静态字段引用的对象
    永久对象不回收Zygote 预加载的系统类

    不同的生命周期对分配器提出不同要求：短期对象需要极速分配，长期对象需要稳定布局。

    1.2 GC 策略对分配器的要求

    ART 支持多种 GC 算法，每种算法对分配器有不同约束：

    GC 类型核心机制对分配器的要求
    Mark-Sweep (MS)标记-清除支持精确释放单个对象
    Concurrent MS (CMS)并发标记-清除同上，且需低停顿
    Semi-Space (SS)复制式 GC连续空间，支持整块回收
    Mark-Compact (MC)标记-压缩连续空间，支持对象移动
    Concurrent Copying (CC)并发复制区域化管理，支持读屏障

    1.3 性能优化的多维目标

    分配器设计需要在多个目标间权衡：

    没有万能分配器，只能针对特定场景优化。

    第二章：DlMallocSpace - 通用分配器

    2.1 核心特点

    DlMallocSpace 是对 Doug Lea 的 dlmalloc（glibc malloc 的前身）的封装。

    设计特点：

    特性描述
    成熟稳定数十年优化，边界情况处理完善
    任意大小支持从几字节到数 MB 的分配
    碎片合并相邻空闲块自动合并，减少碎片
    Bin 组织128 个 bins，按大小分类管理空闲块

    核心数据结构：

    1
2
3
4
5
6

    // malloc_chunk 结构（简化）
struct malloc_chunk {
    size_t prev_foot;   // 前一个块的大小（仅当前一块空闲时有效）
    size_t head;        // 当前块大小 + 标志位
    // 用户数据从这里开始
};

    boundary tags 机制：每个块头部和尾部存储大小信息，释放时可快速找到相邻块进行合并。

    Zygote 进程的特殊性决定了必须使用 DlMallocSpace：

    第三章：RosAllocSpace - 高性能分配器

    3.1 核心特点

    RosAlloc（Runs-of-Slots Allocator）专为 ART 设计，针对 Java 对象分配特点优化。

    设计哲学：用空间换时间，用固定大小槽位换取 O(1) 分配。

    核心参数：

    3.5 为什么适合 App 主堆

    1. 高频小对象分配

    Java 程序中绝大多数对象小于 128 bytes：

    1
2

4

    对象大小分布（典型 Java 应用）：
┌────────────────────────────────────┐
│ ≤128 bytes  ████████████████ 85%   │
│ 128-512 B   ████             10%   │
│ 512B-2KB    ██                4%   │

    2. 线程本地分配无锁

    前 16 个 bracket（覆盖 85% 的分配）完全无锁：

    大小范围分配方式锁竞争
    ≤128 bytes线程本地 Run无

    不支持合并固定大小 slot，无法合并相邻空闲
    依赖 GC只能通过 GC 回收整个 Run
    大对象退化>2KB 退化为页分配

    BumpPointerSpace 使用最简单的指针碰撞（Bump Pointer）分配策略。

    核心实现：

    // art/runtime/gc/space/bump_pointer_space.h
class BumpPointerSpace {
    uint8_t* begin_;    // 起始地址
    uint8_t* end_;      // 当前分配指针
    uint8_t* limit_;    // 结束地址
};

// 分配实现（简化）
Object* AllocNonvirtual(size_t size) {
    uint8_t* old_end = end_.fetch_add(size);  // 原子操作
    if (old_end + size <= limit_) {
        return old_end;  // 成功
    }
    return nullptr;  // 空间不足
}

    特点：

    特性描述
    O(1) 分配仅需一条原子加法指令
    不支持释放Free() 返回 0，不做任何操作
    无碎片对象连续存放
    需要双倍空间from/to 两个空间交替使用

    第九章：线程本地分配优化

    9.1 为什么需要线程本地分配

    问题：多线程场景下全局锁竞争严重。

    无线程本地优化：

Thread 3 ──┤

等待时间 = 线程数 × 单次分配时间

    解决方案：每个线程持有独立缓冲区。

    // 分配路径（简化）
void* RosAlloc::Alloc(Thread* self, size_t size) {
    size_t bracket_idx = SizeToBracketIndex(size);
    
    if (bracket_idx < kNumThreadLocalSizeBrackets) {
        // 线程本地分配（无锁）
        Run* run = self->GetRosAllocRun(bracket_idx);
        if (run != nullptr) {
            return run->AllocSlot();  // 无锁！
        }
    }
    
    // 共享分配（需要锁）
    MutexLock lock(size_bracket_locks_[bracket_idx]);
    Run* run = current_runs_[bracket_idx];
    return run->AllocSlot();
}

    10.2 设计启示

    1. 没有万能分配器

    每种分配器都在特定维度优化：

    • DlMalloc：碎片管理
    • RosAlloc：分配速度 + 线程扩展
    • BumpPointer：极致速度
    • Region：并发 + 区域化

    2. 场景驱动设计

    • Zygote → DlMalloc（长期存活，不可移动）
    • App 主堆 → RosAlloc（高频小对象）
    • 移动式 GC → BumpPointer（整块回收）
    • 并发复制 → Region（读屏障协作）

    3. 权衡是永恒主题

    1
2
3

    速度 ←→ 碎片
线程扩展 ←→ 内存开销
简单 ←→ 功能
