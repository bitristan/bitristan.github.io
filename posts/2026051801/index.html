    Linux 设备驱动模型：总线、设备、驱动的关系

    一、核心思想：一个红娘模型

    Linux 设备驱动模型的核心可以用一句话概括：总线是红娘，负责把设备和驱动撮合到一起。

    想象一个场景：

    • 设备说：“我是谁谁谁，谁能来管我？”
    • 驱动说：“我能管什么样的设备，谁需要我？”
    • 总线说：“别急，我来帮你们配对。”

    三者中，总线是核心枢纽。设备和驱动都必须挂载在某条总线上，由总线负责匹配（match）。匹配成功后，驱动的 probe 函数被调用，驱动开始真正操控设备。

struct bus_type {
    const char      *name;          // 总线名称，如 "platform"、"pci"、"usb"
    int (*match)(struct device *dev, const struct device_driver *drv);  // 匹配函数！核心中的核心
    int (*probe)(struct device *dev);    // 总线层的 probe，会转调驱动的 probe

    void (*shutdown)(struct device *dev);
    const struct dev_pm_ops *pm;    // 电源管理操作
    // ... 其他字段
};

    最关键的字段是 match。每条总线必须实现 match 函数，它决定了"什么样的设备和什么样的驱动能配对"。

    // drivers/base/base.h:166
static inline int driver_match_device(const struct device_driver *drv,
                                      struct device *dev)
{
    return drv->bus->match ? drv->bus->match(dev, drv) : 1;
}

    就这么简单：调用总线提供的 match 函数。返回值 > 0 表示匹配成功，= 0 表示不匹配。

    4.3 probe 的调用链

    匹配成功后，进入 probe 流程：

    // drivers/base/dd.c:572
static int call_driver_probe(struct device *dev, const struct device_driver *drv)
{
    if (dev->bus->probe)           // 优先调用总线的 probe
        ret = dev->bus->probe(dev);
    else if (drv->probe)           // 否则调用驱动自身的 probe
        ret = drv->probe(dev);
    return ret;
}

    大多数总线（如 platform）会提供自己的 probe，在里面做类型转换等通用处理后，再转调驱动的 probe。

    // drivers/base/platform.c:1335
static int platform_match(struct device *dev, const struct device_driver *drv)
{
    struct platform_device *pdev = to_platform_device(dev);
    struct platform_driver *pdrv = to_platform_driver(drv);

    /* 1. driver_override：强制指定驱动（最高优先级） */
    if (pdev->driver_override)
        return !strcmp(pdev->driver_override, drv->name);

    /* 2. 设备树（OF）匹配：最常用！ */
    if (of_driver_match_device(dev, drv))
        return 1;

    /* 3. ACPI 匹配：x86 平台常用 */
    if (acpi_driver_match_device(dev, drv))
        return 1;

    /* 4. ID 表匹配：传统方式 */
    if (pdrv->id_table)
        return platform_match_id(pdrv->id_table, pdev) != NULL;

    /* 5. 名称匹配：最后兜底 */
    return (strcmp(pdev->name, drv->name) == 0);
}

    匹配优先级从高到低：

    1. driver_override：设备强制指定驱动名
    2. 设备树匹配：比较 compatible 属性和驱动的 of_match_table
    3. ACPI 匹配：x86 平台使用
    4. ID 表匹配：驱动提供 id_table，逐项比较设备名
    5. 名称匹配：设备名 == 驱动名

    // drivers/base/platform.c:1379
static int platform_probe(struct device *_dev)
{
    struct platform_driver *drv = to_platform_driver(_dev->driver);
    struct platform_device *dev = to_platform_device(_dev);
    int ret;

    ret = of_clk_set_defaults(_dev->of_node, false);
    if (ret < 0)
        return ret;

    ret = dev_pm_domain_attach(_dev, true);
    if (ret)
        goto out;

    if (drv->probe) {
        ret = drv->probe(dev);    // 转调驱动的 probe！
        if (ret)
            dev_pm_domain_detach(_dev, true);
    }
    // ...
    return ret;
}

    注意类型转换：总线层操作的是通用的 struct device，转调驱动时转换成具体的 struct platform_device。

    八、总结

    概念角色关键结构关键操作
    总线红娘/管理者bus_typematch() 配对，probe() 探测
    设备被管理者device注册时触发匹配
    驱动管理者device_driver注册时触发匹配

    核心流程一句话：设备和驱动分别注册到总线上，总线用 match 函数判断能否配对，配对成功则调用 probe 完成绑定。

    谁后注册，谁主动去找对方——设备后注册就遍历驱动链表，驱动后注册就遍历设备链表。这种双向查找机制确保了无论谁先到，都能找到对方。
