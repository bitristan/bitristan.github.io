帧显示时序总结

Section titled “帧显示时序总结”
VSYNC 时刻屏幕显示SF 正在合成Buffer 状态
N帧 N-1帧 NN-1: ACQUIRED, N: QUEUED
N+1帧 N帧 N+1N: ACQUIRED, N+1: QUEUED
N+2帧 N+1帧 N+2N+1: ACQUIRED, N+2: QUEUED

核心结论：SF 在 VSYNC N 合成帧 N，该帧在 VSYNC N+1 时显示到屏幕上。显示延迟至少 1 帧。

需要拷贝的特殊情况

Section titled “需要拷贝的特殊情况”
场景说明是否 CPU 拷贝
GPU 合成GPU 读取多个源 buffer，合成后写入目标 buffer❌ GPU DMA 操作

零拷贝流程总结

Section titled “零拷贝流程总结”
阶段传递内容是否拷贝
App → SFbuffer_handle_t (fd)❌ 只传递 fd
SF 合成GPU 读取 fd 对应内存❌ GPU DMA 访问
SF → HWCbuffer_handle_t (fd)❌ 只传递 fd
HWC → DRMfd → GEM handle❌ 只是引用转换
DRM → 显示GEM handle → DMA 地址❌ 配置硬件
显示输出DMA 读取内存❌ 硬件 DMA

核心设计：整个流程传递的是”引用”（文件描述符），所有设备（CPU、GPU、显示控制器）都通过 DMA 访问同一块物理内存，实现真正的零拷贝。这种设计避免了大量像素数据的内存拷贝，显著降低了延迟和功耗。

为什么需要软件 VSYNC？

Section titled “为什么需要软件 VSYNC？”

硬件 VSYNC 有以下局限：

  1. 功耗问题：持续开启硬件 VSYNC 会增加功耗
  2. 灵活性不足：无法灵活调整唤醒时机
  3. 精度问题：需要预测下一帧时间

软件 VSYNC 通过预测模型生成，减少对硬件 VSYNC 的依赖。

误差分析

Section titled “误差分析”

在软件 VSYNC 模式下，没有自动检测误差并重新校准的机制，但实际中误差问题不严重：

因素说明
硬件稳定性现代显示屏刷新率非常稳定（晶振精度高）
定期校准用户操作会触发重新校准（屏幕开关、刷新率切换等）
线性回归VSyncPredictor 使用线性回归计算周期，有一定容错

设计哲学：Android 假设显示刷新率足够稳定，软件 VSYNC 短期内误差可忽略，通过”事件驱动校准”而非”周期性校准”来平衡性能和精度。

总结

Section titled “总结”

Android VSYNC 机制是一个多层次、高效率的同步系统：

  1. 硬件层：Display 通过 HWC 产生 VSYNC 信号
  2. 预测层：VSyncPredictor 基于历史数据预测 VSYNC 时间，减少硬件依赖
  3. 分发层：VSyncDispatch 通过定时器精确分发回调
  4. 应用层：Choreographer 协调 App 的绘制时机

这种设计实现了：

  • 低功耗：仅在必要时开启硬件 VSYNC
  • 高精度：通过预测模型实现精确的帧同步
  • 灵活性：相位偏移允许 App 和 SF 错峰执行
  • 可扩展：支持多种刷新率动态切换
  • 自校准：通过事件驱动的校准机制应对刷新率变化
