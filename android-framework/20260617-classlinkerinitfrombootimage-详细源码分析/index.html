ClassLinker::InitFromBootImage 详细源码分析

概述

Section titled “概述”

InitFromBootImage 是 Android Runtime (ART) 启动时从预编译的 Boot Image 初始化 ClassLinker 的核心函数。

背景知识

Section titled “背景知识”

什么是 Boot Image？

Section titled “什么是 Boot Image？”

Boot Image 是一个预先编译好的内存镜像，包含了 Java 核心类（如 java.lang.Object, java.lang.Class, java.lang.String 等）的运行时表示。它由 dex2oat 编译工具在系统构建时生成，主要作用是：

  1. 加速启动：避免运行时解析和加载数百个核心 Java 类
  2. 内存共享：所有应用进程通过 fork 继承 Boot Image，实现内存共享
  3. 预编译优化：核心类的机器码已提前编译，无需解释执行

Boot Image vs App Image

Section titled “Boot Image vs App Image”
特性Boot ImageApp Image
内容Java 核心类库应用特定的类
生成时机系统构建时应用安装或后台优化时
共享范围所有进程共享仅限单个应用
存储位置/system/framework/*.art/data/app/*/oat/

Boot Image 的组成

Section titled “Boot Image 的组成”

Boot Image 通常由多个文件组成：

  • boot.art / boot.oat：核心类（Object, Class, String 等）
  • boot-framework.art / boot-framework.oat：Android 框架类
  • boot-ext.art / boot-ext.oat：扩展类库

每个 .art 文件对应一个 ImageSpace，包含预编译的对象和类结构；.oat 文件包含编译后的机器码。

第一阶段：获取 Boot Image 基本信息

Section titled “第一阶段：获取 Boot Image 基本信息”

目标：从 Heap 获取已加载的 Boot Image Spaces，读取元数据。

实现原理

Section titled “实现原理”
API作用
Runtime::Current()获取当前线程关联的 Runtime 单例。Runtime 是 ART 的核心管理类，包含堆、线程池、ClassLinker 等所有子系统
Thread::Current()获取当前执行的线程对象（通常是主线程）
heap->GetBootImageSpaces()从 Heap 获取所有 Boot Image 的 ImageSpace 列表。Boot Image 可能由多个部分组成（如 boot.art, boot-framework.art）
ImageHeader每个 Boot Image 文件开头的元数据结构，包含指针大小、校验和、各 Section 偏移量等

第二阶段：验证指针大小

Section titled “第二阶段：验证指针大小”

目标：确保 Boot Image 的指针大小与当前运行时兼容。

实现原理

Section titled “实现原理”
  • 指针大小验证：Boot Image 在编译时（dex2oat）生成，数据结构布局依赖指针大小。32 位系统指针 4 字节，64 位系统指针 8 字节，不匹配会导致内存布局错误
  • AOT 编译器例外：dex2oat 本身也是 ART 程序，可能需要加载不同位数的 Image（如在 64 位主机上编译 32 位目标平台的 Boot Image），因此只有 AOT 编译器允许跨位数操作

第三阶段：设置 Runtime 特殊方法

Section titled “第三阶段：设置 Runtime 特殊方法”

目标：从 Boot Image 获取特殊 ArtMethod，用于方法解析和调用。

实现原理

Section titled “实现原理”

这些是 ART 运行时的特殊 ArtMethod，不是真正的 Java 方法，而是运行时内部使用的占位符。

ResolutionMethod (kResolutionMethod)

Section titled “ResolutionMethod (kResolutionMethod)”
  • 用途: 方法调用的惰性解析
  • 原理: 当一个方法尚未解析时，其入口点指向 ResolutionMethod
  • 流程:
    1. 调用 ResolutionMethod 的入口点
    2. 触发 artQuickResolutionTrampoline
    3. 解析目标方法，更新入口点
    4. 跳转到真正的方法代码

ImtConflictMethod (kImtConflictMethod)

Section titled “ImtConflictMethod (kImtConflictMethod)”
  • 用途: 接口方法表 (Interface Method Table, IMT) 的冲突处理
  • 原理:
    • IMT 是一个固定大小的哈希表，用于快速查找接口方法
    • 当多个接口方法哈希到同一个槽位时，该槽位指向 ImtConflictMethod
    • ImtConflictMethod 会遍历冲突链表找到正确的方法

ImtUnimplementedMethod (kImtUnimplementedMethod)

Section titled “ImtUnimplementedMethod (kImtUnimplementedMethod)”
  • 用途: 未实现的接口方法
  • 原理: 当类没有实现某个接口方法时，IMT 槽位指向此方法，调用时会抛出 AbstractMethodError

CalleeSaveMethod (多种变体)

Section titled “CalleeSaveMethod (多种变体)”
  • 用途: 调用者保存寄存器的模板方法
  • 原理:
    • ART 使用寄存器传递参数和返回值
    • 某些调用约定需要保存部分寄存器（如只保存引用、保存所有寄存器等）
    • 这些方法包含保存/恢复寄存器的机器码模板

变体说明:

变体用途
kSaveAllCalleeSaves保存所有被调用者保存寄存器
kSaveRefsOnly只保存引用类型寄存器
kSaveRefsAndArgs保存引用和参数寄存器
kSaveEverything保存所有寄存器（用于 GC 安全点）
kSaveEverythingForClinit用于类初始化
kSaveEverythingForSuspendCheck用于线程挂起检查

第四阶段：注册 OAT 文件并获取 Trampolines

Section titled “第四阶段：注册 OAT 文件并获取 Trampolines”

目标：注册 OAT 文件并获取各种 Trampoline 跳板代码。

实现原理

Section titled “实现原理”

OAT 文件结构

Section titled “OAT 文件结构”

OAT (On-Address Translation) 文件是 AOT 编译的产物：

  • 包含编译后的机器码
  • 与 Boot Image 配对存在
  • 每个 ImageSpace 都有一个关联的 OatFile

RegisterImageOatFiles 实现

Section titled “RegisterImageOatFiles 实现”

源码位置: oat_file_manager.cc

  • 从 ImageSpace 获取并释放其拥有的 OatFile
  • 注册到 OatFileManager 的全局列表中
  • in_memory 标记是否是内存中生成的（如 JIT 编译产物）

Trampoline 详解

Section titled “Trampoline 详解”

Trampoline 是一小段跳板代码，用于在不同执行模式间切换：

Trampoline用途
jni_dlsym_lookup_trampoline_JNI 方法动态查找（通过 dlsym 查找 native 方法）
jni_dlsym_lookup_critical_trampoline_Critical JNI 方法查找（快速 JNI 调用）
quick_resolution_trampoline_Quick 方法解析（未解析方法的首次调用）
quick_imt_conflict_trampoline_IMT 冲突处理（接口方法冲突）
quick_generic_jni_trampoline_通用 JNI 调用桥接
quick_to_interpreter_bridge_trampoline_从 Quick 代码跳转到解释器
nterp_trampoline_Nterp 解释器入口（优化的字节码解释器）

这些 Trampoline 的机器码存储在 OAT Header 中，因为它们需要在 Boot Image 加载前就可用。

第五阶段：Debug 模式下验证 Trampolines 一致性

Section titled “第五阶段：Debug 模式下验证 Trampolines 一致性”

目标：在 Debug 构建中验证所有 OAT 文件的 Trampoline 一致性，防止因编译错误或文件损坏导致的方法调用问题。

实现原理

Section titled “实现原理”
要点说明
为什么需要一致性检查ClassLinker 只保存一份 trampoline 指针，如果不同 OAT 文件的 trampoline 不同，某些方法可能使用了错误的入口点
检查逻辑遍历所有后续 OAT 文件，比较其 trampoline 是否与第一个相同；如果不同，进一步检查该 Image 中的所有 ArtMethod
LiveBitmap 遍历GetLiveBitmap() 返回该 ImageSpace 的存活对象位图，Walk(visitor) 遍历所有标记的对象，检查其方法的入口点

第六阶段：初始化 Class Roots

目标：初始化核心 Java 类的 Class 对象和预分配的异常对象。

实现原理

Section titled “实现原理”

Class Roots

Section titled “Class Roots”

class_roots_ 是一个 ObjectArray<Class>，存储所有核心 Java 类的 Class 对象（源码位置：class_root.h）。

为什么需要 Class Roots：

  1. 快速访问核心类，无需每次都通过字符串查找
  2. 这些类在运行时无处不在，缓存它们避免大量查找开销
  3. 某些类有特殊的运行时语义（如 Reference 的 GC 处理）

Boot Image Live Objects

Section titled “Boot Image Live Objects”

预分配的异常对象和哨兵对象：

对象用途
kOomeWhenThrowingException抛出异常时用的 OOME
kOomeWhenThrowingOome抛出 OOME 时用的 OOME
kOomeWhenHandlingStackOverflow栈溢出时用的 OOME

Sentinel 对象：用于 JNI 弱引用（NewWeakGlobalRef），当弱引用的目标对象被 GC 回收后，引用指向 Sentinel，这样可以区分”引用已清除”和”引用为 null”。

第八阶段：记录 JNI Stubs

Section titled “第八阶段：记录 JNI Stubs”

目标：缓存 JNI 方法的 stub 代码，避免重复生成。

实现原理

Section titled “实现原理”

JNI Stub 缓存：JNI 方法调用需要从 Java 代码跳转到 native 代码，每个 JNI 方法有一个对应的 stub（小段汇编代码），负责保存寄存器、设置参数、跳转到 native 函数。

为什么缓存：相同签名的 JNI 方法可以共享 stub，避免重复生成。boot_image_jni_stubs_ 是哈希表，键是方法签名，值是 stub 地址。

Debuggable 模式例外：Debuggable 模式下可能需要动态插桩，不使用 AOT 代码，因此不缓存 stub。

总结：InitFromBootImage 的整体流程

Section titled “总结：InitFromBootImage 的整体流程”

性能优化意义

Section titled “性能优化意义”

这个函数是 Android 启动性能优化的关键——通过预编译的 Boot Image，避免了运行时解析和加载数百个核心 Java 类的开销，使得 Zygote 可以在几十毫秒内完成初始化。

优化点说明
预编译的类结构所有核心类的 Class 对象、方法表、字段布局都已在编译时确定
预编译的机器码方法的 AOT 编译结果直接可用
预分配的对象异常对象、Sentinel 等在 Image 中预分配，避免运行时分配
内存映射加载Boot Image 通过 mmap 加载，无需解析和构建数据结构
Zygote 共享所有应用进程通过 fork 继承 Boot Image，实现内存共享
